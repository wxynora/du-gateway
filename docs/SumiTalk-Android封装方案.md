# SumiTalk Android 封装方案

## 结论

SumiTalk 适合走 `Capacitor + Android 原生模块` 方案，不走纯 PWA。

原因很简单：

- 你要 `apk`
- 你要悬浮球
- 你要通知栏常驻
- 你要后台能力
- 这些都需要 Android 原生层，不是网页层能单独解决的

## 目标

把现有的 `SumiTalk` 前端封装成一个可安装的 Android App，并逐步补上这些系统能力：

- 悬浮球
- 常驻通知栏
- 前台服务
- 悬浮窗权限
- 精确闹钟
- 电池优化白名单引导
- 开机自启相关能力

同时保留现在的前端开发方式：

- `React + Vite` 继续用
- 页面继续在 `miniapp` 里开发
- UI 后续继续改，不影响 Android 壳本身

## 方案选型

### 推荐方案

`Capacitor + Kotlin 原生模块`

### 为什么选这个

- 能最大程度复用现有前端
- 能直接打出 `apk`
- 后面要补系统权限比较顺
- 比纯 PWA 靠谱
- 比一开始就写纯原生 WebView 壳省事

### 不推荐的方案

#### 1. 纯 PWA

不适合。

原因：

- 不能做真正的悬浮球
- 后台能力弱
- 权限能力不够
- 不能满足你要的“更像系统级 app”

#### 2. 纯原生 WebView 壳

不是不能做，但不建议一开始就这么干。

原因：

- 原生工程维护成本更高
- 你现在主内容还是前端页面
- 前期会把精力浪费在壳子上

## 最终结构

建议拆成两层：

### 1. 前端层

保留现在的 `miniapp`

职责：

- 会话页
- 日常页
- 工具页
- 设置页
- 聊天逻辑
- 页面跳转
- 业务 UI

### 2. Android 原生层

用 `Capacitor` 生成 Android 工程，再补原生代码

职责：

- 打包成 app
- 权限申请
- 悬浮球
- 通知栏
- 前台服务
- 后台保活相关能力
- 原生和前端的桥接

## 需要的原生能力

第一批重点能力如下：

- `SYSTEM_ALERT_WINDOW`
- `POST_NOTIFICATIONS`
- `FOREGROUND_SERVICE`
- `WAKE_LOCK`
- `SCHEDULE_EXACT_ALARM`
- 忽略电池优化引导
- 开机自启引导

说明：

- 有些权限可以正常申请
- 有些权限必须跳系统设置页手动开
- 悬浮球、精确闹钟、电池优化白名单都属于后者

## 典型使用场景

### 1. 正常打开 SumiTalk

- 用户点击 app 图标
- 进入当前的 `SumiTalk` 前端页面

### 2. 打开悬浮球

- 用户在 app 内开启悬浮球
- app 检查悬浮窗权限
- 权限不足就跳设置页
- 权限通过后启动前台服务
- 前台服务创建悬浮球 overlay

### 3. 点击悬浮球

- 悬浮球点击后唤起 SumiTalk
- 可以默认回到首页
- 后面也可以扩展成直接打开会话页或指定页面

### 4. 后台保持运行

- 通过前台服务和常驻通知尽量提高存活率
- 不是绝对不死，但已经是 Android 正规路线

## 悬浮球方案

悬浮球必须放在 Android 原生层做，不在前端做。

### 原因

- 前端页面不能直接创建系统级悬浮窗
- 悬浮球本质是 Android overlay view
- 这属于系统层能力

### 实现结构

建议新增一个原生服务：

- `FloatingBallService`

职责：

- 创建悬浮球
- 处理拖动
- 处理点击
- 处理吸边
- 处理显示/隐藏

### 悬浮球第一版功能

第一版只做最小可用：

- 可显示
- 可拖动
- 可点击
- 可关闭
- 点击后能回到 app

不要第一版就做：

- 多按钮展开菜单
- 小面板
- 太复杂的动画

## 常驻通知栏方案

建议和悬浮球一起做。

### 目的

- 配合前台服务
- 提高后台存活率
- 给用户一个明确的常驻入口

### 第一版功能

- 显示一条常驻通知
- 文案简洁
- 点击通知回到 app

### 注意

这不是样式重点，先保证功能可用。

## 前端与原生的分工

### 前端负责

- 所有页面 UI
- 页面路由
- 业务接口调用
- app 内部状态

### 原生负责

- 权限检查与申请
- 系统设置跳转
- 前台服务
- 悬浮球
- 通知栏
- 启动 app / 唤起 app

## 前端需要配合的点

前端不用大改，只需要留好桥接入口。

### 需要补的能力

- 支持从原生传入“打开哪个页面”
- app 启动时能读取原生带来的参数
- 根据参数跳到指定页

例如：

- 打开会话页
- 打开日常页
- 打开工具页
- 直接进入聊天页

### 结论

前端只负责“被叫醒后显示什么页面”，不负责做悬浮球本身。

## 推荐的 Android 工程结构

在仓库里建议最终有这几块：

- `miniapp/`
- `android/`
- `android/app/src/main/java/...`

原生层里后续大概率会有这些文件：

- `MainActivity.kt`
- `FloatingBallService.kt`
- `OverlayPermissionHelper.kt`
- `NotificationHelper.kt`
- `SumiTalkBridgePlugin.kt`
- `AndroidManifest.xml`

## Composer 2 能帮什么

Composer 2 可以帮你把大部分代码搭出来。

### 适合让它做的

- 接入 `Capacitor`
- 生成 Android 工程骨架
- 写 Manifest 权限声明
- 写前台服务代码
- 写悬浮球服务代码
- 写权限检查和设置页跳转
- 写原生与前端桥接
- 改 Kotlin 文件和 Gradle 配置

### 不适合完全依赖它做的

- Android Studio 本地环境问题
- Android SDK / JDK / Gradle 安装问题
- 真机权限行为验证
- 厂商 ROM 的后台保活兼容
- 最终签名和正式打包

结论：

Composer 2 适合“写和改代码”，不适合替你完成最后一公里调试。

## 实施阶段

### 第一阶段：先跑成 app

目标：

- `miniapp` 正常 build
- 接入 `Capacitor`
- 生成 Android 工程
- 能在 Android Studio 启动
- 能打出 debug apk

这一阶段完成后：

- 你已经有一个可安装的 SumiTalk app
- 但还没有悬浮球和后台能力

### 第二阶段：补系统能力

目标：

- 加通知权限
- 加前台服务
- 加常驻通知栏
- 加悬浮窗权限检测
- 加悬浮球

这一阶段完成后：

- app 可以常驻通知栏
- 可以显示悬浮球
- 点击悬浮球可以回到 app

### 第三阶段：补增强能力

目标：

- 精确闹钟
- 电池优化白名单引导
- 开机自启引导
- 悬浮球长按操作
- 悬浮球展开菜单

这一阶段完成后：

- SumiTalk 才更接近一个长期运行的私人 app

## 建议的最小落地顺序

按这个顺序最稳：

1. 先用 `Capacitor` 封装 `miniapp`
2. 跑通 Android app
3. 打出 debug apk
4. 加前台服务
5. 加常驻通知栏
6. 加悬浮窗权限
7. 加悬浮球
8. 最后再做精确闹钟、电池优化、开机自启

## 风险和边界

### 1. 后台常驻不是绝对不死

Android 这几年对后台限制很严。

能做的是：

- 尽量提高存活率

不能保证的是：

- 永远不被系统杀掉

尤其在：

- 小米
- OPPO
- vivo
- 华为

这些系统上会更明显。

### 2. 悬浮窗权限需要手动开

这不是代码里声明一下就行。

必须：

- 检查权限
- 跳设置页
- 用户自己打开

### 3. 真机调试一定要做

模拟器不够。

悬浮球、通知、后台、权限都必须真机跑。

## 结论再说一次

SumiTalk 现在就可以开始走 Android 封装，不用等前端 UI 全部定稿。

最合适的路线是：

- 前端继续用现有 `miniapp`
- 外层用 `Capacitor`
- 悬浮球、通知栏、前台服务、权限全部放到 Android 原生层做

这条路线是当前最省事，也最符合你目标的方案。
