# SumiTalk Android 封装方案

## 结论

SumiTalk 适合走 `Capacitor + Android 原生模块` 方案，不走纯 PWA。

原因很简单：

- 你要 `apk`
- 你要悬浮球
- 你要通知栏常驻
- 你要后台能力
- 这些都需要 Android 原生层，不是网页层能单独解决的

## 目标

把现有的 `SumiTalk` 前端封装成一个可安装的 Android App，并逐步补上这些系统能力：

- 悬浮球
- 常驻通知栏
- 前台服务
- 悬浮窗权限
- 精确闹钟
- 电池优化白名单引导
- 开机自启相关能力

同时保留现在的前端开发方式：

- `React + Vite` 继续用
- 页面继续在 `miniapp` 里开发
- UI 后续继续改，不影响 Android 壳本身

## 方案选型

### 推荐方案

`Capacitor + Kotlin 原生模块`

### 为什么选这个

- 能最大程度复用现有前端
- 能直接打出 `apk`
- 后面要补系统权限比较顺
- 比纯 PWA 靠谱
- 比一开始就写纯原生 WebView 壳省事

### 不推荐的方案

#### 1. 纯 PWA

不适合。

原因：

- 不能做真正的悬浮球
- 后台能力弱
- 权限能力不够
- 不能满足你要的“更像系统级 app”

#### 2. 纯原生 WebView 壳

不是不能做，但不建议一开始就这么干。

原因：

- 原生工程维护成本更高
- 你现在主内容还是前端页面
- 前期会把精力浪费在壳子上

## 最终结构

建议拆成两层：

### 1. 前端层

保留现在的 `miniapp`

职责：

- 会话页
- 日常页
- 工具页
- 设置页
- 聊天逻辑
- 页面跳转
- 业务 UI

### 2. Android 原生层

用 `Capacitor` 生成 Android 工程，再补原生代码

职责：

- 打包成 app
- 权限申请
- 悬浮球
- 通知栏
- 前台服务
- 后台保活相关能力
- 原生和前端的桥接

## 需要的原生能力

第一批重点能力如下：

- `SYSTEM_ALERT_WINDOW`
- `POST_NOTIFICATIONS`
- `FOREGROUND_SERVICE`
- `WAKE_LOCK`
- `SCHEDULE_EXACT_ALARM`
- 忽略电池优化引导
- 开机自启引导
- `PACKAGE_USAGE_STATS`
- `AccessibilityService`（自用版，可选）

说明：

- 有些权限可以正常申请
- 有些权限必须跳系统设置页手动开
- 悬浮球、精确闹钟、电池优化白名单都属于后者
- `Usage Access` 适合拿“最近 24 小时用了哪些 app、用了多久”
- `AccessibilityService` 更适合拿“刚刚前台是什么 app / 现在大概在干嘛”

### 关于前台 app 快照

这里把两个能力分开，不要混：

- `usage`：最近 24 小时应用使用快照
- `foreground_app`：最近一次观测到的前台 app 快照

结论：

- `usage` 继续走现在这条 `UsageStats` 链路
- `foreground_app` 不和 `usage` 混成一个桶
- `foreground_app` 更适合跟 `AccessibilityService` 一起落

原因：

- `Usage Access` 更像“最近用过什么、用了多久”
- `AccessibilityService` 更像“窗口刚切到什么 app”
- 两者都保留，分别服务不同判断

## 典型使用场景

### 1. 正常打开 SumiTalk

- 用户点击 app 图标
- 进入当前的 `SumiTalk` 前端页面

### 2. 打开悬浮球

- 用户在 app 内开启悬浮球
- app 检查悬浮窗权限
- 权限不足就跳设置页
- 权限通过后启动前台服务
- 前台服务创建悬浮球 overlay

### 3. 点击悬浮球

- 悬浮球点击后唤起 SumiTalk
- 可以默认回到首页
- 后面也可以扩展成直接打开会话页或指定页面

### 4. 后台保持运行

- 通过前台服务和常驻通知尽量提高存活率
- 不是绝对不死，但已经是 Android 正规路线

## 悬浮球方案

悬浮球必须放在 Android 原生层做，不在前端做。

### 原因

- 前端页面不能直接创建系统级悬浮窗
- 悬浮球本质是 Android overlay view
- 这属于系统层能力

### 实现结构

建议新增一个原生服务：

- `FloatingBallService`

职责：

- 创建悬浮球
- 处理拖动
- 处理点击
- 处理吸边
- 处理显示/隐藏

### 悬浮球第一版功能

第一版只做最小可用：

- 可显示
- 可拖动
- 可点击
- 可关闭
- 点击后能回到 app

不要第一版就做：

- 多按钮展开菜单
- 小面板
- 太复杂的动画

## 常驻通知栏方案

建议和悬浮球一起做。

### 目的

- 配合前台服务
- 提高后台存活率
- 给用户一个明确的常驻入口

### 第一版功能

- 显示一条常驻通知
- 文案简洁
- 点击通知回到 app

### 注意

这不是样式重点，先保证功能可用。

## 当前已落地状态

下面这些已经不是“想法”，而是当前代码里已经做了。

### Android 壳

- 已接入 `Capacitor`
- 已能打出 `debug apk`
- APK 现在走线上 `https://duxy-home.com/miniapp`
- Android 壳已接远程页面加载，不再要求每次改 UI 都重打包

### 权限与设置引导

当前已经接了这些权限/设置跳转：

- 麦克风
- 相机
- 媒体读取
- 通知
- 悬浮窗权限
- 精确闹钟权限
- 忽略电池优化
- 应用使用情况访问权限

说明：

- 这里有些是运行时权限
- 有些是跳系统设置页
- 现在 `MainActivity` 会按顺序检查并引导

### 前台服务与悬浮球

当前已经做了第一版：

- 前台服务
- 常驻通知栏
- 悬浮球显示
- 悬浮球拖动
- 点击悬浮球回到 `SumiTalk`

当前版本的定位是：

- 先能用
- 不追求复杂菜单
- 不追求精细动画

### 设备状态上报

当前已经接上的数据链路：

- 亮屏
- 灭屏
- 解锁
- app 激活
- 最近 24 小时应用使用时长快照

还没接上、但建议新增的一条：

- 当前前台 app 快照（建议跟 `AccessibilityService` 一起做）

当前链路是：

- Android 原生层监听/读取
- 上报到网关
- 网关写入 R2 的 sense bucket

说明：

- 屏幕状态写到 `screen`
- 使用时长写到 `usage`
- 当前前台 app 单独写到 `foreground_app`
- 这套是和健康、电量、位置同一类“设备感知数据”

## 设备状态数据流

设备侧感知数据统一按这条链路走：

1. Android 原生层采集
2. 调用网关接口
3. 网关做鉴权和格式收口
4. 写入 R2
5. 后续由网关统一注入或给 app 内页面使用

### 为什么不让 app 直写 R2

不建议让客户端直接写 R2。

原因：

- 不安全
- 凭证不好管
- 数据格式容易散
- 后面不好和健康、电量、位置统一

结论：

- 客户端只负责采集与上报
- 网关负责存储与后续整合

## 当前能力边界

虽然前台服务和悬浮球已经做了，但要明确现在的边界。

### 已经能稳定做的

- 悬浮球常驻
- 常驻通知
- 亮灭屏监听
- 使用时长快照上报
- 点击悬浮球唤起 app

### 还没做的

- `foreground_app` 前台 app 快照
- `AccessibilityService` 接入与设置引导
- 悬浮球长按菜单
- 悬浮球展开小面板
- 通知点击打开指定页面
- 锁屏提醒样式细化
- 本地原生精确闹钟触发业务
- 开机自启后的自动恢复前台服务
- 使用时长页面展示
- 设备状态在网关 prompt 注入里的正式消费

### 当前还不保证的

- 绝对不死的后台常驻
- 后台直接强弹完整页面
- 所有厂商 ROM 下行为完全一致

## 后续要继续做的内容

下面这些是后面值得继续做的，按优先级排。

### 第一优先级：把现在这条链做稳

先把“已存在的能力”从能跑，做成稳定可用。

要补的点：

- 悬浮球位置记忆
- 悬浮球吸边
- 悬浮球显隐开关
- 悬浮球权限丢失后的自动兜底
- 前台服务状态页
- 使用时长上报失败日志
- 亮灭屏上报失败重试

这一阶段的目标：

- 真机上连续使用稳定
- 权限被收回时行为可预期
- 不会静默失效

### 第二优先级：把“唤起入口”做完整

现在悬浮球只能回到 app，本质还是入口太单一。

后面建议补：

- 点击通知回指定页
- 点击悬浮球直达聊天页
- 从原生层带路由参数唤起
- 前端读取路由参数后跳转到：
  - 会话
  - 日常
  - 工具
  - 指定二级页

这一阶段做完后：

- `SumiTalk` 才像真正可被系统唤起的 app

### 第三优先级：把“设备感知”真正用起来

现在只是把数据采上来并存进 R2，还没充分消费。

后面建议补：

- 在网关里把 `screen/usage` 纳入设备感知上下文
- 新增 `foreground_app`
- 和健康、电量、位置一起统一组织
- 按规则注入给渡
- 给 app 内页面提供“最近设备状态”的展示

可以衍生出的能力：

- 深夜亮屏感知
- 刚解锁后的主动关心
- 长时间使用某 app 后的判断
- 刚刚切到某个 app 的即时感知
- 锁屏/亮屏节律感知

### `foreground_app` 的建议实现

这条能力单独写清楚：

- `usage` 和 `foreground_app` 共存，不互相替代
- `usage` 继续保留“24 小时使用快照”
- `foreground_app` 表示“最近一次窗口切换后，前台是什么 app”

推荐来源：

- 第一选择：`AccessibilityService`
- 不建议把它硬塞进 `usage` 快照里

推荐上报字段：

- `packageName`
- `appName`
- `className`（可选）
- `observedAt`
- `source=accessibility`

推荐消费口径：

- 写成“刚刚前台是 QQ / 微信”
- 或“当前可能在用 QQ”
- 不写死成“她现在一定正在用某 app”

### 第四优先级：本地原生精确闹钟

现在权限已经开了，但“业务层的原生精确闹钟”还没真正做完。

后面要补的是：

- Android 本地闹钟注册
- 到点由原生触发
- 触发后走网关
- 可选通知/悬浮球提醒

做完后好处是：

- 不完全依赖服务器轮询
- 更接近真正的本地 app 行为

### 第五优先级：开机恢复

这是后面非常值得做的一项。

目标：

- 手机重启后
- 如果用户允许
- 自动恢复前台服务和悬浮球

需要补的东西：

- `BOOT_COMPLETED`
- 开机广播接收器
- 恢复前台服务逻辑

注意：

- 不同厂商 ROM 对自启动限制不同
- 这块一定要真机验证

## 弹窗能力的现实边界

“弹窗”要分开说，不要混成一个概念。

### 现在容易做的

- 悬浮球
- 悬浮窗
- 常驻通知
- 锁屏通知
- 点击后打开指定页面

### 不建议先做的

- app 在后台时直接强行弹完整页面

原因：

- Android 对后台启动界面限制很多
- 厂商系统行为不一致
- 比悬浮球/通知更不稳

结论：

- 先做“可见入口”
- 再由用户点击进入页面
- 不先做后台硬弹

## 现在最值得继续推进的顺序

如果继续往下做，建议严格按这个顺序：

1. 先把前台服务和悬浮球细节补稳
2. 再把亮灭屏/使用时长这类设备感知链路补完整
3. 再把设备感知正式注入网关上下文
4. 再做本地原生精确闹钟
5. 最后做开机恢复和更复杂的悬浮球菜单

这样做的原因：

- 先把基础设施打牢
- 再做复杂交互
- 不容易做出一堆半成品系统能力

## 前端与原生的分工

### 前端负责

- 所有页面 UI
- 页面路由
- 业务接口调用
- app 内部状态

### 原生负责

- 权限检查与申请
- 系统设置跳转
- 前台服务
- 悬浮球
- 通知栏
- 启动 app / 唤起 app

## 前端需要配合的点

前端不用大改，只需要留好桥接入口。

### 需要补的能力

- 支持从原生传入“打开哪个页面”
- app 启动时能读取原生带来的参数
- 根据参数跳到指定页

例如：

- 打开会话页
- 打开日常页
- 打开工具页
- 直接进入聊天页

### 结论

前端只负责“被叫醒后显示什么页面”，不负责做悬浮球本身。

## 推荐的 Android 工程结构

在仓库里建议最终有这几块：

- `miniapp/`
- `android/`
- `android/app/src/main/java/...`

原生层里后续大概率会有这些文件：

- `MainActivity.kt`
- `FloatingBallService.kt`
- `OverlayPermissionHelper.kt`
- `NotificationHelper.kt`
- `SumiTalkBridgePlugin.kt`
- `AndroidManifest.xml`

## Composer 2 能帮什么

Composer 2 可以帮你把大部分代码搭出来。

### 适合让它做的

- 接入 `Capacitor`
- 生成 Android 工程骨架
- 写 Manifest 权限声明
- 写前台服务代码
- 写悬浮球服务代码
- 写权限检查和设置页跳转
- 写原生与前端桥接
- 改 Kotlin 文件和 Gradle 配置

### 不适合完全依赖它做的

- Android Studio 本地环境问题
- Android SDK / JDK / Gradle 安装问题
- 真机权限行为验证
- 厂商 ROM 的后台保活兼容
- 最终签名和正式打包

结论：

Composer 2 适合“写和改代码”，不适合替你完成最后一公里调试。

## 风险和边界

### 1. 后台常驻不是绝对不死

Android 这几年对后台限制很严。

能做的是：

- 尽量提高存活率

不能保证的是：

- 永远不被系统杀掉

尤其在：

- 小米
- OPPO
- vivo
- 华为

这些系统上会更明显。

### 2. 悬浮窗权限需要手动开

这不是代码里声明一下就行。

必须：

- 检查权限
- 跳设置页
- 用户自己打开

### 3. 真机调试一定要做

模拟器不够。

悬浮球、通知、后台、权限都必须真机跑。

## 结论再说一次

SumiTalk 现在就可以开始走 Android 封装，不用等前端 UI 全部定稿。

最合适的路线是：

- 前端继续用现有 `miniapp`
- 外层用 `Capacitor`
- 悬浮球、通知栏、前台服务、权限全部放到 Android 原生层做

这条路线是当前最省事，也最符合你目标的方案。
