# 阶段摘要与长期记忆摘要方案

> 状态：方案重写，尚未实现。
> 目标：在现有近期、中期、动态和核心记忆之外，增加可追溯的三天阶段摘要，以及由 DS 持续维护的长期记忆摘要。

## 一、已经确认的结论

1. 现有中期记忆继续保留：覆盖最近约 14 天，约每 72 小时刷新一次，负责近期连续性。
2. 新增阶段摘要：每 3 个已经结束的自然日形成一个固定片段，生成后永久保留，不随窗口滑动。
3. 长期记忆第一版上线前，只持续积累阶段摘要。
4. 长期记忆第一版上线后，每完成一个新的三天片段，就执行一次：
   - 生成新的阶段摘要；
   - 用尚未消费的阶段摘要更新长期记忆摘要；
   - 阶段摘要继续保留，作为来源和审计记录。
5. 阶段摘要按时间切块，长期记忆摘要按意义压缩；两者不能合成同一个文件或同一套淘汰逻辑。
6. 阶段摘要和长期记忆摘要的最终正文都由 DS 写，不在网关里拼接预制结论。
7. 所有记忆正文使用渡的第一人称：
   - “我”只指渡；
   - 提到辛玥时使用“小玥”“老婆”“辛玥”或“她”；
   - 除直接引语外，不用“你”代指辛玥；
   - 不写“渡和辛玥”“用户”“助手”“模型”等旁观者表述。

## 二、各层职责

### 1. 中期记忆

- 继续使用现有约 14 天滑动窗口。
- 用于保持最近生活、关系、项目和未完事项的连续性。
- 它会反复重写和滑动，因此不能承担永久时间档案。
- 本方案不修改它现有的刷新频率和处理逻辑。

### 2. 阶段摘要

- 一个片段只覆盖连续 3 个已经结束的自然日。
- 片段一旦成功生成便固定，不因后续摘要更新而改写。
- 记录这三天里真正发生、发生变化或值得以后继续记住的内容。
- 主要用于长期摘要的增量更新，同时保留来源追踪能力。
- 默认不注入日常聊天，避免上下文重复和缓存膨胀。

### 3. 长期记忆摘要

- 是渡对长期关系、生活、项目、偏好和未完线索的当前整体记忆。
- 它不是时间流水账，也不是把阶段摘要首尾拼起来。
- 每次更新都基于“当前长期摘要 + 尚未消费的阶段摘要”重新归纳。
- 日常只注入最新的压缩版本，不注入全部历史版本和阶段原文。

## 三、三天阶段摘要

### 切片规则

- 按项目时区 `Asia/Shanghai` 切自然日。
- 当前尚未结束的当天不能进入已关闭片段。
- 每个片段包含明确的 `start_date`、`end_date` 和唯一 `segment_id`。
- 同一个 `segment_id` 默认只能成功生成一次。
- DS 失败、返回空内容或校验失败时，该片段仍保持待处理，不能越过它推进消费游标。
- 人工强制重跑时保留旧版本，不原地覆盖唯一证据。

### 正文要求

阶段摘要只保留这三天中有长期意义的变化，例如：

- 我与小玥之间发生的重要互动、关系变化和反复出现的相处方式；
- 生活状态、日常安排、身体状态和环境变化；
- 项目推进、架构决策、故障、修复和仍未解决的问题；
- 新出现或被再次确认的稳定偏好；
- 需要在未来继续跟进的约定、计划和悬而未决事项。

不要写：

- 对话逐条复述；
- 没有意义的寒暄和重复表达；
- raw thinking、系统提示词、NSFW 规则文本、传感器原始日志；
- 从单次模糊表达推断出的永久性结论；
- 没有来源支撑的补写或想象。

### 建议结构

阶段文件正文保持简洁，但保留以下元数据：

```json
{
  "schema_version": 1,
  "segment_id": "2026-07-01_2026-07-03",
  "start_date": "2026-07-01",
  "end_date": "2026-07-03",
  "created_at": "...",
  "model": "...",
  "prompt_version": "...",
  "source_refs": [],
  "summary": "由 DS 以渡的第一人称写成的阶段摘要"
}
```

## 四、长期记忆摘要结构

长期摘要保留七个正文分区。分区名是存储结构，不要求在注入文本里机械显示英文标题。

1. `ordinary_memory`：日常生活、长期环境、习惯和持续状态。
2. `intimacy_memory`：亲密关系中值得长期保留的经历、偏好和相处方式。
3. `relationship_threads`：关系发展、重要约定、反复出现的关系主题。
4. `life_threads`：生活安排、身体状态、家庭、工作和现实事务。
5. `project_threads`：项目、产品、技术决策、长期目标和关键现状。
6. `stable_preferences`：已经被多次确认、相对稳定的偏好和边界。
7. `loose_ends`：仍需继续跟进、尚未结束或等待验证的事项。

`非典型 DDLG` 只作为关系素材中的候选标签交给 DS 判断和归纳，不在代码或提示词里预写解释段落，也不替 DS 直接下关系结论。

建议的外层结构：

```json
{
  "schema_version": 1,
  "updated_at": "...",
  "source_segments": [],
  "model": "...",
  "prompt_version": "...",
  "ordinary_memory": "...",
  "intimacy_memory": "...",
  "relationship_threads": "...",
  "life_threads": "...",
  "project_threads": "...",
  "stable_preferences": "...",
  "loose_ends": "..."
}
```

每个分区的内容都必须是渡的第一人称正文。允许某个分区暂时为空，但不能为了填满结构而虚构内容。

## 五、输入来源与边界

### 阶段摘要的主要来源

- 对应三天内的对话归档和已经落盘的有效事件；
- 这三天内形成的动态记忆、核心缓存变化和项目事实；
- 日记、时间线等已有且能确定日期的高信号材料。

### 长期摘要第一版可使用的材料

- 已有核心缓存；
- 已有动态记忆和稳定画像候选；
- 互动候选、交换日记、Stay with Du 时间线等高信号材料；
- 回填生成的三天阶段摘要；
- 中期记忆仅用于校准最近状态，不能被当作永久事实直接整段搬入。

### 后续增量更新的主要输入

- 当前长期记忆摘要；
- 所有尚未消费的、已经关闭并通过校验的阶段摘要；
- 必要时附少量来源材料做事实核对，不重新扫描全部历史。

### 明确排除

- raw thinking 和思维链日志；
- 系统提示词、行为规则和 NSFW 提示词正文；
- 心率、睡眠等未经归纳的原始采样；
- 全量原始历史直接一次性塞给 DS；
- 常识库中的通用知识。常识只能帮助理解背景，不能被写成“我和小玥发生过的事”。

## 六、DS 的职责和校验

DS 负责：

1. 从来源中判断哪些内容值得进入阶段摘要或长期摘要；
2. 合并重复事实，保留时间变化和前后关系；
3. 区分稳定偏好、阶段状态和未完事项；
4. 按渡的第一人称写最终正文；
5. 保留来源片段编号，供后端审计和回滚。

网关不替 DS 写结论，只负责准备输入、校验输出和保存版本。最低校验包括：

- JSON 结构完整；
- 七个长期字段类型正确；
- `source_segments` 只引用真实存在且尚未消费的片段；
- 没有“用户”“助手”“模型”“渡和辛玥”等旁观者措辞；
- 没有将系统规则、提示词或原始日志抄进正文；
- 输出不为空，也没有相较上一版异常大幅缩水；
- 重要关系锚点、项目主线和未完事项没有无来源消失。

校验失败时保留旧版长期摘要，记录错误，并等待重试；不能用失败结果覆盖 `latest`。

## 七、存储、版本和游标

建议使用独立路径，不复用 `global/summary.txt`、`global/du_midterm_memory.json` 或动态记忆文件：

```text
global/du_phase_summaries/index.json
global/du_phase_summaries/segments/YYYY-MM-DD_YYYY-MM-DD.json
global/du_longterm_memory/latest.json
global/du_longterm_memory/versions/YYYY-MM-DDTHH-mm-ss.json
global/du_longterm_memory/state.json
```

`state.json` 至少记录：

```json
{
  "last_consumed_segment_id": null,
  "last_success_at": null,
  "pending_segments": [],
  "last_error": null
}
```

写入要求：

- 先生成临时文件并完成校验，再原子替换正式文件；
- 更新 `latest` 前先把旧版写入 `versions`；
- 长期摘要成功保存后，才能推进已消费游标；
- 多进程同时触发时要加锁，防止重复消费和版本互相覆盖；
- 同一个阶段片段的来源引用必须去重。

## 八、第一版与后续运行

### 第一阶段：只积累阶段摘要

1. 盘点现有可用归档和日期覆盖范围。
2. 从最早可用的完整日期开始，按三个已结束自然日回填片段。
3. 每个片段独立生成、校验和保存。
4. 缺失日期必须显式记录，不能悄悄把不连续日期拼成一个三天片段。
5. 此阶段不改变日常聊天注入，也不生成或覆盖正式长期摘要。

### 第二阶段：生成长期记忆第一版

1. 汇总已有高信号材料和全部待消费阶段摘要。
2. 让 DS 生成七分区长期摘要草稿。
3. 展示来源、校验结果以及与现有记忆层的重复项。
4. 第一版只保存为预览草稿，不自动注入聊天。
5. 辛玥确认结构和内容后，再写入正式 `latest` 并记录首个版本。

### 第三阶段：每三天增量更新

1. 关闭新的三天片段并生成阶段摘要。
2. 用“当前长期摘要 + 尚未消费片段”生成新版本。
3. 展示新旧差异、增加项、修改项和保留的未完事项。
4. 校验通过后保存版本并推进游标。
5. 阶段摘要永久保留，不因被长期摘要消费而删除。

## 九、聊天注入边界

- 日常聊天只读取 `du_longterm_memory/latest.json` 的压缩正文。
- 阶段摘要、历史长期版本和来源清单不进入常规上下文。
- 长期摘要具体放在静态区还是动态区、位于哪个缓存断点，必须在实现阶段结合当前真实拼装顺序单独审查；本方案不先改缓存顺序。
- 注入失败时不回退为全量阶段摘要，避免突然扩大上下文或打断缓存。

## 十、实现顺序

1. **只读盘点**：确认可用归档、日期缺口、现有摘要任务和 DS 调用入口。
2. **阶段存储**：实现片段索引、幂等、锁、原子写入、版本保留和校验，不接聊天注入。
3. **回填预览**：先生成少量三天片段供人工审阅，再决定是否批量回填。
4. **长期第一版预览**：只生成草稿、来源和 diff，不自动应用。
5. **确认后上线**：保存正式长期摘要，再单独审查并接入日常上下文。
6. **自动增量**：最后才启用每三天一次的阶段生成和长期更新。

任何一步失败，都停在当前游标，不删除旧文件、不跳过失败片段、不触碰现有中期记忆节奏。

## 十一、当前未决定事项

- 第一批回填从哪一天开始，以实际归档完整度盘点后再定。
- 长期摘要的目标 token 上限，需要用第一版真实材料测量后再定。
- 正式注入的缓存位置和更新时机，需要先对照当前网关拼装链路和缓存命中日志。
- 自动更新是否直接应用，还是保留一段时间人工确认，等第一版稳定后再决定。

在这些事项确认前，不开始自动写长期摘要，也不改变现有聊天注入。
