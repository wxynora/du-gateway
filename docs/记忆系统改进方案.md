# 记忆系统改进方案（需求备忘）

本文记录待演进的能力与产品取向，**实现以仓库代码为准**。

---

## 一、基础改进：动态层 = 中期记忆（与窗口总结分工）

窗口总结**只承担**「当前对话不断档、最基础不失忆」，**不**改为情感叙事或复杂策略的主载体。**中期记忆**由动态层承担，需求如下。

### 1.1 定位

- **动态层**：跨轮、可检索、可演化的**中期记忆**（离散短句为主）。
- **窗口总结**：同窗口内**短期连贯**，不与动态层抢叙事职责。

### 1.2 记什么（准入）

- **优先**：关系与情感线索、约定与承诺、稳定偏好与底线、会反复影响后续互动的背景。
- **其次**：近期仍会被提起的具体事实（宜短、宜合并）。
- **避免**：流水账、一次性可查信息、语义重复却不 merge 的堆叠。

### 1.3 形态与演化

- 单条宜**短而准**；同一主题优先 **merge** 收敛，避免多条平行变体。
- **merge 的目标不是拼接旧句子**，而是产出这件事的**当前理解**：保留仍成立的部分，覆盖已变化的判断，尽量避免“旧版本 + 新补丁”式堆叠。
- **importance** 表示**中期保留价值**（关系 / 情感 / 约定偏高），而非「这句话写得好不好」。
- **importance 以首次写入时确定为主**，后续 merge 不随意改动；记忆的新鲜度与活跃度主要由 `mention_count / last_mentioned` 体现。
- 新信息与旧条冲突时，应**更新或合并**，减少过期状态长期占位。

### 1.4 边缘淘汰（仅动态层）

- 对**时间已久、长期根本不参与召回、且未进入核心缓存层**的低权重记忆，从动态层落盘与向量索引中直接清理，**不碰**核心缓存 pending。
- 不把所有低权重旧记忆都做成“褪色版”。很多低价值信息本就应像人一样**直接忘掉**，而不是勉强留一个模糊残影。
- 与「7 天内是否仍参与注入」的规则**解耦**：注入有效期管「进不进本轮 prompt」，边缘淘汰管「删不删池子里沉积条」。

### 1.5 检索与注入

- 召回与排序偏向**与当前话题相关且对互动有影响**的条目；**重质不重量**，不追求条数多。
- 整体取向是：**先搜对，再按权重决定先想起哪条**。
- 现有权重机制本身可保留；更值得优先优化的是**进入候选池之前的相关性质量**，避免“歪但重要”的旧记忆被顶到前面。
- 关键词不宜只停留在散词层面；更理想的检索单位是**语义短语 / 意思块**，例如：
  - `我今天肚子不舒服` → 不应只提成 `今天 / 肚子 / 不舒服`
  - 更应收敛成 `肚子不舒服`
- 可考虑在粗关键词之后增加一层**关键词收敛层（语义短语层）**：
  - 第一层：先做便宜的粗提取，得到若干散关键词
  - 第二层：把散词收成 1-3 个更完整的核心短语，再进入检索
- 实现顺序上，**先规则、后模型**：
  - 先用轻规则把明显可收敛的散词拼成短语
  - 若后续发现规则限制过死、误伤明显，再考虑引入模型做补充收敛
- 关键词信号更适合作为**辅助校正**，而不是一票否决的硬门槛；主判断仍以语义相关为主，避免“必须一模一样”带来的误伤。
- 对动态层条目，可增补一个面向检索的内部字段（如 `retrieval_text`），让记忆在“对外表述像人”的同时，“对内检索更好搜”。
- `retrieval_text` 更适合在**写入 / merge 时同步生成**，离线慢整理只做补修或重建，避免前后台生成口径不一致。

#### 1.5.1 关键词收敛层（规则版草案）

- 目标不是把一句话完整改写，而是从粗关键词里收出 **1-3 个更像“人的记忆点”的核心短语**。
- 总体原则：**优先提“态度 / 感受 / 偏好短语”，其次才是事实 / 事件短语。**
- 例如：
  - `我今天肚子不舒服` → `肚子不舒服`
  - `我最近不太想出门` → `不太想出门`
  - `我不喜欢太甜的奶茶` → `不喜欢太甜`
  - `今天跟妈妈吵架了，我真的很烦` → `很烦`、`跟妈妈吵架`

#### 1.5.2 规则分层

1. **先保留现有粗关键词层**
   - 现有逻辑先给出一批散词，不必推翻。

2. **再做关键词收敛层**
   - 以粗关键词和原句顺序为基础，把散词收成更完整的“意思块”。
   - 若规则无法稳定收敛，宁可退回粗关键词，也不要硬拼出奇怪短语。

#### 1.5.3 优先级顺序

1. **偏好 / 态度 / 感受短语优先**
   - 因为更符合“人的记忆”取向：人通常记住的是“她对这件事什么感觉、什么态度”，而不是一串客观标签。

2. **事实 / 状态 / 事件短语其次**
   - 在没有明显态度词时，再保留身体状态、关系事件、生活事件等短语。

3. **壳子词不单独成短语**
   - 时间、指代、语气、连接词可挂在核心词附近，但不应自己成为检索核心。

#### 1.5.4 核心词类型（第一版）

1. **偏好 / 态度核心词**
   - 喜欢系：`喜欢` `更喜欢` `不喜欢` `讨厌`
   - 意愿系：`想` `想要` `不想` `不太想` `想让`
   - 感受系：`开心` `难过` `委屈` `生气` `烦` `不爽` `害怕` `担心` `紧张` `难受`
   - 关系反应系：`在意` `介意` `失望` `安心` `心疼` `依赖`
   - 接受度系：`可以` `不行` `受不了` `接受不了`
   - 倾向系：`更想` `宁愿` `希望` `不希望`

2. **事实 / 状态核心词**
   - 身体状态：`不舒服` `疼` `头疼` `肚子疼` `想吐` `累` `困`
   - 关系事件：`吵架` `冷战` `和好`
   - 生活事件：`上班` `请假` `去医院` `看书` `搬家`
   - 阶段状态：`忙` `压力大` `没睡好`

3. **壳子词 / 低价值词**
   - 时间壳子：`今天` `昨天` `刚刚` `最近` `这几天`
   - 指代壳子：`这个` `那个` `这样` `那样`
   - 语气壳子：`感觉` `就是` `有点` `有一点` `好像` `其实`
   - 连接壳子：`然后` `所以` `因为` `但是`

#### 1.5.5 收敛规则（第一版）

1. **先找偏好 / 态度核心词**
   - 句子里若出现这类词，优先围绕它向左右拼接，组成一个完整短语。
   - 例如：
     - `不喜欢 + 太甜` → `不喜欢太甜`
     - `不太想 + 出门` → `不太想出门`
     - `有点 + 委屈` → `有点委屈`

2. **没有明显态度词时，再找事实 / 状态核心词**
   - 例如：
     - `肚子 + 不舒服` → `肚子不舒服`
     - `跟妈妈 + 吵架` → `跟妈妈吵架`

3. **按原句相邻关系拼接，不做远距离硬凑**
   - 只优先拼原句里相邻或近邻的词，避免把本来不挨着的词硬拉成一个短语。
   - 目标是“一个完整意思”，不是“尽量多拼”。

4. **壳子词可附着，不可单独成核**
   - 如 `有点委屈` 可保留，但不能只留下 `有点`
   - 如 `最近不太想出门` 中，`最近` 可不作为核心；核心应是 `不太想出门`

5. **对象词是挂载，不一定是中心**
   - 若一句话同时包含对象和态度，优先保留态度判断。
   - 例如：
     - `我不喜欢太甜的奶茶`
     - 检索短语优先取 `不喜欢太甜`
     - 而不是只取 `奶茶太甜`

6. **输出数量控制**
   - 每句最终只保留 **1-3 个核心短语**。
   - 若同一句里既有态度又有事件，可保留两条；态度短语优先级更高。

7. **长度控制**
   - 单个短语宜短，不宜拖成整句；优先控制在**短语级**而非整句级。
   - 目标是“可检索”，不是“复述原话”。

8. **失败回退**
   - 若规则无法稳定收敛，就回退为原有粗关键词，不为“必须收敛”而硬造短语。

#### 1.5.6 与 retrieval_text 的关系

- `content` 仍保持“像人的记忆表达”。
- `retrieval_text` 可更偏检索友好：
  - 保留核心态度 / 感受 / 偏好短语
  - 补少量对象锚点或事件锚点
- 例如：
  - `content`: `她不喜欢太甜，甜过头会明显降低兴致。`
  - `retrieval_text`: `不喜欢太甜 偏好 甜度`

## 二、进阶机制（在基础之上叠加）

### 2.1 置信度 / 来源类型（影响排序）

- 为动态层条目维护**来源类元信息**（如用户直说、推断、合并产物、多轮印证等）。
- 可落为轻字段，例如：
  - `source_type`：`user_stated` / `multi_turn_inferred` / `merge_result` / `assistant_inferred`
  - `confidence`：低 / 中 / 高（或数值）
- 重排时倾向**更可信来源**，减少「记错还排很前」对信任的伤害；**不以加条数为目标**。

### 2.2 离线慢整理

- **定时批处理**对动态层做合并、淘汰、标候选等，与实时对话**解耦**，不占主路径延迟。
- 分工建议：**网关先粗筛，DS 再精修**。
  - 网关负责便宜规则：embedding 近邻、tag/关键词重叠、低权重陈旧项、疑似冲突项、疑似平行条目。
  - DS 只处理被筛出的少量小簇：判断是否同主题、该 merge / keep / drop / rewrite，以及给出重写后的单条内容。
- 不建议把“压平同主题多条并存”硬塞进实时链路；更适合在离线整理里慢慢做干净。
- 每次慢整理只处理**一小批疑似脏区**，例如 10-20 组候选，不全量重写整个动态层。
- 触发方式建议采用**固定时间 + 条件触发**混合：
  - 固定时间：如每天凌晨，或每 12/24 小时跑一次，保证池子有人定期收拾。
  - 条件触发：当动态层条数超过阈值、疑似重复条目变多、低权重旧条堆积明显、或近期 `new` 明显多于 `merge` 时，额外补跑一次。
- 默认倾向先上**朴素版本**：先定时跑，再补一个“条数超阈值时加跑”的条件，不必一开始把触发器做复杂。
- 若实时链路里本该 merge 却仍出现平行条目，可先标成“待慢整理候选”，由离线批处理统一收口，而不是在主路径里强补第二刀。

### 2.3 类比式召回槽

- 暂列为**远期实验项**，当前**不作为主线优先级**。
- 原因：在基础检索稳定性仍需继续打磨时，过早引入弱指代 / 类比 / 情绪模式召回，容易放大误召回。
- 若后续要尝试，应以**强约束、默认关闭**的方式接入，仅在普通事实检索明显失效时作为低权重补充候选，而不是主路径。

### 2.4 近期优先级（当前共识）

1. 让 **merge** 更稳定地输出“当前理解”，而不是旧句拼接。
2. 为动态层条目补 **`source_type / confidence`**，帮助排序与后续整理。
3. 优先提升**基础检索稳定性**：
   - 重点不在加更飘的召回，而在把关键词提取得更准
   - 可通过“粗关键词 → 关键词收敛层 → 检索”的两段式来做
   - `retrieval_text` 作为检索辅助字段可保留推进
4. 上线**离线慢整理**，采用“网关粗筛 + DS 精修”，并用“固定时间 + 条件触发”混合调度。
5. 边缘淘汰继续细化，但优先级可略低于前几项。
6. 类比式召回暂不作为当前主线，只保留为远期实验项。

## 三、情感变化体察（非「禁令条例」式规则）

### 3.1 取向

- 目标不是再叠一套「什么词出现就怎样」的**显性规则**（容易变枷锁），而是让系统多带一点**对「人这一会儿怎么在变」的体感**。
- 真生气却被当成开玩笑、与「玩笑式拒绝被当成永久禁令」**同属一类问题**：都是对**情感阶段与变化方向**读不准。

### 3.2 机制方向（示意，实现可多种形态）

以下均为**增强「变化感」的载体**，而非在用户侧展示一长串条例：

1. **短程情感轨迹**  
   用**极少 token** 维护最近若干轮的整体走向（例如从松到紧、从玩到认真、从堵到软），每轮可更新。模型看到的是**连续曲线的一小段摘要**，不是关键词触发器。

2. **「与上一轮相比」的对照**  
   显式带一句**变化方向**（语气、密度、情绪词层面的对比），帮助区分「还在闹」和「已经当真」——由模型或轻量环节生成**描述性**信息，而不是 if-else 规则表。

3. **生成前的「当下更需要什么」侧写（软）**  
   一行**倾向性提示**（更需承接 / 更需空间 / 可轻松接梗），允许偶错；**不**与永久禁令绑定，**不**把单次判断写成不可撤销的硬状态。

4. **从互动里慢慢长的敏感带（软、慢）**  
   不是「禁词表」，而是：某些互动模式后曾出现情绪陡变，以**低权重、可覆盖**的方式进入侧写或排序微调，随后续正向互动**自然淡化**。

### 3.3 与记忆分层的关系

- 中期事实与约定仍在**动态层**；**情感轨迹 / 对照侧写**可以是**独立短字段、短块或与注入拼接**，核心是**服务当轮体感**，不必事事落成长期条。
- 与已落地的「心事」「检索前方向规划」等可衔接：规划侧重**找什么记忆**，本节侧重**这一会儿人可能处在什么相位**——二者互补，仍避免叠成厚规则书。

## 五、当前落地进度（代码现状）

### 5.1 已落地

1. **关键词收敛层（第一版）**
   - 已在检索入口增加“粗关键词 → 收敛短语”的两段式。
   - 当前以规则为主，优先提取态度 / 感受 / 偏好短语，其次才是事实 / 状态短语。

2. **`retrieval_text`**
   - 动态层记忆已支持内部检索字段 `retrieval_text`。
   - 新写入 / merge 时会同步生成；旧数据缺失时会在读取或离线整理时补齐。
   - 检索与向量索引已优先利用该字段。

3. **检索调试可视化**
   - MiniApp 记忆调试页已能显示：
     - `retrieval_query`
     - 收敛后的关键词
     - `phrase / raw` 区分
     - 命中记忆的 `retrieval_text`

4. **离线慢整理（第一版）**
   - 已支持：
     - 为旧记忆补 `retrieval_text`
     - 清理时间久、低权重、未进入核心缓存层的边缘记忆
     - 用同 tag 下 `retrieval_text` 的向量近邻粗筛疑似平行条
     - 对少量候选组调用 DS 做 `merge / keep` 判断
   - 当前实现偏保守，目标是先收一部分明显平行条，而不是重写整个动态层。

5. **离线整理入口**
   - 已有手动入口：
     - MiniApp 按钮
     - `POST /miniapp-api/memory-maintenance`
     - `scripts/run_memory_maintenance.py`

### 5.2 下一阶段

1. **`source_type / confidence`**
   - 方案已定，但代码尚未接入动态层结构、排序与整理逻辑。

2. **`importance` 彻底收死**
   - 方向已定为“首次写入时确定，后续主要靠 `mention_count / last_mentioned` 变化”。
   - 代码层还需要彻底收口，避免 merge 时继续覆盖 `importance`。

3. **离线慢整理继续完善**
   - 当前已有“粗筛 + 少量 DS merge 判断”。
   - 下一阶段再补：
     - 更稳的候选分组策略
     - 更完整的 DS 精修结果应用
     - 更细的节流、分批与失败重试

4. **自动调度**
   - 目前离线整理可手动触发，但尚未接入“固定时间 + 条件触发”的自动运行机制。

5. **类比式召回**
   - 仍保留为远期实验项，当前不进入主线实现。
