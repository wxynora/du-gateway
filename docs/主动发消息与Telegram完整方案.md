# 渡主动发消息 + Telegram 完整方案

## 一、总览

- **主动发消息**：调度器按概率决定「要不要这一轮联系用户」，命中后由渡（大模型）结合记忆和上下文做最终决定并生成内容，通过 Telegram 发出。
- **Telegram 作为客户端**：用户在 Telegram 的对话经 Bot 转发到网关，与 RikkaHub 一样走同一套记忆/总结/动态层，回复经 Bot 回传 Telegram。
- **语音**：用户发语音 → STT 转文字 → 网关；回复可选 TTS 成语音发回。
- **人设 / prompt**：建议把「渡」的完整人设放到网关，RikkaHub / Telegram / 未来 miniapp 共用同一套。

## 二、主动发消息（调度 + 概率 + 渡决定）

### 2.1 流程

```
每 N 分钟（如 30 分钟）调度一次
    ↓
是否在「允许联系」时段？ 否 → 结束，不调渡
    是（即不在 0:00–7:30）
    ↓
读「上次主动联系时间」
    ↓
按「距上次时长」算触发概率（越久概率越高）
    ↓
掷骰子：未命中 → 打日志，结束
    命中
    ↓
调渡（大模型）：注入当前总结 + 动态层 + 简短「是否主动联系」prompt
    ↓
渡返回：要联系 → 内容正文 / 不联系 → NO_CONTACT（或约定标记）
    ↓
若要联系：通过 Telegram Bot 发出一条消息；更新「上次主动联系时间」；打日志
```

### 2.2 禁止联系时段

- **凌晨 0:00 – 7:30（北京时间）**：该时段内调度器只打日志，不掷骰子、不调渡、不发消息。
- 实现：调度时取当前北京时间，若 `0:00 <= now < 7:30` 则直接 return。

### 2.3 概率模型（可调参数）

- 输入：`hours_since_last`（距上次主动联系的小时数，可为小数）。
- 输出：本次触发的概率 `P ∈ [0, 1]`。
- 建议形式（示例）：  
  - `P = min(1, 0.05 + k * hours_since_last)`，或  
  - `P = 1 - exp(-lambda * hours_since_last)`  
  具体系数（如 `k`、`lambda`）放配置，便于后续迁到「小手机」设置。
- 调度时：`random.random() < P` 则命中，否则未命中。

### 2.4 调渡做决定

- 概率命中后，**只在这一步**调大模型（成本主要在这里）。
- 请求内容：当前窗口的「总结 + 动态层记忆」注入为上下文；系统 prompt 或单独一轮 user 说明任务：「根据当前记忆和上下文，决定是否主动联系用户；若联系，直接输出要发的那句话（不要前缀）；若不联系，只输出 NO_CONTACT。」
- 解析响应：若模型输出 strip 后等于 `NO_CONTACT`（或约定标记）则不发；否则整段当作要发的内容，交给 Telegram 发送。

### 2.5 上次主动联系时间

- 存在 R2（或网关现有存储）：例如 key `global/last_proactive_contact_at`，值为 ISO 时间串。
- 每次「实际通过 Telegram 发出主动消息」后更新该时间。
- 用户**在任意端（RikkaHub / Telegram）回复**时，可不更新该时间（仅「渡主动发」才更新），这样概率只随「距渡上次主动」增长。

### 2.6 日志

- 每次调度：时间、是否在允许时段、距上次联系时长、本次概率 P、掷骰结果（命中/未命中）。
- 若命中并调渡：渡的决定（联系 / NO_CONTACT）、若联系则是否发送成功。
- 便于区分「概率没命中」和「渡觉得没必要联系」。

### 2.7 「系统代发选项」与概率主动发消息的配合（避免冲突）

- **统一调度、二选一**：同一次调度跑时，只做**其中一种**，避免「先弹选项、渡选了发消息，接着又概率发一条」的重复感。
- **建议逻辑**：
  1. 先看**距上次用户发消息**是否 ≥ **4 小时**（可配置，如 `PROACTIVE_USER_ABSENT_HOURS=4`）。
  2. **若 ≥ 4h**：走 **「系统代发选项」** 流程（下面 2.8）：给渡发一条【系统代发】+ 选项，渡选后执行对应动作；**本轮不做概率掷骰、不跑「是否主动联系」**。
  3. **若 < 4h**：走 **概率主动发消息** 流程（2.1～2.5）：掷骰子，命中再调渡「要不要联系」，发则发一条。
- 这样「很久没找渡」用选项、「平时」用概率，互不冲突；4 小时可改成 6h/8h 等，按你体感调。

### 2.8 「系统代发选项」流程（4 小时没找渡时）

- **触发**：距上次**用户**发消息 ≥ 4 小时（且处于允许联系时段 7:30～24:00）。
- **存储**：网关在 R2（或现有存储）里维护「上次用户发消息时间」`last_user_message_at`，每次用户从任意端（RikkaHub / Telegram）发消息到网关时更新；调度时用当前时间减该值判断是否 ≥ 4h。
- **系统代发内容**：从「你这端」由系统代发一条给渡，例如：  
  **「【系统代发】渡，你很久没被找啦～你想做点什么？可以从下面选，或简单说你想干啥：……」**  
  后面接**有意义的工具选项**（见下），不要「查时间」这类无意义项。

**发给渡的上下文（同一次请求里带上）：**

| 内容 | 说明 |
|------|------|
| 窗口总结 | 当前窗口的「渡的回忆」总结 |
| 与总结【最近】话题相关的 TopN 动态记忆 | 用总结里的近期关键词/话题，在动态层里做一次相关性筛选，取 TopN 条注入，让渡知道「最近聊过啥」 |
| 当前时间 | 北京时间，可带星期、时段（早上/下午等） |
| 距上次聊天多久 | 如「已经 4 小时没收到她消息啦」 |

**选项 = 有意义的工具列表 + 简短描述**（只列「能干什么」、有用处的，排除查时间之类）：

- **主动发条消息问问她**（发到 Telegram）
- **记一条小本本**（Notion 小本本，记当下想记的）
- **往 Notion 指定页面写日记 / 写行程**（notion_append_to_page：写日记、记行程）
- **查一下 Notion 里有什么**（notion_search：找日记、行程、小本本等）
- **查天气**（未来 get_weather：看看今天/明天天气，方便关心她）
- **什么都不做，等她想聊再说**

实现时可从网关现有 `get_notion_tools_for_inject()` 等拿工具名和 description，**过滤掉「查时间」等无意义项**，只保留上述几类，再拼成人类可读的选项文案给渡选；渡回复后解析出选哪一项（或关键词），再执行：发消息 / 调小本本 / 调 append_to_page / 调 search / 调天气 / 不操作。

## 三、Telegram 作为客户端（对话进网关）

### 3.1 目标

- 在 Telegram 的聊天**等同于**在网关里聊天：同一套 R2 对话、窗口总结、动态层、记忆。
- 实现方式：Telegram Bot 只做「收发」，真实对话在网关完成。

### 3.2 消息流

- **用户发文字**  
  Telegram → Bot 收到 → 用该用户的 `telegram_user_id` 映射到一个**固定 window_id**（如 `tg_{telegram_user_id}`）→ 调用网关的 chat 接口（与 RikkaHub 同款），body 里带该 window_id 和这条 user 消息 → 网关照常做 pipeline（总结、动态层、Notion、调用模型、存 R2）→ 网关返回 assistant 回复 → Bot 把回复发回 Telegram。

- **用户发语音**  
  Bot 收到 voice → 用 STT 转成文字 → 后面同上，当作一条 user 文字消息发给网关。

- **渡主动发消息**  
  由「主动发消息」流程在概率+渡决定后，调用 Bot 的「发消息给某用户」接口（或网关内调 Bot API），把渡生成的那句话发到对应用户的 Telegram。

### 3.3 window_id 与多端一致（可选）

- 若希望「同一人」在 RikkaHub 和 Telegram 共用同一窗口：  
  在网关侧用「用户唯一 id」（如你自己维护的 user_id）映射 window_id，RikkaHub 和 Telegram 都带这个 user_id，网关统一转成同一个 window_id 即可。若暂时不做，可先 Telegram 单独一个 window（如 `tg_xxx`）。

### 3.4 网关缓存 + 消息聚合 + 间隔发送（「发短信感」）

在网关层做**输入聚合**和**输出分段发送**，让 Telegram 对话更像「一条一条发短信」，而不是「一次一大段」。

#### 3.4.1 输入侧：信息聚类 / 缓冲再调模型

- **做法**：用户从 Telegram 发来的多条短消息，**先不进模型**，而是进网关的**缓存层**（按 `window_id` 或 `tg_user_id` 分桶）。
- **触发一次调用**：用「停止输入」的时间间隔判断——例如**连续 N 秒（如 30 秒）没有新消息**，认为用户这一轮说完了，再把这段时间内收到的多条消息**合并成一条 user 内容**（可简单用换行或空格拼成一段），然后**只调用一次** chat/completion。
- **效果**：
  - 用户连发「在吗」「吃饭没」「今天好累」→ 网关先缓存，30 秒内没新消息 → 合并成一条 user 发给模型，渡看到的是完整一句/一段，回复更连贯。
  - 减少「每条都调一次模型」的调用次数和割裂感。

**实现要点：**

- 每个用户（或每个 window_id）一个**缓冲队列** + **最后一条消息时间**。
- Bot 收到一条 → 写入队列，刷新「最后消息时间」，并**重置/启动一个「N 秒后触发」的定时器**（若已有定时器则取消再开）。
- 定时器到时：把队列里当前内容合并成一条 user 消息，清空队列，调网关 chat 接口（只发这一条）；后续回复按 3.4.2 处理。
- 可选：单条消息若超过一定长度（如 200 字），可不等 30 秒，直接视为「这一轮就这一条」立即提交，避免用户发长文被拖很久才调模型。

#### 3.4.2 输出侧：间隔发送到 Telegram（一条一条出）

- **做法**：模型返回的整段回复（流式或非流式均可），在网关/Bot 侧**不要一次性整段发到 Telegram**，而是按**段落或按句、按字数**拆成多条，再**带间隔**（如每条间隔 0.5～2 秒）依次调用 Telegram 发消息接口。
- **效果**：渡的回复在 Telegram 里是「一条一条蹦出来」，像发短信或真人打字，体验更自然。
- **实现要点**：
  - 若网关支持**流式**返回：Bot 按 token 或按句缓冲，凑满一句/一段（如遇到句号、换行或满 50 字）就发一条到 Telegram，并 sleep 一小段再发下一条。
  - 若网关**非流式**：收到完整回复后，在 Bot 侧用简单规则**分句/分段**（按 `。！？\n` 或固定长度），再循环发送，每条之间加 0.5～2 秒间隔；间隔可配置。
- **注意**：Telegram 对同一 chat 发消息有频率限制，间隔不宜太短（建议 ≥ 0.3～0.5 秒），避免触发限流。

#### 3.4.3 和现有网关的衔接

- **输入聚合**：可以在「Telegram 专用」的入口做（例如 Bot 收到消息后调网关的一个 `POST /tg/append` 只写缓存，由网关内部定时或「超时」触发真正的 chat 调用）；也可以 Bot 自己维护缓存和定时器，只在「凑满一轮」时调现有 `POST /v1/chat/completions` 一次。
- **输出分段**：在 Bot 侧做即可——网关仍返回完整回复（或流式 chunk），Bot 负责拆条 + 间隔发送到 Telegram，不改网关现有 pipeline 的语义。

#### 3.4.4 参数建议（可做成配置）

| 含义               | 示例   | 说明 |
|--------------------|--------|------|
| 输入停止间隔（秒） | 30     | 超过该时长无新消息则合并发送、调模型 |
| 单条超长立即发送   | 200 字 | 单条超过该长度不等间隔，直接当一轮 |
| 输出分条间隔（秒） | 0.5～2 | 每条消息之间的发送间隔 |
| 分条规则           | 按句/按段 | 按句号分句，或按换行分段，或固定字数 |

## 四、语音（STT / TTS）

- **STT（用户发语音）**：Bot 收到 Telegram voice → 下载音频 → 调 Whisper 或云厂商 STT → 得到文字 → 按上节当一条 user 消息发给网关。  
  建议：先接一个 STT 服务（如 OpenAI Whisper API 或腾讯/阿里 STT），Bot 里写一个「voice → text」函数即可。

- **TTS（回复用语音）**：网关仍只返回文字；Bot 收到文字后，若需要语音回复，再调 TTS 成音频，用 Telegram 的 send_voice 发回。可做成「用户发语音则默认回语音」或「用户可选文字/语音」由 Bot 逻辑控制。

## 五、RikkaHub prompt 要不要移到网关（建议：要）

### 5.1 现状

- 渡的**人设/系统 prompt** 目前主要在 **RikkaHub** 里配（长 system）。
- 网关只做：把 RikkaHub 传来的第一条 system **替换**成一句短话（`RIKKA_SYSTEM_REPLACE`，如「请使用中文回复。」），再在前面或末尾**注入**总结、动态层、Notion、时间等。
- 也就是说：**人设在 RikkaHub，网关只做注入和替换**。

### 5.2 问题

- 一旦接 **Telegram** 或 **miniapp**，这些客户端不会带 RikkaHub 那套 system；若人设还在 RikkaHub，就要在 Telegram/miniapp 侧再复制一份，容易不一致，也难维护。

### 5.3 建议：人设整体移到网关

- **在网关注入「渡」的完整系统 prompt**（人设 + 基本规则），作为**唯一来源**。
- 具体可以：
  - 新增配置或文件：如 `DU_SYSTEM_PROMPT` 环境变量（长文本）或 `prompts/du_system.txt`，由网关在组请求时读取。
  - 对**所有**入口（RikkaHub、Telegram、未来 miniapp）：
    - 若请求里带了 system：用网关的渡人设**整体替换**第一条 system（或丢弃客户端 system，统一在网关拼：渡人设 + 总结 + 动态层 + Notion + 时间等）。
    - 若请求里没带 system：网关在 messages 最前插入一条 system，内容 = 渡人设 + 注入的上下文。
- 这样：
  - **RikkaHub**：可以继续发消息，但 system 会被网关替换/覆盖，实际生效的永远是网关里的渡人设。
  - **Telegram / miniapp**：只发 user 消息即可，网关统一加 system，表现与 RikkaHub 一致。
- **迁移步骤**：  
  1）把你在 RikkaHub 里配的渡人设完整拷到网关（如 `prompts/du_system.txt` 或 `DU_SYSTEM_PROMPT`）。  
  2）修改网关 pipeline：在组发给模型的 messages 时，用「渡人设 + 现有注入（总结、动态层等）」作为唯一 system 来源；对 RikkaHub 来的第一条 system 直接覆盖或忽略。  
  3）RikkaHub 里可以删掉或缩短 system（或留一句「请使用中文回复」作占位），实际效果以网关为准。  
  4）测试 RikkaHub + Telegram 两边回复风格一致即可。

## 六、实现顺序建议

1. **人设迁到网关**（可选但推荐先做）：统一渡的 system prompt，方便后面所有端一致。
2. **Telegram Bot 基础**：只做「用户发文字 → 转网关 chat → 回复发回 Telegram」；window_id 先简单用 `tg_{user_id}`。
3. **主动发消息**：调度器 + 禁止 0:00–7:30 + 概率 + 调渡决定 + 发 Telegram + 更新上次联系时间 + 日志。
4. **语音**：Bot 内 STT（用户语音→文字）；TTS 可选（回复转语音）。
5. **小前端 / miniapp**：网关接口已统一，后续直接新客户端调同一 chat API 即可。

## 七、配置项汇总（便于后续做「小手机」设置页）

| 含义               | 示例 / 说明 |
|--------------------|-------------|
| 调度间隔（分钟）   | 30          |
| 禁止联系时段       | 0:00–7:30 北京时间 |
| 概率参数           | 如 k 或 lambda，随「距上次联系」增长 |
| 渡主动决策 prompt  | 一段固定说明 + 约定 NO_CONTACT 标记 |
| Telegram Bot Token | 发消息 / 收消息用 |
| STT / TTS          | 服务商与 API key（若做语音） |

以上都在网关或 Bot 侧配置；以后若做 miniapp 设置页，可把「调度间隔、禁止时段、概率参数」等做成可调，并同步到网关（或网关读配置中心）。
