# 部署后测试清单

只验证本次部署涉及的真实链路。不要调用 `/admin/wipe_all`，也不要把生产 R2 当测试环境。

## 1. 服务状态

```bash
systemctl --no-pager --full status du-gateway.service
curl -fsS http://127.0.0.1:5000/health
```

主服务应为 `active (running)`，健康检查应返回成功。

## 2. 模型与 active 上游

```bash
curl -fsS http://127.0.0.1:5000/v1/models
```

确认模型列表能读取。模型探活只能检查连通性，不能覆盖 App 已保存的 active upstream / model。

## 3. 最近窗口与状态页

```bash
curl -fsS http://127.0.0.1:5000/admin/windows
curl -fsS http://127.0.0.1:5000/admin/status
```

`/admin/windows` 只返回最近窗口；`/admin/status` 用于检查 R2 可读、总结、动态记忆、核心缓存、记事本、最近窗口和关键配置状态。

## 4. 聊天入口

用本次实际改动涉及的入口发一条真实消息，确认：

- 请求进入正确窗口；
- 回复回到正确渠道；
- 流式正文、reasoning 和工具结果按该入口正常展示；
- 成功回复后才进入归档链路；
- 没有聊天白名单/黑名单后缀或自动分流。

## 5. 独立 worker

改到哪个入口，就检查哪个 worker：

- SumiTalk：`du-sumitalk-chat-worker.service`
- Telegram webhook：对应 webhook worker
- Telegram 主动唤醒：对应 proactive worker

不能只看 `du-gateway.service` 正常就判定异步入口已经恢复。

## 6. 最小回归

- 小改动只跑与本次行为相关的定向测试。
- 不默认跑全量脚本。
- 不触发生产 R2 写入、删除、迁移或清空。
- 最后检查 `journalctl` 中没有新异常，再结束部署验证。
