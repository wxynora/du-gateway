# SumiTalk 发图与本地缓存方案

## 结论

`SumiTalk` 的发图方案定成：

- 首轮发给模型时用 `data:image/...;base64,...`
- 网关存档不存 `base64`
- `SumiTalk` 聊天记录本地数据库优先
- 图片文件本地缓存用 `@capacitor/filesystem`
- 共享上下文延续靠图片描述，不靠 `base64`

这样做的原因：

- 首轮识图最稳
- 不把 `base64` 塞进历史和 `last4`
- 聊天界面能像正常聊天软件一样先显示再等网络
- 后续还能继续并到 QQ / 微信 / TG 同一个共享房间

## 本次方案要解决的问题

当前有 4 个问题：

1. `SumiTalk` 现在没有用户发图链路。
2. `SumiTalk` 发消息是先改页面内存，没先落本地库，中途切出去会丢这轮消息。
3. 共享上下文里，图片首轮模型能看，但 `last4` 只剩 `[图片]`。
4. 如果每次显示图片都重新从服务器拉，聊天体验会很差。

## 总原则

这条链分成 4 层，各管各的：

- 模型输入：负责“这一轮让渡看见图”
- 本地消息库：负责“像聊天软件一样不丢消息”
- 本地文件缓存：负责“图片显示快”
- 网关/R2：负责“共享上下文和跨入口一致”

不能混着做。

## 方案总览

### 1. 首轮识图

用户在 `SumiTalk` 里选图后：

- 前端先本地压缩出一份适合发模型的图片
- 转成 `data:image/...;base64,...`
- 作为 `image_url` part 发给 `/v1/chat/completions`

这一层只保证：

- 渡本轮能看见图
- 不依赖外链鉴权、URL 过期、跨域

这一层不负责长期存储。

### 2. 本地消息库

`SumiTalk` 的聊天显示要改成“本地数据库优先”，不是“React 内存优先”。

发送时流程改成：

1. 点发送瞬间，先写本地消息库。
2. 立刻插入一条 `user` 消息。
3. 再插入一条 `assistant pending`。
4. 页面直接从本地消息库刷新显示。
5. 请求回来后，把 `assistant pending` 更新成正式回复或失败状态。

这样即使：

- 中途切后台
- 页面重建
- WebView 被回收

这轮消息也不会消失。

### 3. 本地图片缓存

图片文件缓存单独用 `@capacitor/filesystem`。

它只负责：

- 保存原图或聊天尺寸图
- 保存缩略图
- 给聊天气泡提供本地 `localUri`

它不负责：

- 聊天消息数据库
- 共享上下文
- 网关存档

一句话：

- 消息状态用本地数据库
- 图片文件用 `filesystem`

### 4. 网关/R2 存档

网关继续保持现状：

- 发给模型时图片保留
- 存 R2 时图片不存 `base64`
- 会话存档里还是 `[图片]`
- 图片描述继续单独异步写到 `images/*.txt`

也就是说：

- 首轮看图靠 `base64`
- 长期存档不存 `base64`

## 共享上下文怎么延续图片语义

现在的现状是：

- QQ / 微信 / TG / 未来的 `SumiTalk` 首轮发图，模型能看图
- 但进 R2 后，这轮会被清洗成 `[图片]`
- 后面的 `last4` 不会再带原图

所以共享上下文要补一层“图片描述回注”：

- 网关已经会生成图片描述并存到 `images/*.txt`
- 后续 `last4` 注入时，要把这张图对应的描述一起带回来
- 不是回注 `base64`
- 也不是回注原始图片 URL

目标效果：

- 首轮：模型直接看图
- 下一轮切到 QQ / 微信 / TG / `SumiTalk` 任一入口
- 模型至少知道“刚才那张图是什么”

## 为什么不把 base64 直接存历史

不这么做，原因很明确：

- 太大
- 聊天历史会膨胀
- `last4` 很容易把上下文撑爆
- Dexie、R2、调试快照都会变脏

所以这版明确规定：

- `base64` 只存在“这次请求”里
- 不进本地聊天历史
- 不进 `/sumitalk-history`
- 不进 R2 对话轮次

## 为什么不只存本地文件

也不能只存本地，原因是：

- 共享房间是 QQ / 微信 / TG / `SumiTalk` 共用
- 只存本地文件，别的入口拿不到
- 网关也拿不到
- 换设备后也没了

所以主存储和缓存一定要分开：

- 主语义在网关/R2
- 本地文件只是缓存层

## 本地数据结构建议

### 本地消息表

`SumiTalk` 本地消息表至少要支持：

- `id`
- `windowId`
- `role`
- `status`
- `content`
- `reasoning`
- `tokenCount`
- `attachments`
- `createdAt`
- `updatedAt`

其中：

- `status` 取值：`pending` / `sent` / `failed`
- `attachments` 先只支持图片

### 图片附件结构

图片附件建议至少有：

- `id`
- `kind: "image"`
- `mime`
- `width`
- `height`
- `localUri`
- `thumbUri`
- `remoteKey`
- `remoteUrl`
- `desc`

说明：

- `localUri` / `thumbUri` 只给本机显示
- `remoteKey` / `remoteUrl` 只做远端引用
- `desc` 用于后续语义延续

## 远端展示历史怎么存

`/sumitalk-history` 远端不存图片本体。

只存：

- 文本消息
- 图片附件元数据
- 不存 `base64`
- 不存本地路径

这样远端展示历史仍然干净，可同步，可调试。

## 首版范围

首版只做这些：

- `SumiTalk` 用户发送单张静态图
- `jpg/png/webp`
- 本地消息持久化
- 本地图片缓存
- 模型首轮看图
- 图片描述参与共享上下文接力

首版不做：

- GIF
- 多图一次发送
- 视频
- 相机拍照
- 渡发表情包
- 跨设备完整回放历史图片

## 实现顺序

建议按下面顺序做，别一把梭：

### 第一阶段：消息底座

先把 `SumiTalk` 聊天改成：

- 发送即落本地数据库
- `assistant pending` 可恢复
- 切出去不会整轮丢

这一阶段不碰图片。

### 第二阶段：用户发图

再补：

- 选图入口
- 本地压缩
- `base64` 发模型
- 本地图片气泡显示

### 第三阶段：图片缓存

再接：

- `@capacitor/filesystem`
- 原图/缩略图本地缓存
- 聊天气泡优先本地显示

### 第四阶段：共享上下文延续

最后补：

- 图片描述与 round 关联
- `last4` 注入时补图片描述

## 这套方案里插件的角色

`@capacitor/filesystem` 要加，但角色很明确：

- 它是图片缓存插件
- 不是聊天消息数据库

所以：

- 消息不丢：靠本地消息库
- 图片秒开：靠 `filesystem`

不能拿 `filesystem` 代替消息数据库。

## 最终预期效果

做完后，`SumiTalk` 应该有这几个行为：

1. 你发文字或图片，消息立刻出现在本地会话里。
2. 中途切出去，这轮消息不会丢。
3. 图片第一次发给渡时，渡能直接看见图。
4. 图片在 `SumiTalk` 里再次打开时，优先走本地缓存，不重复拉远端。
5. 切到 QQ / 微信 / TG 再聊时，渡至少能接住这张图的语义，不会只剩一句空泛的 `[图片]`。

## 一句话总结

这套方案的核心不是“把图片存哪”，而是把职责拆干净：

- 首轮识图靠 `base64`
- 消息稳定靠本地数据库
- 图片速度靠 `filesystem`
- 共享延续靠图片描述
