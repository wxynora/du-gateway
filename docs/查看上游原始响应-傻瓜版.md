# 怎么知道网关收到的原始内容长什么样

## 一、RikkaHub 发来的原始请求（未经过任何处理）

网关在 **DEBUG** 下会打出**收到聊天请求时**的原始 body 和关键 headers（未做清洗、注入、转发前的样子），方便你做关键字段提取和匹配。

**操作**：`.env` 里设 `LOG_LEVEL=DEBUG`，重启网关，用 RikkaHub 发一条消息，看网关日志。会看到：
- `body_keys`：请求体顶层字段名
- `body_id` / `body_assistant_id` / `body_window_id`：body 里这几个 key 的值
- `headers`：如 X-Window-Id、X-Assistant-Id 等
- `messages`：前几条消息的 role、content 长度与预览
- `raw_body_sample`：整段 body 的 JSON（截到约 3000 字符）

看完改回 `LOG_LEVEL=INFO` 并重启。

---

## 二、上游 AI 的原始回复（未加黑名单前缀等）

网关会把上游（TARGET）的原始 JSON 响应在 **DEBUG 日志**里打出来，方便你核对格式和内容。

---

## 步骤（傻瓜版）（仅针对「上游 AI 回复」）

1. **打开 DEBUG 日志**  
   在 **.env** 里加一行（或把原来的 `LOG_LEVEL=INFO` 改成）：
   ```env
   LOG_LEVEL=DEBUG
   ```
2. **重启网关**，然后照常发一条聊天请求（RikkaHub 或 curl 都行）。
3. **看网关日志**（终端或 systemd/docker 的日志）。  
   会看到类似：
   ```text
   [Chat] DEBUG: 上游原始响应 top_keys=[...] choices[0].message keys=[...] content_preview=... full_sample={"id":"...","choices":[...],...}
   ```
   - **top_keys**：上游返回的 JSON 顶层有哪些字段。
   - **choices[0].message keys**：`choices[0].message` 里有哪些字段（例如 `content`、`role` 等）。
   - **content_preview**：回复正文前 200 字（或前 200 字符的摘要）。
   - **full_sample**：整段响应的 JSON 字符串（截到约 2500 字符），可直接看出完整结构和格式。

这样你看到的就是**网关收到的、未加「（黑名单）」等任何修改之前的**上游原始回复的样子与格式。

---

## 看完记得关掉 DEBUG

调试完后把 **.env** 改回：

```env
LOG_LEVEL=INFO
```

再重启网关，避免日志太多、影响性能。
