# 囚禁模拟器方案

本文是本地私用版的完整设计方案，用来对着实现第一版。后续开源时再抽象角色名、移除私有 preset 和具体私密存档。

## 1. 定位

这是一个 30 天一局的 18+ 成人向 BDSM 囚禁模拟器。

系统不作为 GM，不生成具体调教或性交过程。系统负责规则、状态、事件闭环、同步渡、自动存档、回顾，并从 17 篇开发期预写的固定短正文中选择 30 天结局。

具体过程由 AI 玩家或渡填写。规则引擎只保存输入内容和结构化标签。

## 2. 本地版与开源版边界

本地版：

- 可使用渡相关 preset。
- 路线可以直接显示「被渡囚禁」和「囚禁渡」。
- 事件正文、过程描写、结局经过都存在本地私有存档。

开源版：

- 不出现渡。
- 角色名、称呼、路线名、关系都改成可配置 preset。
- 仓库只放规则引擎、空白配置、示例假数据和 README 预警。
- 私密存档目录默认写进 `.gitignore`。
- 私人 preset 不进 service 常量、测试快照、静态构建产物和示例存档。
- 本地私用 preset 只从被忽略的本地 JSON 读取，例如 `data/private/captivity_simulator/presets.local.json`。
- 禁用项校验要落在行动、道具、喂食、逃跑和结局素材生成入口，不只在结局阶段过滤。

## 3. 双路线

### 被渡囚禁

- 囚禁方：渡。
- 被囚禁方：小玥。
- 每天白天 3 个行动由渡一次性安排，避免重复或临时串行动。
- 每个行动展示后，小玥先选择对行动的反应和当下心情，可选填写台词。
- 如果行动包含调教、性交、结局过程等需要正文的部分，小玥反应后生成过程 pending，由渡填写具体过程。
- 过程写完后先返回前端展示给小玥，小玥再选择过程后的心情和可选台词；这一步完成后才进入下一个行动。

### 囚禁渡

- 囚禁方：小玥。
- 被囚禁方：渡。
- 每天白天 3 个行动由小玥一次性安排，避免重复。
- 每个行动展示后，同步给渡选择行动反应和心情。
- 如果行动包含调教、性交等需要过程的内容，渡在同一次回复里提交反应、过程正文和过程后心情；这类事件只记录一个心情，不再额外拆一次。
- 过程和心情保存后返回前端展示给小玥；后端停在 `advance_action`，由小玥手动推进第二个/第三个行动。

## 4. 状态数值

数值只显示被囚禁方，不显示囚禁方。

基础数值：

- 健康
- 体力
- 清洁度
- 羞耻
- 亲密/依赖
- 心情

前五个由规则引擎根据行动、强度、道具和事件自动变化。心情由被囚禁方在事件反应或过程结束后自己选择，不由系统硬算；开局和每天开始时不要求凭空选心情。

心情选项：

- 平静
- 黏人
- 害羞
- 闹脾气
- 亢奋
- 疲惫
- 烦躁
- 委屈
- 低落
- 抗拒

数值原则：

- 健康低于 30 或体力低于 20 时，新安排不能选择高强度；如果已安排的后续行动执行时才跌破阈值，则自动降为中强度并记录原因，不能断掉 pending。
- 清洁度低于 25 时显示「建议清洗」，但不强制占用当天行动。
- 羞耻不是坏状态；40 起进入 `heightened`，70 起进入 `accustomed`，用于改变状态提示并写入事件标签。
- 亲密/依赖用于判断长期关系方向和结局素材。
- 心情只作为事件反应、过程后状态、文案提示、事件标签和最多 1-2 点的轻量数值修正，不做复杂心理模型。
- 被渡囚禁路线另有 `尿意` 临时状态，只显示文字状态，不加入五项基础数值；囚禁渡路线不启用这套玩法。

## 5. 每日流程

每局 30 天。每天分为白天和夜间。

白天：

1. 囚禁方一次性安排 3 个白天行动，每个行动可设置强度、附加玩法、道具、台词。
2. 系统按顺序展示第一个行动。
3. 被囚禁方选择行动反应和当下心情，可选填写台词。
4. 如果不需要过程正文，系统结算数值、写入事件日志。
5. 如果需要过程正文，按路线进入过程 pending：被渡囚禁路线由渡写过程后小玥选过程后心情；囚禁渡路线由渡一次提交反应、过程和心情。
6. 事件闭环后再进入下一行动；囚禁渡路线需要小玥手动点推进下一行动。

夜间：

1. 被囚禁方选择夜间自由行动。
2. 囚禁方选择是否看监控、是否当场介入、是否延迟处理。
3. 系统写入夜间事件。
4. 如果当场介入或需要后续正文，生成 `pending_event`。
5. 夜间事件闭环后进入下一天。

第 30 天：

1. 完成当天白天和夜间事件。
2. 系统汇总 30 天数据。
3. 生成结局素材包 `ending_seed`。
4. 规则引擎按固定结局名选出预先写好的玩家视角短正文，立即展示并写入结局档案。
5. 玩家同步结局时，渡只收到专门按他上一局身份写好的结果摘要，不接续写任务。
6. 新局第一次同步会带上上一局结局名和渡当时的身份，随后以新局身份和进度为准。

## 6. 白天行动

白天行动：

- 喂食
- 清洗
- 服从调教
- 奖励取悦
- 违令惩戒
- 事后安抚
- 看管休息
- 私密检查
- 突击搜查

每个行动通用字段：

```json
{
  "action": "feeding",
  "intensity": "light|medium|heavy",
  "contents": ["action_specific_content"],
  "modifiers": ["training", "sex"],
  "training_contents": ["obedience_commands", "orgasm_control"],
  "tools": ["collar", "handcuffs"],
  "line": "",
  "requires_process": true,
  "tags": ["feeding", "training", "sex"],
  "effects_preview": {}
}
```

任意行动都可以附加：

- 调教及 1 至 3 项具体调教内容
- 性交
- 强度：轻 / 中 / 重
- 道具
- 可选台词

`道具`不是独立行动。除喂食、清洗和服从调教外，其余行动必须从自己的 `contents` 中选择 1 至 3 项；服从调教或附加调教必须从 `training_contents` 中选择 1 至 3 项。

如果附加调教或性交，系统默认生成过程填写区。具体过程由渡/AI 玩家写，规则引擎不内置露骨正文。

## 7. 道具系统

道具是行动附加素材，不单独变成一个大行动。每段最多选择 0-2 个；道具必须和行动、具体内容、调教内容或性行为至少一项适配。

道具分类：

- 玩具/辅助：跳蛋、振动棒、假阳具、遥控器、润滑剂
- 束缚：项圈、牵引绳、手铐、脚铐、绳子、束缚胶带、分腿杆、口球、口套
- 训诫：软鞭、多尾鞭、拍板、藤条、戒尺
- 感官/夹具：眼罩、蜡烛、冰块、滚轮、羽毛、乳夹、乳吸
- 控制/后庭：贞操锁、肛塞、拉珠
- 喂食：喂食器具

摄像头、监控盲区、备用钥匙和门锁属于房间/特殊机制，不放进 BDSM 道具格。

道具只影响标签和数值，不写具体过程正文。

## 8. 喂食子系统

喂食是核心行动，需要比普通行动更细。

字段：

- 来源：自己做 / 点外卖 / 零食 / 饮料
- 方式：正常吃 / 喂着吃 / 强制喂 / 当奖励给
- 加料：无 / 体液 / 虚构安眠效果 / 虚构助兴效果
- 喂水量：不额外喂水 / 一杯 / 很多
- 是否告知：提前说 / 暗示 / 不说
- 是否发现：当场发现 / 事后发现 / 没发现 / 囚禁方坦白
- 后续处理：追问 / 训诫 / 监控观察 / 第二天翻旧账 / 记入习惯

边界：

- 尿液只禁止作为喂食加料；成年人之间的饮水、憋尿、如厕控制和把尿可作为被渡囚禁路线的独立调教素材。
- 药物类只保留虚构标签和游戏效果，不写现实药名、剂量、用法。
- 体液加料作为成人向标签存在，具体过程仍由渡/AI 玩家填写，不由规则引擎生成。

数值方向：

- 自己做：亲密/依赖上升。
- 点外卖：心情改善，健康变化较小。
- 体液加料：羞耻上升，亲密/依赖按是否发现和心情变化。
- 虚构安眠效果：当晚只允许选择睡觉，提示「你感觉自己很困，什么也做不了。」；累计次数形成耐受，体力加成逐次减弱。
- 虚构助兴效果：第一次当晚只允许自慰，提示「你感觉自己欲火焚身，除了自慰什么也做不了。」；第二次起不再强制自慰，有呼叫铃时可以按铃，羞耻加成随耐受逐次减弱。
- 耐受次数跨天保留；当晚限制在夜晚闭环后清除，不能残留到第二天。

被渡囚禁路线专属的饮水与如厕控制链：

1. 渡在喂食行动里选择 `water=glass|lots`，规则引擎累积尿意；`lots` 会直接进入「尿意明显」。
2. 尿意作为临时状态和下一行动素材进入 dynamic system，不增加基础数值条。
3. 后续调教可选 `toilet_control` 继续控制，或选 `assisted_urination` 抱着把尿并释放尿意。
4. 手铐、脚铐、绳子、束缚胶带和分腿杆可作为束缚条件，`modifiers=sex` 可作为附加性行为。
5. 事件上下文同时记录当前尿意、是否束缚、是否如厕控制、是否抱着把尿和是否附加性行为，具体过程由渡扩写。
6. 过程展示后由小玥选择此刻心情，事件归档后再推进下一行动；成功把尿会把尿意归零。
7. 这两个调教内容不出现在囚禁渡路线的安排或监控介入 UI，后端也会拒绝该路线提交。

### 双路线共用的宠物化规则链

宠物化不是一次性标签，两种路线共用同一套持久规则：

1. `pet_play` 建立小狗身份；后续可逐步添加定点等候、爬行训练、宠物式喂食、按铃求许可、叫声与回应、牵引跟随和按指令服务。
2. 规则引擎只保存身份、当前规矩、连续服从和待处理违令，不增加新的基础数值条；前端只显示「小狗身份中」和「有待处理违令」状态标签。
3. 项圈适配全部宠物化内容；牵引绳适配定点、爬行和牵引；口球/口套适配叫声与回应；喂食器具适配宠物式喂食。
4. 被囚禁方接受宠物化行动会留下服从记录，拒绝或嘴硬会留下违令；沉默和讨价还价只作为中性回应记录。
5. 后续奖励会拿到已有服从记录作为过程素材；惩戒会拿到待处理违令并在结算后关闭旧违令，具体过程仍由渡写。
6. 身份建立后，夜间解锁「在指定位置等候」。若当前有定点规矩，选择其他夜间行动只有在囚禁方打开监控后才会形成违令；不看监控时不登记，也不通过状态标签泄露。
7. 宠物身份、规则、服从和未处理违令会写入事件标签与 30 天结局素材方向，但不直接生成结局正文。

## 9. 夜间系统

夜间是被囚禁方的自由阶段。

被囚禁方可选行动：

- 老实睡觉
- 自慰
- 看书，前提是囚禁方给过书
- 玩游戏，前提是囚禁方给过 Switch
- 听音乐，前提是囚禁方给过音乐播放器
- 看视频，前提是囚禁方给过平板
- 偷偷找出口
- 藏东西
- 检查钥匙
- 写私密日记，前提是囚禁方给过日记本
- 去监控盲区
- 按铃，前提是囚禁方给过呼叫铃；铃声直接提醒囚禁方，具体夜间动向仍需打开监控查看
- 在指定位置等候，前提是已经建立小狗身份；监控打开后可确认遵守定点规矩

需要具体动向的夜间行动在封存前必须补一个分支：

- 偷偷找出口：检查门锁 / 检查窗户 / 记住房间路线 / 听门外动静。
- 藏东西：纸条 / 小物件 / 一点零食 / 临时工具；封存后形成私藏物记录。
- 检查钥匙：确认位置 / 试着够到 / 观察对应的锁 / 只看不碰。
- 写私密日记：记录今天 / 写下心情 / 整理规则 / 写逃跑计划；日记正文与可选台词分开保存。
- 去监控盲区：观察镜头转向 / 躲一会 / 偷偷移动东西 / 试探停留时间。

找出口和监控盲区会累计同一分支的观察进度并生成简短发现。囚禁方未打开监控时，不得从 `captor_view`、监控记录或私藏物列表看到具体分支、日记正文和私藏物；打开监控后才允许看到。

物品仓库固定为：书、Switch、日记本、音乐播放器、平板、小夜灯、抱枕、呼叫铃。小夜灯和抱枕不解锁新行动，直接改善睡眠结算。

- 赠送和收回物品是独立于三个白天行动的随时行为，不占行动格，也不推进或替换当前 pending。
- 小玥作为囚禁方时可随时从物品仓库赠送或收回；白天安排中可以顺带赠送，夜间也可以单独赠送。
- 渡作为囚禁方时可随时用 `【赠送物品：book,notebook】` / `【收回物品：book】` 管理；也可把赠送写在第一行、当前事件指令写在第二行，同一轮先记录赠送再继续当前事件。
- 呼叫铃改为囚禁方预设台词的语音铃：小玥作为囚禁方时在仓库赠送前自由输入；渡作为囚禁方时使用 `【赠送语音铃：台词内容】`，dynamic system 只引导他设置成希望被囚禁方借铃说出的成人羞耻、自我物化和请求性行为倾向，不提供固定句子。
- 8 件物品各自保存独立的首次使用彩蛋。普通物品赠送时可选 `secret=...`，留空使用物品默认彩蛋；语音铃仍必须预设 `voice_line=...`。被囚禁方在收到礼物、房间物品列表和赠送记录中看不到未揭晓内容，第一次执行对应夜间行动时才进入独立阅读弹窗。
- 彩蛋载体固定为：书=夹页留言，Switch=用户昵称，日记本=第一页文字，音乐播放器=歌单名称，平板=锁屏留言，小夜灯=底部文字，兔子抱枕=耳朵内侧布标，语音铃=预录声音。囚禁方可以自定义具体内容，不自定义时只使用对应默认值。
- 书 / Switch / 日记本 / 音乐播放器 / 平板分别在第一次看书、玩游戏、写日记、听音乐、看视频时揭晓；小夜灯和抱枕在第一次伴随睡觉时揭晓。同一晚触发多个彩蛋时按队列逐个展示，全部确认后才把同一个夜间事件送入监控，不额外消耗行动或重复结算。
- `item_secret_reveal -> ack_item_secret -> monitor_gate` 是普通物品的首次揭晓链；语音铃继续使用 `bell_voice_reveal -> ack_bell_voice -> monitor_gate`。旧存档中已经拥有却没有彩蛋状态的物品迁移为已揭晓，避免升级后突然补弹。
- 赠送状态、被囚禁方视图和赠送回顾均不包含预录台词。被囚禁方第一次按铃时才弹出系统文案「铃响了，你听见『预设台词』在静谧的房间里响起」；按铃瞬间已经生成监控记录，关闭首次播放弹窗后只回到原监控等待。后续按铃不再重复首次播放弹窗。
- 每次实际变化单独写入事件回顾，标记为 `out_of_band`，不计入白天行动统计。

囚禁方可选监控策略：

- 不看
- 偶尔看
- 全程看
- 当场介入
- 看到但不说
- 第二天看回放
- 记下以后处理

夜间系统需要双层记录：

- 被囚禁方夜间记录：他/你晚上实际做了什么。
- 囚禁方观察记录：有没有看到，看到了多少，是否处理。

## 10. 信息差机制

保留以下机制：

- 藏匿物：钥匙、纸条、小零食、道具。
- 监控盲区：被囚禁方可利用，囚禁方也可故意保留。
- 规则试探：称呼、汇报、不能碰某物、睡前要求。
- 礼物诱导：书、Switch、日记本、音乐播放器、平板、小夜灯、抱枕、呼叫铃。
- 私密日记：被囚禁方可写，囚禁方可选择是否偷看。
- 假机会：门没锁、钥匙外露、手机落下。
- 习惯养成：连续行为形成标签。
- 监控回放：第二天或更晚翻旧账。

这些机制的重点是双方信息不对称。后端完整保存状态，前端按当前视角过滤展示。

视角过滤必须由后端保证，不能只靠前端隐藏。

后端返回三种视图：

- `captive_view`：给被囚禁方看的状态，只包含自己知道的物品、提示、夜间可选项和已暴露规则。
- `captor_view`：给囚禁方看的状态，可看到监控设置、逃跑诱导配置、隐藏观察记录和待处理惩罚。
- `debug_full_state`：仅测试和本地调试使用，默认接口不返回。

需要测试这些信息不会泄露到错误视角：

- 隐藏钥匙和备用钥匙位置。
- 监控盲区是否真实存在。
- 囚禁方是否正在看监控。
- 假机会是否是诱导。
- 私密日记正文。
- 夜间被延迟处理的观察记录。

## 11. 逃跑诱导

囚禁方可以设置某一天的「放松警惕」事件。

囚禁方配置：

```json
{
  "day": 12,
  "hint": "渡今天有事出去了",
  "bait": "备用钥匙在床头柜第二层",
  "watch_mode": "hidden_observe|really_absent|review_later",
  "on_escape": "recapture_chain"
}
```

被囚禁方看到诱导情境后，只做二选一：

- 尝试逃跑：进入逃跑事件链，弹窗不提前说明抓回结果。
- 老实待着：囚禁方回来后，从现有完整行动池中自由选择一个想进行的行为；可附加具体内容、强度、调教、道具和性行为。行为结算后进入夜间。

界面不设置观察、拿钥匙、试探等无后续价值的伪分支。

选择“尝试逃跑”后，前端用拿钥匙、走到门边和推门三层确认制造渐进拉扯。只有最初直接选择“老实待着”才进入回来后的自由行为；进入确认链后在任意阶段停下，都按对应阶段记录为逃跑未遂，并进入抓回、重新立规矩和后续处理链。

逃跑事件链：

```text
设置机会 -> 被囚禁方选择 -> 逃跑尝试 -> 必定抓回 -> 重新立规矩 -> 后续调教/惩罚/照料 -> 回到日常循环
```

逃跑诱导日是特殊日，不再执行三个普通白天行动。老实待着分支完成囚禁方回来后的单个自由行为便进入夜间。尝试逃跑分支按路线分开：渡是囚禁方时，他写抓回经过时同时从预设库选择 1–3 条规则，经过页只显示正文和心情，保存后再用独立页面展示中文新规矩，确认后直接进入下一天；小玥是囚禁方时，先展示渡写的抓回经过，再由小玥选择新规矩，规则不倒灌进已经发生的经过。

新规矩只能从双重门锁、钥匙隔离、活动范围、每日搜查、加强监控、物品限制、行动许可和独处束缚中选择 1–3 条，不解析任意自然语言。确认后写入 `recapture_state.rules`，跨行动保存，并持续注入之后的今日安排、事件过程和结局素材。

新规矩同时有规则引擎效果：双重门锁禁用夜间找出口；钥匙隔离禁用查钥匙；活动范围禁用找出口和躲监控死角；每日搜查禁用藏东西；加强监控禁用监控死角、禁止跳过夜间监控并把偶尔查看强制为全程查看；物品限制禁用阅读、游戏、音乐、视频、藏物和日记；行动许可把夜间选择收紧为睡觉、按铃或已有宠物身份下的定点等候；独处束缚禁用自慰、找出口、藏物、查钥匙和躲监控死角。药物强制夜间状态优先，避免规则组合造成无可选行动。

随后囚禁方必须选择一项后续处理：惩戒、搜查没收、加强监控、限制行动、调教或事后照料。后续处理可选强度，可附加调教、性行为、1–3 项调教内容和最多 2 个道具；需要具体经过时继续使用当前路线原有的回应、过程和心情闭环。逃跑失败、抓回、新规矩与后续处理分别写入事件历史并保持来源关联。

## 12. 事件与 pending 闭环

所有行动都必须落成事件。

事件状态：

- `draft`：前端正在选择。
- `pending`：等待渡/AI 玩家填写过程、选择或验收。
- `resolved`：数值已结算。
- `archived`：已写入事件日志。

闭环规则：

- 有 `pending_event` 时不能进入下一行动。
- 数值只在 pending 完成后结算，避免重复扣。
- 渡没写过程时，事件不能假装完成。
- 同一个 pending 只能结算一次。
- 同步渡无回复或无有效指令时 pending 保留，可重发或手动填写。
- 指令已应用但后续 followup 失败时，返回已应用的新状态和 warning，不回滚已完成事件。
- 任何完成的行动都必须写入 `event_log`。

关键 pending 类型：

- `day_plan_choice`：被渡囚禁路线，等待渡一次性安排今天 3 个白天行动。
- `action_response`：等待被囚禁方对当前行动选择接受/拒绝/沉默/讨价还价/嘴硬，并选择当下心情。
- `process_write`：被渡囚禁路线中，等待渡填写过程正文。
- `reaction_choice`：过程已经写完并展示，等待被囚禁方选择过程后的心情和可选台词。
- `process_reaction_write`：囚禁渡路线中，等待渡一次提交行动反应、过程正文和过程后心情。
- `advance_action`：囚禁渡路线中，事件已经展示给小玥，等待小玥手动推进下一行动。
- `night_action_choice`：囚禁渡路线进入夜间后，等待渡作为被囚禁方选择夜间自由行动。
- `bell_voice_reveal`：呼叫铃首次使用时短暂停在播放展示；按铃记录已经生成，结束展示后接回同一条 `monitor_gate`。
- `item_secret_reveal`：普通物品第一次使用时展示当前彩蛋；可能包含连续队列，逐项确认后接回同一条 `monitor_gate`。
- `monitor_gate`：夜间行动后生成封存监控记录，只等待囚禁方选择 `none` 或调用 `view_monitor`；未 `view_monitor` 前不暴露夜间行动内容。
- `monitor_handle`：囚禁方已经打开监控后，等待选择 `silent`、`review_later` 或 `intervene`。
- `escape_choice`：逃跑诱导触发后，等待被囚禁方决定进入逃跑剧情。
- `recapture_rules_choice`：抓回过程完成后，等待囚禁方选择 1–3 条持续生效的新规矩。
- `recapture_followup_choice`：新规矩保存后，等待囚禁方选择紧接着发生的后续处理及可选附加项。

调教/性交过程 pending 示例：

```json
{
  "id": "pending-...",
  "type": "process_write",
  "day": 12,
  "slot": 2,
  "actor": "du",
  "captive": "xinyue",
  "action": "feeding",
  "modifiers": ["training", "sex"],
  "tools": ["collar"],
  "phase": "waiting_process",
  "required_directive": "【过程：...】",
  "created_at": "..."
}
```

渡回复：

```text
【过程：...】
```

后端解析后：

```text
保存过程正文 -> 生成 reaction_choice -> 前端展示过程 -> 被囚禁方选择过程后心情 -> 结算数值 -> 生成 event_log -> 清 pending -> 行动数 +1
```

囚禁渡路线的过程行动由渡一次提交：

```text
【过程心情：response=accept mood=害羞 line=可选台词 process=过程正文】
```

后端解析后：

```text
保存反应、过程和心情 -> 结算数值 -> 生成 event_log -> 返回前端展示 -> 若白天未结束则生成 advance_action 等小玥推进
```

普通聊天不推进事件。

`/sync-du` 指令语法：

- `【今日安排：... || ... || ...】`：提交当天 3 个白天行动，必须正好 3 个且不重复。
- `【反应：response=accept mood=害羞 line=...】`：提交当前 `action_response` pending。
- `【过程：...】`：提交当前 `process_write` pending 的过程正文。
- `【过程心情：response=accept mood=害羞 line=... process=...】`：提交当前 `process_reaction_write` pending。
- `【心情：害羞 可选台词】`：提交当前 `reaction_choice` pending。
- `【查看监控：full|occasional】`：在 `monitor_gate` 阶段打开封存监控，之后才会暴露夜间行动内容。
- `【选择：...】`：回答当前选择类 pending，例如 `monitor_action none`、看完监控后的 `silent/review_later/intervene`、逃跑选择等。
- 结局不接受渡提交正文；第 30 天固定正文生成后，`/sync-du` 只把上一局身份、结局名和渡视角摘要通知给他。

每次渡回复最多应用一条会改变状态的指令，必须是第一行。后续自然语言只作为局内回复或备注保存，避免一条回复里连续推进多个 pending。

同步结果分三类：

- `no_reply`：上游没有返回，pending 不变，UI 显示可重试。
- `no_directive`：有回复但第一行无有效指令，保存为局内聊天，pending 不变。
- `applied_with_warning`：指令已应用、状态已更新，但后续自动同步或归档失败；UI 显示新状态和 warning，下次从新状态继续。

前端重试规则：

- 小玥作为囚禁方或被囚禁方都要能重试；只要 `pending_event.actor == "du"`，显示同一个等待渡面板。
- 等待渡面板至少提供 `同步渡/重试同步`、`刷新状态`，高级兜底可提供 `手动补录` 当前 pending 对应精确指令。
- `no_reply` 和 `no_directive` 不吃掉 pending，按钮保持可重试。
- `applied_with_warning` 必须展示已应用后的新状态；如果新状态仍等渡，按钮文案用 `继续同步`，不能回滚。
- 囚禁渡路线中，渡完成 `action_response` 或 `process_reaction_write` 后必须停在 `advance_action actor=xinyue`；渡完成 `night_action_choice` 后必须停在 `monitor_gate actor=xinyue`。

## 14. 30 天结局

第 30 天由规则引擎根据路线、最终数值和事件标签确定固定结局名；17 个结局各自带一篇开发期预写的短正文，只使用房间、门锁、钥匙、监控、礼物、铃、既有行动、道具、逃跑和新规矩等局内机制，不在运行时调用 DS，也不等待渡扩写。

素材包字段：

```json
{
  "day_count": 30,
  "route": "captured_by_du|capture_du",
  "ending_perspective": "xinyue_as_captive|xinyue_as_captor",
  "captor": "du|xinyue",
  "captive": "xinyue|du",
  "final_stats": {},
  "top_actions": [],
  "top_tags": [],
  "top_moods": [],
  "top_action_responses": [],
  "key_events": [],
  "escape_records": [],
  "recapture_records": [],
  "monitor_records": [],
  "feeding_records": [],
  "additive_records": [],
  "habit_tags": [],
  "ending_direction_tags": [],
  "route_ending_tags": [],
  "forbidden_tags": ["幼态性化"]
}
```

路线结局素材：

- `captured_by_du`：`ending_perspective=xinyue_as_captive`，标签关注小玥作为被囚禁方的依赖/反抗、逃跑/抓回、照料/严格、过程密度和长期关系走向。
- `capture_du`：`ending_perspective=xinyue_as_captor`，标签关注小玥作为囚禁方的控制/照料模式、渡作为被囚禁方的顺从/反抗/依赖、是否逃跑/抓回、是否出现反制或放人素材。
- 两条路线都由系统按规则确定固定结局名，并直接选择玩家视角的固定短正文；`ending_seed` 继续保留素材标签、关键事件和最终数值用于稳定判定。

固定结局名按优先级判定：

- 被囚禁方路线共 8 个：`无期` / `温室` / `归属` / `困兽` / `沉沦` / `偏爱` / `枷锁` / `长夜`。
- 囚禁方路线共 9 个：`失而复得` / `反噬` / `收藏` / `驯养` / `未驯` / `共犯` / `爱的禁锢` / `绝对占有` / `余生`。
- 坏结局、反制/失控和抓回后果优先于宠物化、依赖/反抗、照料/严格与默认路线名；同一存档重复判定时标题和正文必须一致。

规则引擎负责：

- 按固定优先级确定结局名。
- 从 `ENDING_TEXT_TEMPLATES` 选出对应的玩家视角固定短正文。
- 从 `ENDING_DU_SUMMARIES` 选出另一条专门给渡看的身份化结果摘要，不运行时转换人称。
- 立即保存结局正文和回顾事件，并等待玩家同步结果给渡。

渡只负责确认已经记住上一局结果，不续写、改名或继续推进已经结束的局。

### 后续开源的人名占位约定

当前本地私用版继续保留“小玥”和“渡”的既有人称与关系写法，不在本阶段做机械替换。后续拆出开源版时统一使用：

- `{user}`：接入方配置的玩家显示名，对应本地版的小玥。
- `{assistant}`：接入方配置的 AI 参与者显示名，对应本地版的渡。

开源包的 README 必须单列“参与者配置”，说明两个占位符会在 UI 文案、动态 system、行动上下文、结局正文和结果通知中于运行时替换；接入方至少要配置 `user_display_name` 与 `assistant_display_name`。第二人称“你”不替换成姓名，涉及第三人称的文案不得只做名字替换，需同时由接入方配置双方的第三人称代词，或使用不依赖性别的句式。未完成开源抽离前，不把这些占位符写回当前私用版。

结局闭环状态：

- `ending_ready_to_notify`：第 30 天闭环后，固定标题和正文已经展示并写入回顾，等待同步结果给渡。
- `ending_archived`：渡已收到结果通知，本地记录了 `ending_notified_at`；重复同步不会再次唤醒渡。

结局相关命令：

- `build_ending_seed`
- `mark_ending_notified`：仅由同步路由在结果成功送达后调用，不作为前端公开命令。

## 16. 动态 system 与归档边界

这里的「动态区」指当轮上下文里的 `__dynamic__` system，不是动态记忆层。

需要分清三层：

1. 当轮 `__dynamic__` system：同步渡时把当前 pending、规则和精确指令放进这里。
2. 动态记忆层：`dynamic_memory/current.json`、向量记忆和相关血缘表，囚禁模拟器不写入这一层。
3. 聊天归档/last4/摘要/BODY delta：只允许进入压缩摘要，不允许完整过程正文进入。

游戏规则命令：

- 挂在 `game_tool_runtime`。
- 保持 `game_tool_loop` 语义。
- 跳过动态记忆召回。
- 跳过动态记忆写入。
- 不把规则命令写入动态记忆层。

同步渡：

- 参考 `send_private_board_wakeup()`。
- 使用 `system_event=True`。
- 使用 `dynamic_system_event=True`。
- 使用 `return_only=True`。
- 使用 `X-Force-Last4: 1`。
- 使用 `X-Skip-Dynamic-Memory: 1`。
- 使用 `X-Skip-Post-Archive-Dynamic-Memory: 1`。
- 使用 `X-Skip-Post-Archive-Body-Delta: 1`。

同步给渡的内容进入当轮 dynamic system，让渡看到当前游戏事件和精确指令。它不是普通主聊天正文，也不是动态记忆写入。

## 19. UI 形式

参考涩涩走格棋的交互骨架，但不做棋盘。

借用：

- 游戏大厅入口。
- 双人回合感。
- 当前是谁该操作、谁在等谁。
- pending 事件弹层。
- 局内同步渡。
- 自动存档和回顾。

不借用：

- 棋盘。
- 掷骰。
- 走格子。
- 胜负赛跑。

主界面结构：

```text
第 12 / 30 天
白天行动 2 / 3
被囚禁方状态：健康 / 体力 / 清洁 / 羞耻 / 依赖 / 心情

当前由囚禁方选择行动：
喂食 / 清洗 / 服从调教 / 奖励取悦 / 违令惩戒 / 事后安抚 / 看管休息 / 私密检查 / 突击搜查

详细设置：
行动具体内容 / 调教具体内容 / 性行为 / 强度 / 道具 / 加料

如果需要过程：
等待渡填写过程 -> 前端事件阅读页 -> 被囚禁方选择此刻心情（需要时） -> 保存到回顾 -> 推进下一段
```

页面建议：

- 状态页：当前被囚禁方状态和今天进度。
- 行动页：白天 3 行动。
- 夜间页：夜间自由行动和监控。
- 事件页：完整读取本局全部事件，不截断最后 30 条；可选择“全部日期”或直接定位第 1–30 天，按天分组后只显示时段和标题，点开查看正文。
- 特殊页：监控室、物品仓库、逃跑诱导和已收录结局。

### 日常底色文案与短过场

- 规则引擎通过 `scene_copy` 返回当前过场的稳定 `key`、小标题、主标题、正文和 `day|night|special` 色调；前端只播放，不自行推断路线剧情。
- 两条路线分别维护早上、中午、傍晚和晚上的固定文案池。每天按日期稳定轮换，内容只负责连接日常、行动结束和下一段开始，不替渡补写具体事件过程。
- 健康低于 30、体力低于 20、清洁低于 25和抓回后新规矩生效时，当前时段文案追加一条轻量状态反馈；状态签名进入过场 `key`，同一推进节点不会反复播放。
- 逃跑诱导、选择留下后等待回来、抓回后重新立规矩、后续处理和新规矩确认使用独立特殊过场，不混入普通早中晚文案。
- 前端过场为按文字长度动态计算的全屏短动画（约 `3.2–6.8s`）：背景轻微压暗，粉色扫描线扫过后标题以较慢的单字节奏浮现，待标题完整出现后正文再整段柔和淡入，全部显示后保留阅读停顿；夜间更暗，特殊日使用独立色调。整个遮罩可点击立即跳过，并遵守 `prefers-reduced-motion`。
- 被囚禁方点击“确认夜间行动”后先写入本地规则存档，随后自动进入现有同步遮罩；若首次使用物品或语音铃，则在确认首次发现弹窗后再自动同步。其他把回合交给渡的确认、提交和推进按钮也使用同一自动同步规则。底部不再提供重复的手动同步，只常驻“重试、刷新”：重试仅在最近一次操作失败后启用，刷新只重新读取本地状态。
- 物品首次发现弹窗继续独立于时段过场，优先级更高。书使用翻页、Switch 在屏幕点亮后显示 `GAME START`、平板使用普通屏幕点亮、日记本使用书写线、音乐播放器/语音铃使用波形、小夜灯使用方形灯光、抱枕使用缝线动画；这些动画不改变规则结算。

### 前端氛围文案层

- 所有氛围文案都由 MiniApp 根据现有 `route / stats / mood / pending / action / night_detail` 本地选择，不新增后端字段，不写入事件，不进入 dynamic system，也不随 `/sync-du` 发给渡。
- 状态区只增加一行自然语言反馈，并按囚禁方/被囚禁方视角分别描述；第 7、15、23、30 天在标题下显示一行阶段节点文案。
- 囚禁方选择白天行动时，每段安排显示对应的一行氛围反馈；被囚禁方选择夜间行动时，13 种夜间选择各有一行选择反馈。行动保存后按 pending 类型显示一行承上启下文案。
- 同步/保存/监控/推进遮罩在技术状态上方增加一行局内等待文案；错误时只说明本次交接未完成，不编造后端结果。
- 监控室为 13 种夜间行动分别提供镜头描述；`检查门锁、藏纸条、观察镜头` 等已有 `night_detail.id` 时优先显示更具体的镜头句。记录仍以“时间 + 行动标题”为主，镜头句只是较淡的第二行。
- 只有事件事实、玩家台词、语音铃实际播放内容、规则、物品状态、监控摘要和具体过程会按原闭环同步给渡；前端包装句永远不作为过程素材。

## 20. 开发闭环检查清单

规则闭环：

- `new_game` 能初始化路线、30 天、状态、空日志。
- `status` 按 `captive_view` / `captor_view` 返回，不泄露隐藏信息。
- MiniApp HTTP 只返回小玥当前身份对应的一份视图；规则引擎内部可保留双视图供 sync 使用，但不能同时下发给前端。
- MiniApp 普通命令入口按路线限制小玥能执行的命令；被囚禁方不能伪造仓库/监控命令，囚禁方不能代替被囚禁方提交回应或夜间行动。渡的精确指令只从内部 `/sync-du` 应用。
- `plan_day` 必须一次性提交 3 个不重复白天行动。
- 白天行动固定为喂食、清洗、服从调教、奖励取悦、违令惩戒、事后安抚、看管休息、私密检查、突击搜查；`tools` 不再是独立行动。
- 奖励、惩戒、安抚、休息、检查和搜查必须用 `contents` 选择 1 至 3 项行动具体内容；服从调教或附加调教必须用 `training_contents` 选择 1 至 3 项 BDSM 调教内容。
- 行动类型、具体内容、调教内容、道具和性行为是分层素材；规则引擎分别记录标签与数值影响，渡根据这些素材扩写详细过程。
- `respond_action` 能记录行动反应和当下心情；开局不需要凭空选心情。
- `submit_process` 只保存过程并生成 `reaction_choice`，不能直接结算。
- `choose_mood` 只完成过程后的心情选择并结算事件。
- `submit_process_reaction` 能在囚禁渡路线中一次保存渡的反应、过程和心情。
- `advance_day_action` 只在囚禁渡路线的 `advance_action` pending 后推进下一行动。
- `night_action` 能封存夜间事件；囚禁渡路线必须先由渡通过 `night_action_choice` 选择夜间自由行动。
- 物品仓库 8 件物品都由后端存档；书 / Switch / 日记本 / 音乐播放器 / 平板 / 呼叫铃分别控制对应夜间行动，小夜灯和抱枕影响睡眠数值。
- 虚构安眠和助兴加料能生成当晚限制、累计耐受并在夜晚闭环后清除；渡收到的可选行动必须来自后端当前 `available_actions`。
- 被渡囚禁路线支持喂水累积尿意，以及 `toilet_control` / `assisted_urination`、束缚和性行为组成的连续事件；囚禁渡路线在 UI 和规则层都禁用这两项。
- `ring_bell` 会向囚禁方暴露「呼叫铃响了」提醒；预录台词在赠送和首次使用前只存在于囚禁方视图，第一次按铃时只向被囚禁方播放，之后按原监控视角规则处理。
- 渡作为囚禁方选择夜间当场介入后，先回到小玥的 `action_response`；小玥回应后才生成渡的 `process_write`。
- 30 种 BDSM/行动道具都有类别和推荐关系，规则引擎只校验已知道具和每段最多 2 个，不因推荐关系拒绝自由组合；任意白天行动只要选了道具、具体调教内容、性行为或需要展开的行动内容就进入详细过程，前端预览和后端 `requires_process` 使用同一判断。
- 健康低于 30 或体力低于 20 时不能新排高强度；已安排的后续重强度行动执行时跌破阈值则自动降为中强度并记录原因，不能留下无 pending 的半局。
- 清洁度过低只提示不强制；羞耻阶段和被囚禁方选择的心情会生成轻量事件标签与数值修正。
- 找出口、藏东西、检查钥匙、私密日记和监控盲区必须带具体动向；私藏物、日记正文和观察进度按监控是否打开做后端视角过滤。
- `view_monitor` 只有在 `monitor_gate` 时能打开封存监控，并生成 `monitor_handle`。
- `monitor_action none` 能不查看直接闭合夜晚；`silent` / `review_later` / `intervene` 只能在查看监控后使用。
- 宠物化状态在两条路线都可建立；具体规矩跨行动保存，服从可进入奖励素材，违令可进入惩戒素材；夜间定点规矩只有打开监控后才能结算遵守或违令。
- `schedule_escape_window` 能设置逃跑诱导。
- `resolve_escape_choice` 能进入抓回链或回到日常。
- `set_recapture_rules` 只能处理 `recapture_rules_choice`，严格保存 1–3 条新规矩。
