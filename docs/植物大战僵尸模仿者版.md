# 植物大战僵尸模仿者版机制草案

> 这份文档先记录核心机制和讨论边界。当前目标不是复刻原版资源，而是做一个纯 Python、无前端面板、给渡作为 AI 玩家玩的“随机模仿者版”塔防模拟器。后续如果规则跑通，再考虑 pygame 或其它可视化层。

## 1. 核心定位

这是一个 headless 塔防模拟器。

参考口径：

- [随机模仿者手机版资料](https://down.7po.com/apps/2945.html)明确写到：当模仿者变成没用的植物时，玩家可以用铲子铲掉。
- [融合版 WIKI 基本玩法](https://wiki.biligame.com/pvzrh/%E8%82%A5%E6%96%99)写到：铲子功能与原版一致，可以铲掉选中的一株植物，无冷却、无需解锁。
- 因此本项目支持铲子，但把它当作“制造选择和后悔空间”的规则动作，不当作降低难度的保底。

核心机制是“随机模仿者”：

- 渡不能直接种确定植物。
- 渡主要能种的是“随机模仿者”。
- 模仿者种下后有变身延迟。
- 变身结果可能是普通植物、特殊植物、条件植物、事件结果，或直接生成僵尸。
- 渡根据当前局面、阳光、冷却和反应延迟选择行动或等待。

第一版不做：

- 不做 HTML / Web 前端。
- 不做 pygame 面板。
- 不做原版素材和名字的公开复刻。
- 不做真人实时操作。
- 不做固定教学关。

第一版只做：

- 纯 Python 规则引擎。
- 文本日志 / 结构化事件输出。
- 随机模仿者开奖。
- 僵尸波次与行动。
- 渡 AI 玩家间歇决策。
- 可复盘的 seed。

## 2. 玩家与游戏关系

渡是 AI 玩家，不是规则引擎。

规则引擎负责真实世界推进：

- tick 递增。
- 僵尸移动。
- 植物攻击。
- 子弹飞行和命中。
- 模仿者倒计时和开奖。
- 阳光产出。
- 冷却恢复。
- 出怪。
- 胜负判定。

渡只在决策点介入：

- 观察当前局面。
- 选择行动。
- 可能选择等待。
- 行动经过反应延迟后才落地。

渡不每 tick 操作，也不直接决定世界结算。

## 3. 时间模型

### 3.1 固定游戏 tick

建议默认：

```text
1 tick = 0.1 秒游戏时间
10 tick = 1 秒游戏时间
```

这个频率对文本模拟足够细：

- 模仿者 3 秒开奖 = 30 ticks。
- 植物攻击间隔可以用 tick 表示。
- 僵尸速度可以用“每 tick 前进多少格”表示。
- 观察窗口和反应延迟也能直接换算。

### 3.2 游戏时间不因外部调用冻结

现实里 AI 思考和工具调用会慢，但不能让游戏完全暂停到没有风险。

正确规则：

- Python 引擎可以在外部等待期间暂停真实执行。
- 但外部耗时会折算成游戏内反应延迟。
- 引擎在执行行动前，会先模拟这段游戏时间。
- 这段时间里僵尸照样走，植物照样打，模仿者照样开奖，波次照样推进。

也就是：

```text
给渡 tick 500 的快照
现实等待渡返回动作
现实耗时映射为游戏内延迟
引擎从 tick 500 推进到 tick 500 + 延迟 ticks
到执行 tick 再校验并落地动作
```

### 3.3 游戏内时间与现实推进

所有时间参数先按游戏时间写，现实执行只负责“算完并返回”。

例如：

```text
模仿者 3 秒变身 = 游戏内 30 ticks
```

这里不是让外部工具真的 sleep 3 秒。规则引擎可以立刻循环 30 次 `step()`，把这 30 ticks 内发生的移动、啃咬、攻击、开奖和出怪都结算完，再返回事件摘要。

玩家行动后的推进也一样。渡提交动作计划后，可能游戏内已经过了 560 ticks，但现实体验应该是：

```text
接收动作计划
引擎一口气推进到下一个停止点
返回这 560 ticks 的压缩结果和下一次观察
```

停止点由规则决定，而不是每过固定 tick 就打断：

- 动作计划执行完。
- wait 条件满足。
- 观察窗口合并完一组强事件。
- 出现紧急事件。
- 游戏胜负已定。
- 达到 `max_fast_forward_ticks` 兜底上限。

也就是说，时间在游戏里连续流动，但在工具循环里是“离散快进”。这样既有风险，又不会让渡和外部调用被每 tick 拖死。

## 4. 现实耗时到游戏延迟

不能把现实 45 秒硬等价为游戏 45 秒，也不能完全不惩罚。

第一版用“软折算”：

```python
game_delay_seconds = clamp(
    base_delay_seconds + max(0, real_elapsed_seconds - grace_seconds) * delay_scale,
    min_delay_seconds,
    max_delay_seconds,
)
```

建议初始参数：

```text
grace_seconds = 15
base_delay_seconds = 2.0
delay_scale = 0.06
min_delay_seconds = 1.5
max_delay_seconds = 5.0
```

效果示例：

| 现实耗时 | 映射游戏延迟 |
| ---: | ---: |
| 10 秒 | 约 1.5-2.0 秒 |
| 30 秒 | 约 2.9 秒 |
| 45 秒 | 约 3.8 秒 |
| 90 秒 | 5.0 秒封顶 |

秒数转 tick 时统一向上取整：

```python
delay_ticks = ceil(game_delay_seconds / tick_seconds)
```

这只是初始值。后续测试时要动态调：

- 如果渡经常因为工具慢直接死，降低上限或提高 grace。
- 如果渡像暂停开挂，提高 base 或上限。
- 普通模式可以更宽容。
- 硬核模式可以更接近真实耗时惩罚。

## 5. 主循环

核心循环分成两层。

内层是游戏循环：

```python
while not game_over:
    events = engine.step()
    scheduler.add(events)

    if scheduler.should_request_decision(state):
        observation = build_observation(state, scheduler.flush())
        return observation
```

外层是 AI 工具循环：

```text
engine.run_until_decision()
  -> 返回观察快照

渡 AI 返回动作计划 JSON

engine.apply_ai_action_plan(action_plan, real_elapsed_seconds)
  -> 折算游戏内延迟
  -> 先模拟延迟 ticks
  -> 按计划逐个安排动作
  -> 每个动作落地前继续模拟并二次校验
  -> 每个动作执行或失败
  -> 一口气推进到下一个停止点
  -> 返回执行结果、新事件摘要和下一次观察
```

工具循环不处理每 tick 结算。引擎一次可以自己跑很多 tick，只在真正需要渡决策时停下来。

## 6. 事件触发与决策调度

### 6.1 事件不等于立刻决策

事件先进入调度器，不直接每个事件都叫渡。

```text
事件发生
进入 pending_events
游戏继续推进观察窗口
窗口结束后合并事件
给渡一次局面快照
```

例如：

```text
tick 300：模仿者开奖成僵尸
tick 305：第 4 行刷出普通僵尸
tick 309：阳光到 100
tick 310：观察窗口结束，合并成一次观察
```

重点：`tick 300-310` 不是冻结时间。开奖出来的僵尸已经在动，新僵尸也正常出现。

### 6.2 观察窗口

观察窗口用于合并短时间内连续事件。

建议初始值：

```text
observe_window_seconds = 0.5 - 1.0
observe_window_ticks = 5 - 10
```

窗口规则：

- 第一个重要事件出现时打开窗口。
- 窗口内继续 step 游戏。
- 后续事件合并进 pending_events。
- 窗口结束后生成一次观察。

### 6.3 决策冷却

渡刚做完一次决策后，普通事件不要马上再次触发。

建议初始值：

```text
decision_cooldown_seconds = 1.5 - 3.0
```

冷却期间：

- 普通事件进入 pending_events。
- 弱事件不立刻打断。
- 紧急事件可以突破冷却。

### 6.4 紧急打断

这些事件可以突破观察窗口或决策冷却：

- 僵尸即将进家。
- 小推车触发。
- 某一路危险值超过高阈值。
- 渡上一个计划动作执行失败。
- 关键防线植物被吃掉。
- 坏开奖在近家位置生成僵尸。

紧急打断也不要无限连。需要有最小间隔或合并规则，避免一秒内反复叫渡。

## 7. 决策触发来源

强触发：

- 模仿者开奖。
- 开奖结果是 `bad` / `chaos` 或 `good` / `rare_good`。
- 新僵尸出现。
- 新一波开始。
- 某一路首次进入危险区。
- 植物被吃掉。
- 小推车触发。
- 动作计划失效。
- 游戏即将失败。

弱触发：

- 阳光增加。
- 阳光达到某个心理阈值。
- 植物冷却完成。
- 僵尸位置普通变化。
- 模仿者快开奖。
- 太久没有操作。

弱触发通常只进入合并队列；强触发达到阈值时打开观察窗口；紧急触发才突破冷却。

## 8. 渡的观察内容

给渡的观察分三层。精确状态和结构化摘要给引擎、测试和外层工具用；真正给渡看的优先是棋盘文本视图。

### 8.1 精确状态

规则引擎内部状态要完整：

```python
state = {
    "tick": 420,
    "sun": 75,
    "grid": [...],
    "zombies": [...],
    "projectiles": [...],
    "cooldowns": {...},
    "pending_imitators": [...],
}
```

精确状态用于结算，不一定完整给渡。

### 8.2 可见摘要

渡决策更适合看摘要：

```json
{
  "schema_version": 1,
  "observation_id": "obs_18",
  "tick": 620,
  "sun": 75,
  "game_status": "running",
  "advance_summary": {"from_tick": 500, "to_tick": 620, "advanced_ticks": 120},
  "reason": ["imitator_revealed_bad", "lane_4_danger"],
  "events": [
    {"type": "imitator_revealed", "lane": 2, "col": 4, "result": "zombie"},
    {"type": "zombie_spawned", "lane": 4, "zombie": "normal"}
  ],
  "lanes": [
    {"lane": 1, "danger": 0.10, "defense": "sunflower"},
    {"lane": 2, "danger": 0.78, "defense": "pending chaos"},
    {"lane": 3, "danger": 0.20, "defense": "wallnut"},
    {"lane": 4, "danger": 0.66, "defense": "none"},
    {"lane": 5, "danger": 0.05, "defense": "safe"}
  ],
  "valid_actions": ["plant_imitator", "plant_card", "wait"]
}
```

观察摘要应少而准：

- 哪几行危险。
- 最近发生了什么。
- 付费功能卡是否够阳光。
- 哪些格子能种。
- 哪些模仿者快开奖。
- 哪些行动合法。

不要让渡读一大堆子弹坐标和内部公式。

### 8.3 玩家版棋盘视图

当前给渡看的观察优先使用 `player_view.format = board_text_v1`，不要直接把完整 JSON 丢给玩家模型。

示例：

```text
Lv1 场地:白天 tick 620
资源: 阳光75 | 推车:1,2,3,4,5

事件:
- 1路4列开奖: 模仿者这次站到了僵尸队。
- 3路3列变成豌豆射手

新单位:
- 路 路障僵尸: 带路障防具，血量高于普通僵尸。

列: 1 2 3 4 5 6 7 8 9
1: 空 空 模 普 空 空 空 空 空
2: 空 向 空 空 路 空 空 空 空
3: 空 空 豌 空 空 空 空 空 空
4: 空 空 空 空 空 桶 空 空 空
5: 空 空 空 空 空 空 空 空 空

卡槽: 模仿者x3(0), 樱桃炸弹x1(150)
动作: 种植, 铲子, 等待, 结束本局
动作格式: 种 模仿者 3-4; 种 豌豆射手 2-3; 种 咖啡豆 2-3; 铲 4-5; 等待; 结束本局
规则: 咖啡豆会唤醒目标格的沉睡蘑菇；铲子只移除植物/未开奖模仿者，同格有僵尸也只铲植物；动作失败会中断后续动作
```

棋盘显示规则：

- `空` 表示该格无植物、无格子化僵尸。
- `模` 表示未开奖模仿者。
- 植物和僵尸同格时用 `豌+普`，如果僵尸正在啃该植物则用 `豌+普咬`。
- 多个同类僵尸同格时用 `普x2`；多个不同僵尸同格时用 `普+路`。
- 未武装土豆雷显示为 `土待`，武装后显示为 `土`。
- 僵尸实际位置仍由引擎用浮点 `x` 结算，棋盘文本只是给玩家看的格子化快照。
- 事件和开奖文案排在棋盘前面，先交代“刚发生了什么”，再给最新棋盘。
- 新植物/新僵尸第一次出现在本局玩家视图时，只补一句中性特性说明；后续只显示简称。

## 9. 渡的动作计划

渡一次决策不限制只能种一个模仿者。他可以返回一个有顺序的动作计划，里面包含多次种植、铲除，也可以在计划末尾选择 wait。理论上可以连续种很多颗，但必须受格子、选卡栏卡槽冷却和操作顺序限制，不能瞬间把全场糊满。

但动作计划不是瞬间完成，也不会让僵尸等他种完。

规则：

- 同一个 `observation_id` 默认只接受一次动作计划；如果失败、中断或 wait 条件满足，引擎会生成新的观察。
- AI 思考耗时先折算成游戏内反应延迟。
- 到达开始执行 tick 后，动作按顺序逐个落地。
- 每个种植动作都有自己的操作耗时，建议初始 `plant_action_ticks = 3`，也就是 0.3 秒游戏时间。
- 每个铲除动作也有自己的操作耗时，当前 `shovel_action_ticks = 1`。铲子不占卡槽、不耗阳光、不冷却，但世界不会暂停。
- 选卡栏槽位数是配置，不把随口举例写死。建议开局 `card_slot_count = 6`，硬上限 `max_card_slot_count = 10`。
- 每个卡槽独立冷却。当前 P2 先支持“多个槽位都带随机模仿者”，所以 `imitator_1`、`imitator_2` 这类槽位可以轮着下。
- 开局选卡阶段使用 `card_selection_text_v1`，候选卡直接带费用，例如：`候选卡: 模仿者(0), 咖啡豆(75), 向日葵(50), 豌豆射手(100), 坚果墙(50), 土豆雷(25), 樱桃炸弹(150)`。
- `imitator_slot_cooldown_ticks = 20` 表示每张模仿者卡自己的冷却时长，不是全局只有一张模仿者冷却。
- 如果计划里的下一个种植动作到点时所有可用卡槽都在冷却，引擎不会冻结等待，而是继续推进游戏，直到最早的槽位冷却结束后再尝试；期间僵尸照常行动。
- 默认允许等待卡槽冷却，但受 `max_action_wait_ticks` 限制；超过上限才算 `cooldown_not_ready` 失败。
- 两个种植动作之间，游戏继续 step：僵尸会走、会啃、会出怪，模仿者会开奖，向日葵等产阳光实体也会按规则结算。
- 每个动作落地前都要重新校验；如果格子被占、卡槽冷却未好、游戏已结束，就该动作失败或触发重决策。
- 动作计划中的失败不会自动回滚前面已经成功的动作。
- 动作计划必须明确顺序。第一个种哪里、第二个种哪里、第三个种哪里，都按数组顺序执行；引擎不自动替渡重排。

玩家文本动作格式：

```text
种 模仿者 3-4
种 咖啡豆 2-3
铲 4-5
等待
结束本局
```

外层可以把这些文本动作薄解析成内部 `ActionPlan`。内部合同仍然用 JSON，方便校验、测试和回放；玩家不需要直接书写完整 JSON。

动作计划格式：

```json
{
  "schema_version": 1,
  "observation_id": "obs_18",
  "action_plan_id": "plan_18",
  "interrupt_policy": "interrupt_on_emergency",
  "actions": [
    {"action": "plant_imitator", "lane": 4, "col": 3},
    {"action": "plant_imitator", "lane": 4, "col": 4, "slot_id": "imitator_2"},
    {"action": "plant_imitator", "lane": 3, "col": 3},
    {"action": "wait", "max_wait_ticks": 40}
  ]
}
```

单动作可以作为兼容简写：

```json
{
  "action": "plant_imitator",
  "lane": 4,
  "col": 3
}
```

兼容简写只给人工调试用；进入引擎前必须归一化成带 `schema_version`、`observation_id` 和 `action_plan_id` 的 `ActionPlan`。

可选动作：

```text
plant_imitator
plant_card
shovel_plant
wait
end_game
```

当前第一版支持：

- `plant_imitator`
- `plant_card`（当前用于咖啡豆这类非随机模仿者卡）
- `shovel_plant`
- `wait`
- `end_game`

特殊道具后续再加。

铲子规则：

- `shovel_plant` 的目标必须是已揭晓植物或待揭晓模仿者。
- 铲掉已揭晓植物会立刻腾出格子。
- 铲掉待揭晓模仿者会取消本次开奖。
- 铲子只提供选择空间，不提供正确答案。玩家可以铲掉废格子，也可能刚铲掉一个“看起来没用”的植物，下一秒就刷出本来会被它克制的僵尸。
- 铲子事件会进入 `recent_rounds`，方便玩家把“这次铲错了/铲对了”的经验写进复盘。

主动结束规则：

- 玩家可以在任何观察点选择 `end_game`，用于放弃本局并重新开始。
- `end_game` 是单独动作，不能和种植、铲除或 wait 混在同一个动作计划里。
- `end_game` 不再折算现实思考延迟，不会为了结束本局再先推进几十 ticks。
- 本局状态记为 `ended_by_player`，不记为 `lost`。
- 结束事件会返回 `restart_level = 1`；下一局从 lv1 重新开始。
- 这是随机版的止损/重开机制，接近无限版体验：玩家可以因为判断必死、开局连续开奖成僵尸、不想继续这局等任意原因结束；长期进步主要靠玩家复盘经验继续保留，而不是强制把每局打到底。

动作执行前必须二次校验：

- 格子是否还空。
- 阳光是否还够。
- 目标行是否还存在。
- 游戏是否已经结束。
- 该动作是否仍在合法动作列表内。

批量种植示例：

```text
tick 500：渡看到第 4 行危险，决定连种 3 个模仿者。
现实思考耗时映射为 38 ticks。
tick 538：开始执行第 1 个动作，种在 4-3。
tick 541：游戏推进 3 ticks 后，第 2 个动作排队等待；如果还有其它模仿者槽位已冷却，就直接换那张卡种。
tick 544：期间第 2 行刷出普通僵尸，但不会打断正在执行的计划。
tick 544：如果所有槽位都在冷却，就继续推进游戏，等最早槽位冷却。
tick 558：最早槽位冷却结束，执行第 2 个动作，种在 4-4。
tick 578：下一张卡冷却结束或已有可用槽位时，执行第 3 个动作前校验，若 3-3 仍可种就落地，否则该动作失败。
```

批量动作的中断策略第一版建议：

```text
单个动作正在落地时，不中断。
两个动作之间，可以被紧急事件中断。
等待卡槽冷却期间，可以被紧急事件中断。
普通事件只进 pending_events，不打断当前计划。
```

紧急中断包括：

- 僵尸即将进家。
- 小推车触发。
- 坏开奖在近家位置生成僵尸。
- 关键防线植物被吃掉。
- 目标行危险值超过高阈值。
- 当前动作失败。

第一版只实现 `interrupt_on_emergency`。其它策略如 `continue_unless_game_over`、`stop_on_any_failure`、`skip_optional_failure` 可以先保留为未来扩展。

如果失效：

```json
{
  "schema_version": 1,
  "action_plan_id": "plan_18",
  "accepted": true,
  "executed_actions": [],
  "failed_actions": [
    {
      "action_index": 0,
      "action": "plant_imitator",
      "reason": "target_cell_no_longer_empty",
      "lane": 3,
      "col": 3,
      "occupants": [
        {"kind": "plant", "plant_id": "sunflower", "entity_id": "p1", "status": "active"}
      ]
    }
  ],
  "advance_summary": {},
  "events": [
    {
      "type": "action_failed",
      "reason": "target_cell_no_longer_empty",
      "lane": 3,
      "col": 3,
      "occupants": [
        {"kind": "plant", "plant_id": "sunflower", "entity_id": "p1", "status": "active"}
      ]
    }
  ],
  "need_next_decision": true,
  "observation": {}
}
```

动作失败后不能静默跳过。默认处理：

- 记录 `action_failed`。
- 暂停剩余动作计划。
- 继续推进 `failure_observe_window_ticks = 3 - 5`。
- 合并失败前后的事件。
- 返回新的观察给渡。

失败原因第一版枚举：

| reason | 含义 |
| --- | --- |
| `target_cell_no_longer_empty` | 目标格子已经被占；事件和 `failed_actions` 会附带 `lane`、`col`、`occupants` |
| `not_enough_sun` | 需要付费的功能卡阳光不足 |
| `cooldown_not_ready` | 冷却未好且等待超过 `max_action_wait_ticks` |
| `target_out_of_bounds` | 目标越界 |
| `action_not_legal` | 动作不在合法动作列表里 |
| `game_already_over` | 游戏已经结束 |
| `plan_interrupted_by_emergency` | 批量计划被紧急事件打断 |

## 10. 随机模仿者机制

### 10.1 基础规则

第一版主种植单位：

```text
随机模仿者
```

建议默认：

```text
cost = 0 sun
reveal_delay_game_seconds = 3.0
reveal_delay_ticks = 30
reveal_real_wait_seconds = 0
pending_hp = 300
blocks_zombie = true
imitator_slot_cooldown_ticks = 20
```

`reveal_delay_game_seconds` 是游戏时间，不是现实等待。引擎可以立刻快进 30 ticks，把这段时间内所有事情结算完，再决定是否到达下一个观察点。

流程：

```text
渡选择格子种下随机模仿者
不扣阳光
该格子进入 pending_imitator
pending_imitator 临时占格并阻挡僵尸
倒计时结束
根据随机池开奖
替换为植物 / 触发事件 / 生成僵尸
```

### 10.2 临时阻挡机制

随机模仿者版里，模仿者不是“等开奖的透明占位”，而是一个真实临时障碍。

示例局面：

```text
某一路只剩一个输出植物，前面没有坚果。
输出植物攻击慢，还没打死僵尸。
玩家在僵尸前方种下模仿者。
模仿者开奖前占格，僵尸会按障碍物规则处理它。
```

所以第一版必须支持：

- `pending_imitator` 占格。
- 僵尸遇到 `pending_imitator` 会停下并啃它。
- `pending_imitator` 在开奖前可以被吃掉、砸掉、炸掉或被铲掉。
- 被吃掉后不会开奖。
- 被铲掉后也不会开奖，且格子立即腾空。
- 开奖时如果还活着，才替换成开奖结果。
- 临时阻挡会进入行状态摘要和 `danger` 计算。

规则口径：

- 模仿者从种下到开奖前视为可被攻击的临时实体。
- 同一行可以连续种多个模仿者，按种植顺序、格子状态和冷却状态分别结算。
- 开奖成植物时替换为对应植物。
- 开奖生成僵尸时，在该格生成对应僵尸。
- 无论后续开奖结果如何，开奖前已经发生的阻挡、啃咬、伤害和移动都保留在时间线中。

对应事件：

```json
{
  "type": "imitator_blocked_zombie",
  "tick": 520,
  "lane": 3,
  "col": 4,
  "zombie_id": "z12",
  "pending_hp": 180
}
```

```json
{
  "type": "imitator_destroyed_before_reveal",
  "tick": 537,
  "lane": 3,
  "col": 4,
  "reason": "eaten_by_zombie",
  "reveal_cancelled": true
}
```

```json
{
  "type": "reveal_spawned_zombie",
  "tick": 552,
  "lane": 3,
  "col": 4,
  "result": "normal_zombie",
  "source": "imitator_reveal"
}
```

AI 观察里记录当前状态：

```json
{
  "lane": 3,
  "danger": 0.74,
  "lane_state": "one_peashooter_plus_pending_imitator",
  "nearest_zombie": "conehead@col4.2",
  "pending_block_seconds_estimate": 2.1
}
```

### 10.3 开奖结果类型

先分四类，不急着定全量内容：

```text
good：普通植物
rare_good：特殊植物
bad：条件植物、低输出植物、场景限制植物
chaos：事件、原地刷僵尸、爆炸、全局扰动
```

示例占位：

```python
GOOD_POOL = ["peashooter", "sunflower", "wallnut"]
RARE_GOOD_POOL = ["squash", "cherry_bomb", "snow_pea"]
BAD_POOL = ["sleeping_mushroom", "fragile_wallnut"]
CHAOS_POOL = ["spawn_zombie_here", "damage_neighbors", "speed_up_zombies"]
```

这些名字只是机制占位，后续再决定具体植物和僵尸清单。

随机模仿者版的目标候选池按“原版植物 + 原版僵尸”理解：墓碑吞噬者、睡莲、花盆这类正常游戏里有场景限制的植物，也应该进随机池。当前 P2 只是分批实现，不代表没实现的植物不属于最终池。

咖啡豆单独处理：它不进入随机模仿者开奖池，而是可以作为玩家选择带进卡槽的功能卡，用来唤醒目标格的沉睡蘑菇。

当前 P2 代码里的开奖权重是手调初稿，不是魔改版精确原表。原版植物大战僵尸本身没有“随机模仿者开奖成植物/僵尸”的官方概率，只能参考植物/僵尸血量、速度、攻击等数值；开奖概率属于魔改版规则。

当前 `du_imitator_pvz/data/reveal_pools.py` 的权重汇总：

| 结果大类 | 权重 | 概率 |
| --- | ---: | ---: |
| plant | 144 | 59.02% |
| spawn_zombie | 97 | 39.75% |
| boss_event | 3 | 1.23% |

按分类看：

| 分类 | 权重 | 概率 |
| --- | ---: | ---: |
| good | 46 | 18.85% |
| rare_good | 54 | 22.13% |
| bad | 44 | 18.03% |
| chaos | 100 | 40.98% |

这个版本不再放空包；实际出植物约 59.0%，出普通僵尸约 39.8%，出僵王事件约 1.2%。`good + rare_good` 约 41.0%；其余包含僵尸、僵王、墓碑吞噬者、睡莲、花盆、睡觉菇、磁力菇这类条件结果。新增反气球/反蹦极/反跳跳/反车辆答案后，直接好抽仍控制在 45% 以下。如果后续要切模式，可以保留概率预设：

| 预设 | 口径 | 适合 |
| --- | --- | --- |
| `patched_random` | 植物约 2/3，非植物约 1/3 | 植物占比较高 |
| `classic_chaos` | 植物约 1/2，僵尸/事件约 1/2 | 植物和非植物接近均分 |
| `current_p2` | 植物约 59.0%，僵尸/僵王约 41.0%，无空包，含条件植物和特殊克制植物 | 当前默认口径 |

### 10.4 难度与概率调整

贴近随机模仿者魔改版时，开奖池不要按关卡隐藏结果。全部结果从一开始就在池子里。关卡变难主要靠正常刷怪数量、波次密度和僵尸潮频率，不靠把随机池换成另一套。

概率调整规则：

- 可以记录连续 `bad` / `chaos` 结果次数。
- 连续 `bad` / `chaos` 结果后，可以调整 good / rare_good 权重。
- 行 `home_eta_ticks` 较低时，可以调整本行生成僵尸类结果的权重。
- 离家较近的位置，早期可以降低高强度僵尸的权重，但不要把它们完全移出池子。
- 如果要提高后期难度，优先增加正常僵尸数量、波次密度和僵尸潮频率；随机池尽量保持同一套口径。

这些调整要记录进 replay 日志，保证同一 seed 可复现。

### 10.5 P1 难度配置草案

随机池按模式配置概率，但同一局内不要按关卡偷偷换池。下面是不同模式的起始权重，不是“通关后解锁结果”。

| 难度 | good | rare_good | bad | chaos | 说明 |
| --- | ---: | ---: | ---: | ---: | --- |
| easy | 58 | 12 | 25 | 5 | 植物占比较高 |
| normal | 48 | 10 | 32 | 10 | 默认权重 |
| hard | 38 | 10 | 35 | 17 | `bad` / `chaos` 占比提高 |
| chaos | 30 | 10 | 35 | 25 | `chaos` 占比最高 |

`chaos` 建议再细分；这些子类可以从一开始就留在池里，只是高强度结果的权重可以很低：

| chaos 子类 | 例子 | 第一版限制 |
| --- | --- | --- |
| `spawn_local_zombie` | 原格刷普通僵尸、小鬼 | 早期降低第 1-2 列高强度僵尸权重 |
| `damage_neighbors` | 伤害周围植物 | 伤害量有上限 |
| `buff_zombies` | 本行僵尸短时加速 | 持续时间和叠加层数有上限 |
| `waste_or_weak` | 睡觉菇 / 弱植物 | 可作为 bad，不一定进 chaos |

高强度结果不做硬解锁，按权重档位处理：

| 结果类型 | 早期权重口径 | 后期权重口径 |
| --- | --- | --- |
| 睡觉菇、睡莲、花盆、墓碑吞噬者、普通僵尸 | 常驻 | 常驻 |
| conehead、buckethead、撑杆、读报 | 常驻中低权重 | 可略增 |
| jack_in_the_box、football、gargantuar | 常驻低权重 | 可略增，但主要靠波次密度变化 |
| zomboss boss_event | 常驻极低但不是 0.x 级别 | 僵尸潮中可略增或作为特殊事件 |

当前 P2 的 `zomboss` 先按“事件型 boss”处理，不是可被植物打死的场上实体：默认持续 `600 ticks`，每 `60 ticks` 执行一次召唤或砸格动作。AI 观察中的 `boss_events` 给出 `remaining_ticks`、`next_action_in_ticks` 和 `actions_taken`，避免玩家只看到僵王出现一下就被快进窗口吞掉。现实耗时映射属于主持/引擎内部结算，不放进玩家观察字段。

概率调整建议：

- 连续 `bad_draw_count >= 2` 时，下一次 good 权重 +10。
- 连续 `bad_draw_count >= 3` 时，下一次 rare_good 权重 +5，chaos 权重 -5。
- 同一行连续两次 `bad` / `chaos` 开奖后，第三次降低近家刷僵尸概率。
- 如果本行 `home_eta_ticks < 40` 且小车已用掉，植物类结果权重轻微上调。
- 如果玩家连续在同一路使用 pending imitator，只按当前 `bad_draw_count`、`home_eta_ticks` 和小车状态调整权重。

AI 现实耗时映射也可按难度分层：

| 难度 | grace_seconds | base_delay_seconds | max_delay_seconds |
| --- | ---: | ---: | ---: |
| easy | 25 | 1.0 | 3.0 |
| normal | 15 | 2.0 | 5.0 |
| hard | 10 | 2.5 | 7.0 |
| chaos | 5 | 3.0 | 9.0 |

先用 normal 作为默认。等跑模拟后，再看渡是不是死得太冤或太像暂停开挂。

### 10.6 选卡栏与金币成长

卡槽不是写死玩法，建议作为局外成长：

```text
初始卡槽：6
最大卡槽：10
解锁方式：通关奖励金币，花金币购买 card_slot_count +1
```

卡槽增加会减少冷却等待，尤其是多个槽都带随机模仿者时。为了保持关卡参数可控，平衡原则是：

- 卡槽 +1 买的是选择空间，不是白送胜率。
- 通关后下一批关卡提高正常僵尸数量、波次密度、混合僵尸比例和僵尸潮频率。
- 随机池不因为关卡推进而硬换，仍保持“什么都可能开出来”的魔改味。
- 墓碑吞噬者、睡莲、花盆等条件植物默认属于随机池候选；抽到时是否有用取决于当前场地/格子/是否实现对应环境机制。
- 咖啡豆不进随机池，但可以占用一个卡槽；当前 P2 已支持 `card_loadout=("coffee_bean",)` 和 `plant_card` 动作，只唤醒目标格的沉睡蘑菇。
- 当前 P2 已支持多个模仿者槽位、咖啡豆卡槽唤醒、第二批植物/僵尸，以及墓碑吞噬者/睡莲/花盆的“条件弱牌”身份；墓碑吞噬者吃墓碑、水池/屋顶格子、普通植物直种和金币商店后续再做。

## 11. 僵尸与波次

僵尸仍按规则从右向左推进。

### 11.1 数值口径

第一版不要追逐原版像素级复刻，先用“原版参考 + 本项目基准”。

已知原版参考：

- 普通豌豆伤害是 20。
- 多数僵尸啃咬按约 100 damage / second 计算。
- 多数普通植物血量约 300，因此普通植物被多数僵尸完整啃掉约 3 秒。
- 坚果血量约 4000，高坚果约 8000。
- 普通僵尸血量约 200，路障约 640，铁桶约 1370。

本项目基准单位：

```text
1 tick = 0.1 秒
1 格 = 一个可种植格子的宽度
walk_speed = 每秒前进多少格
walk_per_tick = walk_speed / 10
bite_dps = 每秒啃咬伤害
attack_interval_ticks = 植物攻击间隔 tick 数
range_type = 攻击范围定义，不用像素
```

| normal | 0.10 | 0.010 | 普通行进速度 |
| fast | 0.16 | 0.016 | 小鬼、小丑、暴怒读报 |
| sprint | 0.22 | 0.022 | 撑杆起跑等短时冲刺 |
| slowed | x0.5 | x0.5 | 被寒冰减速后的倍率 |

这组值只是第一版速度基准。后续如果行进节奏需要调整，优先调这张速度映射。

## 13. 危险值

为了判断何时叫渡，需要计算每行危险值。

危险值不是胜负事实，只是调度和 AI 摘要用。

### 13.1 输入字段

可能输入：

- 最近僵尸离家的距离。
- 最近僵尸预计进家时间 `home_eta_ticks`。
- 僵尸总血量、速度和特殊状态。
- 该行输出植物数量与预计 DPS。
- 该行阻挡植物数量和剩余血量。
- 是否有 pending imitator。
- pending imitator 当前血量、预计拖延时间、剩余开奖时间。
- 小推车是否还在。
- 该行最近是否发生坏开奖。
- 本行是否刚被 wave 或 reveal 连续施压。

输出：

```text
0.0 - 0.3 安全
0.3 - 0.6 注意
0.6 - 0.8 危险
0.8 - 1.0 紧急
```

### 13.2 第一版公式

第一版不用追求完美，但要稳定可调。

```python
danger = clamp01(
    home_pressure * 0.35
    + zombie_mass_pressure * 0.20
    + speed_pressure * 0.10
    + defense_gap_pressure * 0.20
    + pending_risk * 0.10
    + recent_bad_event_pressure * 0.05
    - rescue_buffer * 0.15
)
```

字段口径：

| 字段 | 含义 |
| --- | --- |
| `home_pressure` | 最近僵尸越接近家越高，可由 `home_eta_ticks` 反推 |
| `zombie_mass_pressure` | 本行僵尸总血量和数量 |
| `speed_pressure` | 快速、暴怒、冲刺等状态加压 |
| `defense_gap_pressure` | 本行输出和阻挡不足时增加 |
| `pending_risk` | pending imitator 快开奖且可能变坏时增加 |
| `recent_bad_event_pressure` | 本行刚坏开奖、关键植物刚被吃等 |
| `rescue_buffer` | 小车、坚果、冰冻、强输出等缓冲量 |

`pending_imitator` 的处理要双向：

- 它当前能挡路，所以会降低短期 `home_pressure`。
- 它可能开奖成僵尸或废物，所以会增加 `pending_risk`。
- 它如果快被吃掉，拖延价值下降。
- 它如果快开奖，调度器可以等开奖后合并观察，而不是提前生成一次观察。

建议派生：

```python
pending_stall_ticks = pending_hp / bite_damage_per_tick
pending_reveal_ticks = max(0, reveal_tick - current_tick)
pending_effective_block_ticks = min(pending_stall_ticks, pending_reveal_ticks)
```

如果 `pending_effective_block_ticks` 足够长，短期危险下降；如果 `pending_reveal_ticks <= observe_window_ticks`，开奖本身是强触发候选。

小车规则：

- 小车还在时，`rescue_buffer` 上升，但不把危险值降到 0。
- 小车已用掉后，同一路 `home_pressure` 放大。
- 小车触发是紧急事件，会打断 wait 和批量动作间隙。

危险值用于：

- 是否触发观察窗口。
- 是否突破冷却。
- 给渡摘要。
- 判断 wait 是否合理。

## 14. 等待也是行动

渡可以选择 wait。

wait 不是跳过无风险时间，而是明确表达：

```text
暂时不种，继续观察 / 等卡槽冷却 / 等模仿者开奖
```

第一版 `wait` 不带条件表达式，只允许一个上限：

```json
{
  "action": "wait",
  "max_wait_ticks": 80
}
```

wait 后游戏继续推进，直到：

- 新强事件触发，比如开奖、新僵尸、新波次、植物被吃、小车触发、胜负变化。
- 达到 `max_wait_ticks`。
- 达到引擎默认 `max_fast_forward_ticks`。

`wait` 本质上也是快进：

```text
渡选择 wait
引擎持续 step
中间可能推进 20 ticks，也可能推进 560 ticks
一旦强事件 / 紧急事件 / 上限满足，就返回新 observation
```

旧草案里的 `until` 条件先不做。这样第一版更干净：玩家明确选择“先不种”，引擎自己按强事件或上限把局面回传。

## 16. tick 内结算顺序

为了复盘稳定，同一个 tick 内必须有固定结算顺序。

建议顺序：

| phase | 顺序 | 内容 |
| --- | ---: | --- |
| `scheduled_actions` | 1 | 执行到达当前 tick 的玩家动作，比如种植、铲子；不在这里判断 wait 条件 |
| `reveal` | 2 | 到点的模仿者开奖 |
| `plant_status` | 3 | 植物准备、睡眠、冷却、产阳光等状态更新 |
| `plant_attack` | 4 | 植物选目标、开火、即时植物触发 |
| `projectile` | 5 | 子弹移动、命中、伤害、击杀 |
| `zombie_status` | 6 | 僵尸状态更新，比如减速、冻结、读报暴怒、特殊技能计时 |
| `zombie_move` | 7 | 僵尸移动，或因被阻挡而停下 |
| `zombie_bite` | 8 | 僵尸啃咬植物 / 模仿者 |
| `lawnmower_home` | 9 | 小车触发、清线、进家判定 |
| `wave_spawn` | 10 | 波次推进、右侧正常出怪 |
| `scheduler` | 11 | 收集事件、观察窗口、冷却、wait 上限、紧急打断判断 |
| `win_loss` | 12 | 胜负最终判定 |

同 phase 内也要稳定：

- 玩家动作按动作计划数组顺序。
- 植物按 lane 从小到大、col 从小到大。
- 僵尸按 lane 从小到大，同 lane 内离家更近者优先。
- 子弹按创建 tick 和 projectile_id。
- 事件输出按 tick、phase、severity、event_id。

关键判定：

- 如果植物攻击和僵尸啃咬同 tick 发生，植物攻击先结算；被打死的僵尸本 tick 不再啃咬。
- 如果模仿者开奖和僵尸啃咬同 tick 发生，先开奖，再根据开奖结果承受后续啃咬。
- 如果开奖生成僵尸，第一版建议该僵尸本 tick 不移动、不啃咬，从下一 tick 开始行动，保证同 tick 结算顺序可复现。
- 如果开奖生成即时植物，比如樱桃炸弹或窝瓜，可以在 `plant_attack` phase 触发。
- 如果同 tick 小车触发并清掉最后一只僵尸，不算失败。
- 如果僵尸越过防线且该行小车已用掉，`game_lost` 优先于后续胜利判定。
- 如果最后一波僵尸全部死亡，且没有 pending wave / pending reveal 生成敌人，tick 末尾判定 `game_won`。

## 19. P0：开写代码前必须冻结的合同

P0 是“地基合同”。它不追求好玩，也不追求数值平衡，只确保后面写 Python 引擎时不会边写边改世界观。

### 19.1 P0 完成标准

P0 先冻结这些内容：

- 棋盘坐标和移动方向。
- 游戏时间、tick 和 phase 顺序。
- 关键实体 schema。
- 观察 JSON、动作 JSON、结果 JSON。
- 随机调用和复盘日志格式。
- 胜负规则。
- 最小验收测试。

P0 过了，才能说“可以开写引擎骨架”。实现时以 P0 为唯一硬合同；如果 P1/P2 和 P0 冲突，先改 P1/P2，不允许在代码里绕过 P0。P1 是策略和手感合同，P2 是最小可玩原型。

### 19.2 棋盘与坐标

默认棋盘：

```text
lanes = 5
cols = 9
lane: 1 - 5，自上而下
col: 1 - 9，自左向右
home_side = left
spawn_side = right
home_x = 0.0
spawn_x = 10.0
```

坐标口径：

- 植物和 pending imitator 占用整数格 `(lane, col)`。
- 僵尸使用浮点 `x` 表示横向位置，向左移动，`x` 越小越接近家。
- 僵尸从右侧生成，初始可用 `spawn_x = cols + 1.0`，也就是 10.0。
- 僵尸进入某格时，按 `floor(x)` 或规则函数映射到 `col`，实现时只保留一个函数，不散落在各处。
- 当僵尸 `col - 0.5 <= x <= col + 0.5` 且该格有阻挡实体时，开始啃咬该阻挡实体；已经走过该格左边界的僵尸不能再“回头咬”该格。
- 僵尸到达 `home_x`，也就是 `x <= 0` 时触发进家判定。

占格规则：

- `grid[(lane, col)] = occupant_entity_id | None`。
- `grid` 只记录植物和 pending imitator 这类可占格实体的 `entity_id`，不记录植物类型名。
- 僵尸不进入 `grid`，只使用 `lane/x`。
- 一个格子同一时刻只能有一个植物或一个 pending imitator。
- 开奖前的 pending imitator 视为占格实体。
- 开奖成植物时，继承原格。
- 开奖成僵尸时，pending imitator 消失，并在该格对应位置生成僵尸。
- 如果 pending imitator 被吃掉，格子释放，开奖取消。

### 19.5 JSON 入出口合同

给渡的观察必须是 JSON 可序列化对象：

```python
Observation = {
    "schema_version": 1,
    "observation_id": "obs_1",
    "tick": 0,
    "sun": 150,
    "advance_summary": {},
    "reason": [],
    "events": [],
    "lanes": [],
    "valid_actions": [],
    "action_constraints": {},
    "game_status": "running",
}
```

渡返回动作计划：

```python
ActionPlan = {
    "schema_version": 1,
    "observation_id": "obs_1",
    "action_plan_id": "plan_1",
    "interrupt_policy": "interrupt_on_emergency",
    "actions": [
        {"action": "plant_imitator", "lane": 3, "col": 3}
    ],
}
```

引擎返回动作结果：

```python
ActionResult = {
    "schema_version": 1,
    "action_plan_id": "plan_1",
    "accepted": True,
    "executed_actions": [],
    "failed_actions": [],
    "advance_summary": {},
    "events": [],
    "need_next_decision": True,
    "observation": {},
}
```

硬规则：

- 同一个 `observation_id` 默认只接受一份动作计划。
- `ActionPlan` 必须带 `schema_version` 和 `action_plan_id`。
- `wait` 只能作为唯一动作，或作为 `actions` 的最后一个动作。
- `end_game` 只能作为唯一动作；它表示主动结束本局并从 lv1 重开，不和其它行动混排。
- 未知 action、非法字段和不合法顺序一律返回 `action_failed(action_not_legal)`。
- 所有动作都必须能 JSON 序列化。
- 引擎不信任动作合法性，执行前必须二次校验。
- 所有返回都带 `advance_summary`，哪怕只推进了 0 ticks。

### 19.6 tick 与 phase 单一事实源

P0 直接采用第 16 节 tick 内结算顺序，代码里只能有一份 phase 定义。

也就是：

- phase 顺序写在 `config.py` 或 `events.py`。
- `engine.py` 按同一个列表执行。
- 事件里的 `phase` 必须来自这个列表。
- 测试也按这个列表断言，不另写一套顺序。
- `scheduled_actions` 只执行到期动作。
- `scheduler` 负责把强事件或 wait 上限整理成下一次 observation。

这样可以避免“文档说开奖先于啃咬，代码里啃咬先发生”的隐形 bug。

### 19.7 随机与日志红线

P0 开始就要求所有随机调用走统一入口：

```python
rng.roll(rng_stream, purpose, pool, base_weights, context)
```

不允许在业务逻辑里直接 `random.choice()`。

每次 roll 记录：

- `roll_index`
- `tick`
- `rng_stream`
- `purpose`
- `pool`
- `base_weights`
- `adjusted_weights`
- `context`
- `selected`

每局保存：

- `seed`
- `config_hash`
- `event_log`
- `rng_rolls`
- `final_result`

P0 验收必须证明：同一个 seed + 同一个 `config_hash`，`rng_rolls` 和 `event_log` 完全一致。

### 19.8 胜负规则

失败：

- 僵尸到达 `home_x`，也就是 `x <= 0`。
- 如果该行 `lawnmower_available = true`，先触发小车，不立刻失败。
- 小车触发后会清掉该行全部僵尸，而不是只清触发进家的那一只。
- 小车触发后该行 `lawnmower_available = false`。
- 如果该行小车已消耗，任意僵尸到达 `home_x` 就判定 `game_lost`。

胜利：

- 所有 wave 已结束。
- 场上没有僵尸。
- 没有 pending 的正常出怪或 scheduled spawn。
- 没有 pending reveal。
- 没有其它可能生成敌人的 scheduled event。

P0 可以先不做完整关卡胜利，只做“测试 wave 清空后胜利”。但失败判定必须先有，不然紧迫感测不出来。

## 24. 当前实测口径

### 24.1 时间映射实测

当前默认：

```python
tick_seconds = 0.1
decision_time_scale = 1 / 15
max_decision_delay_ticks = 60
```

也就是说，外部工具循环真实耗时不会按原秒数直接砸进游戏，而是先乘 `1/15`：

| 现实耗时 | 游戏延迟 ticks | 游戏内延迟 |
| ---: | ---: | ---: |
| 1s | 1 | 0.1s |
| 4.5s | 3 | 0.3s |
| 15s | 10 | 1.0s |
| 45s | 30 | 3.0s |
| 90s | 60 | 6.0s，上限截断 |
| 180s | 60 | 6.0s，上限截断 |

这档的参数口径是：现实耗时会折算为游戏内反应延迟，但受 `max_decision_delay_ticks` 截断。

### 24.2 平衡模拟入口

仓库内模拟入口：

```bash
.venv/bin/python scripts/sim_imitator_pvz_balance.py \
  --seeds 30 \
  --levels 1,4 \
  --delay-profiles none,tool_45,mixed \
  --max-decisions 120 \
  --max-ticks 4000
```

当前脚本玩家是固定规则 baseline：有可用模仿者卡槽且冷却好，就选择 `danger` 最高的行补模仿者；否则 wait。它只用于粗测数值。

旧 205 权重池口径下，20 seed 重跑记录如下；这些结果使用过旧的 45s -> 45 ticks 映射，只作为历史参考，不代表当前默认时间映射：

| 档位 | 无延迟 | 固定 45s 现实延迟 |
| --- | --- | --- |
| level 1 | 0 胜 / 20 负 | 1 胜 / 19 负 |
| level 4 | 1 胜 / 19 负 | 0 胜 / 20 负 |

当前结论：

- 当前默认 `decision_time_scale = 1 / 15`、`max_decision_delay_ticks = 60`，45s 现实耗时映射为 30 ticks。
- 默认池去掉空包且扩到 205 权重后，level 1 在该 baseline 下胜率为 1/20（45s 映射）。
- level 4 在该 baseline 下胜率为 0/20（45s 映射），后续需要评估正常波次预算和密度。
- 随机池不按关卡隐藏/解锁 `bad` / `chaos` 结果；关卡难度后续主要调正常波次数量、密度和僵尸潮频率。

### 24.3 真实小工玩家盲测

2026-07-02 跑了一次同一模型玩家连续盲测：

- 玩家：同一个小工玩家 `Raman` 连续读 observation 决策；主线程只负责裁判推进，不脚本代打、不中途换玩家。
- 规则：level 1；`real_elapsed_seconds_per_decision = 45`，历史测试当时按 45 ticks 计入，并且每个 action_plan 都先扣这段时间，`wait` 也一样；`imitator_cost = 0`；`auto_collect_sun = false`；6 个卡槽全是 `imitator`。
- seed：`real-player-lv1-1783009070-297399`，开局对玩家隐藏，结束后公开。
- 结果：第 14 次决策后在 tick 737 失败，失败路为第 3 路。
- 行动统计：种下 42 个模仿者，开奖 36 次，开奖分类为 `bad = 6`、`chaos = 16`、`good = 6`、`rare_good = 8`；3 个模仿者被啃掉未开奖。
- 小车：第 3 路和第 5 路小车各触发一次并清整路，最终第 3 路小车已消耗。
- 关键结局：第 3 路在 tick 531 已用掉小车；tick 635 模仿者开出撑杆僵尸，tick 729 又开出路障僵尸，tick 732 开出睡莲；撑杆僵尸在 tick 736 到家失败。

玩家行为观察：

- 小工不保守：第一手就铺 6 个模仿者，后续多次在高压路一口气补 6 个。
- “模仿者当临时墙拖时间”的玩法自然出现了，尤其第 3 路、第 5 路多次靠 pending imitator 和小车兜底。
- `wait` 只能等开奖的空窗有紧张感；历史测试里的“先扣 45 ticks 再执行动作”在残局救火时非常狠，当前默认已收为 45s -> 30 ticks，但仍保留风险。

- 真实模型玩家和脚本 baseline 行为差异很大；模型玩家会根据局面补线、押高压路和使用模仿者挡路。

## 25. 玩家决策记录与复盘

该机制分成两部分：

- `recent_rounds`：引擎自动记录玩家每轮决策和结果，每轮 action_result 结束后实时写入。
- `notes`：玩家自己写的思考、复盘、备注，可以任意轮次新增、覆盖、禁用或编辑。

### 25.1 自动轮次记录

玩家每次给出 action_plan 后，引擎完成本轮结算，并立刻把这轮压缩成一条 `round_record`：

```json
{
  "round_id": "round_7",
  "observation_id": "obs_7",
  "action_plan_id": "plan_7",
  "from_tick": 300,
  "to_tick": 356,
  "advanced_ticks": 56,
  "stop_reason": "wait_strong_event",
  "actions": [
    {"action": "plant_imitator", "lane": 4, "col": 3, "slot_id": "imitator_1"},
    {"action": "wait", "max_wait_ticks": 80}
  ],
  "executed_actions": [
    {"action": "plant_imitator", "lane": 4, "col": 3, "slot_id": "imitator_1"}
  ],
  "failed_actions": [],
  "result_events": [
    {"tick": 345, "type": "imitator_revealed", "lane": 4, "col": 3, "result": "chaos_normal_zombie"},
    {"tick": 345, "type": "reveal_spawned_zombie", "lane": 4, "zombie_type": "normal"}
  ]
}
```

规则：

- 自动记录只写事实，不写策略建议。
- 每轮都记录，不只在失败或胜利后记录。
- action_result 返回的新 observation 里已经能看到最新 `recent_rounds`。
- `stop_reason` 会区分 `action_plan_completed`、`action_failed`、`wait_strong_event`、`wait_max_wait_ticks` 等结果。
- `result_events` 会压缩保留开奖、刷怪、攻击、击杀、产阳光、受伤、小车和胜负等关键事实。
- observation 只注入最近几轮，避免上下文被流水账挤爆；完整历史可以由外层存储保留。

### 25.2 玩家手写复盘

玩家可以在任意轮次写或改自己的经验：

```json
{
  "memory_id": "loss_l1_001",
  "scope": {"level": 1, "mode": "random_imitator"},
  "player_note": "上一局第 4 路连续开出僵尸；小车触发过一次。",
  "source_run_id": "run_001",
  "source_round_id": "round_12",
  "updated_tick": 825,
  "enabled": true
}
```

`player_note` 只能来自玩家自己写下、确认或改写的内容。系统可以提供客观回放摘要，但不替玩家下策略结论。

下一轮或下一局 observation 里按两部分注入：

```json
{
  "player_experience": {
    "recent_rounds": [
      {
        "round_id": "round_7",
        "action_plan_id": "plan_7",
        "from_tick": 300,
        "to_tick": 356,
        "actions": [{"action": "plant_imitator", "lane": 4, "col": 3}],
        "result_events": [{"tick": 345, "type": "reveal_spawned_zombie", "lane": 4, "zombie_type": "normal"}]
      }
    ],
    "notes": [
      {
        "memory_id": "loss_l1_001",
        "note": "上一局第 4 路连续开出僵尸；小车触发过一次。",
        "source_run_id": "run_001",
        "source_round_id": "round_12",
        "updated_tick": 825
      }
    ]
  }
}
```

### 25.3 整局客观摘要

每局结束后，引擎还可以生成一份客观摘要，方便玩家自己决定要不要写复盘：

```json
{
  "run_id": "run_001",
  "level": 1,
  "result": "lost",
  "final_tick": 825,
  "loss_lane": 4,
  "lawnmowers_used": 2,
  "key_events": [
    "tick 30: 第 4 路模仿者开出 ladder",
    "tick 230: 第 4 路补防开出 normal",
    "tick 796: 第 4 路小车触发"
  ]
}
```

规则边界：

- 自动 `recent_rounds` 和玩家 `notes` 分开，不混成一类。
- 复盘可以失败后写，也可以中途写；可编辑，不锁死。
- 注入内容只放在 observation 的 `player_experience`，不混进规则事实或僵尸文案。
- `notes` 条数要限制；第一版每个模式最多注入 3-5 条。
- `notes` 可以按 `level` / `mode` / `zombie_seen` 做 scope，避免无关局乱注入。

自动摘要只提供事实；玩家 `notes` 由玩家自己编辑。
