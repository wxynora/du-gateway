# 经网关回复截断 — 自检清单

现象：直连中转站正常，经网关就截断（思维链或正文被砍）。

## 1. 网关侧（已做的与可看的）

- **转发前字符数**：日志 `转发前 messages 总字符数约 N`。过大时上游可能因 input 限长截断；已支持 `MAX_REQUEST_CHARS` 裁对话。
- **max_tokens**：未带或过小会强制为 `MAX_COMPLETION_TOKENS`（默认 8192）。日志有 `转发已设 max_tokens=8192（原=xxx）` 即已生效；若仍截断，多半是中转站不认或另有上限。
- **流式读超时**：`requests.post(..., timeout=STREAM_TIMEOUT_SECONDS)` 的 timeout 同时作 connect/read；流式时若该秒数内未收到上游任何数据会 **ReadTimeout 断流**，回复会中途截断。默认 300；若 `.env` 里设得过短（如 30/60）务必改大或删掉用默认。
- **实际收到长度与时长**：流结束打 `本轮流式回复收集长度约 N 字符，共转发 M 个 data 块，流式持续约 T 秒`（**DEBUG 级**，默认不打出；排查时设日志级别为 DEBUG）。若 N 很小（如几百），说明**上游/中转站只给了这么多**。若**流式持续 T 秒**多次都差不多（如总在 12–18s）而字数越来越短，可能是上游按**时长**限流或断流。
- **首包**：`流式收到首包` 表示上游已开始推流。若「转发前」到「收到首包」间隔很长，可能是上游思考久或网络慢；客户端/代理若对「首包前等待」有超时，可能先断连。

## 2. 网关不会做的事

- 不按长度截断回复、不删 SSE 行；流式是「收一行发一行」。
- 不修改上游返回的 content，只做透传。

## 3. 可能原因与对应排查

| 可能原因 | 怎么排查 |
|----------|----------|
| 中转站对「经网关」请求限输出（如按 IP/Key 限 max_tokens） | 看中转站后台：单次输出 token 上限、是否区分直连/API。 |
| 中转站不认 `max_tokens`（或认别的参数如 `max_completion_tokens`） | 查中转站文档/客服，确认参数名与取值范围；必要时网关改传他们要求的参数。 |
| 客户端/代理「首包前」超时 | 看 Rikkahub 或前端是否有「等待首包」超时；前面有 Nginx 则看 `proxy_read_timeout`、`proxy_connect_timeout`，适当调大。 |
| 上游主动断流（错误、限流、模型停） | 看网关日志里 `流式上游结束 finish_reason=...`（**DEBUG 级**）：**stop**=正常结束，**length**=被 max_tokens 截断，**null/未提供**=异常中断。若**根本没有**「流式上游结束」，说明流未读完就断了（ReadTimeout 或客户端断开）。排查时把日志级别设为 DEBUG。 |
| 请求体经网关后过大或结构不同触发中转站限流 | 对比直连与经网关的请求：body 大小、system 条数；用 `MAX_REQUEST_CHARS` 或减注入量把总长压到与直连接近再试。 |

## 4. 流式「读/发」解耦（缓解上游因读速断流）

无 tools 时，网关用**生产者线程 + 队列**：单独线程尽快从上游读入队列，主线程只从队列取出发给客户端。这样「读上游」不再被「发客户端」拖慢，上游不易因「对端读太慢」而断流。若仍截断，多半是上游按别的原因限流（IP/时长等）。

「流式收到首包」「流式上游结束 finish_reason=…」「本轮流式回复收集长度…」等已改为 **DEBUG** 级别，默认不输出，避免收流过程中打日志影响上游读速、加重截断。需要排查时把日志级别设为 DEBUG 即可。

## 5. 无法问上游时的缓解

- **缩小请求体**：在 `.env` 里设 `MAX_REQUEST_CHARS=6000` 或 `8000`，让网关裁掉更早的对话再转发，有时能减轻上游限流导致的截断（可先试 8000，仍截断再试 6000）。
- **看「流式持续约 N 秒」**：若每次都在 10–20s 左右就结束，多半是上游按时长断流，只能换节点/换上游或接受现状。

## 6. 建议一次排查顺序

1. 看当次请求的 **本轮流式回复收集长度**：若就几百字，问题在上游/中转站，不是网关砍包。
2. 看是否有 **转发已设 max_tokens=8192**：有则说明网关已带大 max_tokens；若仍短，问中转站实际生效的 output 上限与参数名。
3. 看 **转发前** 与 **流式收到首包** 的时间差：若 >30s，检查前端或 Nginx 的「首包前」超时并调大。
4. 若有 Nginx 反向代理，把 `proxy_read_timeout`、`proxy_send_timeout` 调到 ≥300s 再试。  
5. 无法问上游时：设 `MAX_REQUEST_CHARS` 试缩小请求；看日志里「流式持续约 N 秒」是否每次都差不多（按时长断流）。
