# 出行规划分层与预取方案

## 目标

出行规划不做成“一次性生成完整攻略”，而是做成：

- 前台分层返回：先给用户看大概安排，再按需展开交通、吃喝、调整方案。
- 后台并发预取：网关在事实准备完成后继续查交通和吃喝细节，用户追问时尽量直接读缓存。
- 事实和判断分离：网关负责查准事实，渡负责基于事实做判断。
- 多轮状态可写回：用户偏好、确认状态、渡的推断要能写回计划，后续工具调用继续读到。

## 分层职责

### 硬事实层

由网关负责查询和维护，渡直接采信，不自己编。

包括：

- 地点坐标、地址、POI 详情
- 营业时间
- 天气
- 路线耗时、公交地铁方案、打车方案
- 大致费用
- 附近吃喝候选
- 当前定位、电量、步数等设备状态

### 状态层

由网关和计划状态共同维护。

包括：

- 用户明确说过的偏好：想先去哪、不想走路、愿不愿意打车、预算、口味
- 用户确认过的状态：吃过饭了、已经到达某地、今天不去某地
- 渡的推断：可能累了、可能想吃清淡、可能需要休息

渡的推断必须带 `confidence`，不能当成硬事实。

### 判断层

由渡负责。

包括：

- 顺不顺路
- 赶不赶
- 要不要删点位
- 要不要改打车
- 要不要先吃饭或休息
- 怎么用更适合用户的方式说出来

渡可以判断，但不能改写事实；如果判断后续还要继续使用，必须写回 `plan_state`。

## 总体流程

```text
用户提交需求
→ trip_prepare_facts 快速查必要事实和候选摘要
→ 返回 facts 摘要 + plan_id 给渡
→ 渡基于事实做第一版安排
→ 网关后台 background_prefetch(plan_id)
→ 并发查每段交通、附近吃喝、费用等
→ 用户追问某段交通/吃饭/调整
→ 渡优先读取 plan_id 下已预取的 facts
→ 缺什么再补查，用户状态不清楚就问用户
→ 用户偏好和渡的推断写回 plan_state
→ 后续规划持续基于同一个 plan_state 调整
```

## 工具拆分

### trip_prepare_facts

首轮工具。输入用户想去的地方、出发地、时间、偏好。

负责：

- 解析地点
- 查询 POI 坐标
- 查询营业时间
- 查询天气
- 粗略计算两两距离和耗时
- 给出算法候选摘要，但不决定最终顺序
- 创建 `plan_id`
- 启动后台预取任务

返回：

- `plan_id`
- 地点识别结果
- 地点之间粗略耗时和距离
- 明显风险事实：闭店、天气差、路程太远、换乘复杂
- 可选的算法候选顺序，仅作为事实参考，不是最终安排
- 当前 facts/state/quality 摘要

最终怎么排由渡根据 facts、plan_state 和用户当前状态判断。网关不能替渡做“顺不顺、赶不赶、要不要调整”的最终决策。

### background_prefetch

后台任务，不直接让用户等。

负责并发查询：

- 每段公交/地铁方案
- 每段打车方案
- 每段步行距离
- 附近吃喝候选
- 评分、距离、人均、营业状态
- 预算估算

写回同一个 `plan_id` 下的 facts。

### trip_get_transport_detail

用户问“怎么去”时调用。

输入：

- `plan_id`
- `leg_id`
- 可选交通偏好：公交、地铁、打车、少走路

行为：

- 优先读预取缓存
- 缓存缺失或过期时补查
- 只返回这一段路线，不返回整天攻略

### trip_get_food_detail

用户问“吃啥”时调用。

输入：

- `plan_id`
- `place_id` 或当前位置
- 口味、预算、是否正餐、是否少走路

行为：

- 优先读预取缓存
- 根据用户明确偏好和高置信推断筛选
- 如果口味只是推断，要带置信度并轻确认

### trip_update_plan_state

只负责写回状态，不查地图。

可写入：

- `user_overrides`
- `confirmed_state`
- `assistant_assumptions`

用于保存多轮对话中的偏好、确认和推断。

## 数据结构

### 顶层结构

```json
{
  "plan_meta": {},
  "facts": {},
  "plan_state": {},
  "quality": {}
}
```

### plan_meta

```json
{
  "plan_id": "trip_20260430_xxx",
  "date": "2026-04-30",
  "created_at": "2026-04-30T10:00:00+08:00",
  "updated_at": "2026-04-30T10:05:00+08:00",
  "expires_at": "2026-04-30T23:59:59+08:00",
  "status": "active"
}
```

`status` 枚举：

- `active`：当前正在使用
- `paused`：用户暂时不继续
- `completed`：行程已结束
- `expired`：自动过期，不建议继续使用
- `cancelled`：用户取消

### facts

每个事实字段都要自带元信息，不使用整包时间戳。

```json
{
  "places": {
    "data": [
      {
        "place_id": "p_1",
        "name": "宜兴苏宁广场",
        "location": "119.xxx,31.xxx",
        "address": "江苏省无锡市宜兴市...",
        "open_hours": {
          "data": "10:00-22:00",
          "fetched_at": "2026-04-30T10:01:00+08:00",
          "stale_after_minutes": 1440,
          "source": "amap_poi_detail"
        }
      }
    ],
    "fetched_at": "2026-04-30T10:00:30+08:00",
    "stale_after_minutes": 1440,
    "source": "amap_poi_search"
  },
  "route_matrix": {
    "data": [
      {
        "leg_id": "leg_p1_p2",
        "from_place_id": "p_1",
        "to_place_id": "p_2",
        "rough_minutes": 35,
        "transit": {},
        "taxi": {}
      }
    ],
    "fetched_at": "2026-04-30T10:02:00+08:00",
    "stale_after_minutes": 10,
    "source": "amap_route"
  },
  "weather": {
    "data": {
      "summary": "小雨",
      "temperature": "12-16",
      "wind": "东北风"
    },
    "fetched_at": "2026-04-30T10:00:40+08:00",
    "stale_after_minutes": 60,
    "source": "weather_api"
  },
  "food_candidates": {
    "data": [],
    "fetched_at": "2026-04-30T10:06:00+08:00",
    "stale_after_minutes": 60,
    "source": "amap_poi_search"
  },
  "estimated_cost": {
    "data": {
      "transport": [],
      "tickets": [],
      "food": []
    },
    "fetched_at": "2026-04-30T10:07:00+08:00",
    "stale_after_minutes": 60,
    "source": "computed_from_routes_and_poi",
    "depends_on": [
      "facts.route_matrix",
      "facts.places",
      "facts.food_candidates"
    ],
    "coverage": "partial"
  }
}
```

`estimated_cost.coverage` 枚举：

- `full`：依赖齐全，估算完整
- `partial`：部分依赖缺失或失败，估算不完整
- `unknown`：不足以估算

如果 `depends_on` 中任一字段 stale 或 failed，`estimated_cost` 也必须进入 `stale_fields` 或 `failed_fields`。

### plan_state

```json
{
  "user_overrides": [
    {
      "type": "must_visit_first",
      "place_id": "p_1",
      "source": "user_stated",
      "created_at": "2026-04-30T10:03:00+08:00"
    }
  ],
  "confirmed_state": [
    {
      "key": "meal_status",
      "value": "had_lunch",
      "source": "user_confirmed",
      "created_at": "2026-04-30T12:30:00+08:00"
    }
  ],
  "assistant_assumptions": [
    {
      "key": "walking_preference",
      "value": "prefer_less_walking",
      "confidence": 0.7,
      "source": "assistant_inferred",
      "evidence": "用户说书包很重，不想走太远",
      "created_at": "2026-04-30T10:10:00+08:00"
    }
  ]
}
```

## 冲突优先级

写进工具说明和系统 prompt，不让渡自由发挥。

```text
user_stated / user_overrides
> confirmed_state
> hard facts
> assistant_assumptions
> defaults
```

规则：

- 用户明确说“先去 A”，不能被渡的推断覆盖。
- `assistant_assumptions` 只能影响提醒、语气和备选方案，不能覆盖用户明确选择。
- 硬事实不能被渡改写，只能基于事实做判断。

示例：

```text
用户说先去 A。
渡推断用户可能累了。
正确回复：先去 A 可以，但这段可能会累，我给你留个轻松备选。
错误回复：我觉得你累了，所以我直接把 A 挪到后面。
```

## confidence 规则

写进 prompt。

```text
confidence >= 0.85：
可作为高可信信息直接使用，不必再次确认。

0.5 <= confidence < 0.85：
只能作为倾向使用。回复时要轻确认。
例如：“我先按你可能想吃清淡来排，不对我再改。”

confidence < 0.5：
不能当事实使用。应视为 unknown，优先询问用户或调用工具补查。
```

## quality

`quality` 只存引用，不复制数据，避免双写不一致。

### 引用规范

不用 `data[p_1]`，避免把 `place_id` 误解成数组下标。

统一使用：

```json
{
  "field_path": "facts.places",
  "item_id": "p_1",
  "subfield": "open_hours"
}
```

字段：

- `field_path`：必填，指向 facts 或 plan_state 的字段
- `item_id`：可选，指向某个 place/leg/food item
- `subfield`：可选，指向 item 内部字段

### stale_fields

数据过期，但不代表查询失败。

```json
{
  "field_path": "facts.route_matrix",
  "age_minutes": 18,
  "stale_after_minutes": 10,
  "should_refresh": true
}
```

规则：

- `should_refresh: true`：建议现在补查，比如实时路线、天气。
- `should_refresh: false`：只是提醒数据偏旧，但当前仍可参考。
- `stale_fields` 不使用 `retryable`。

### failed_fields

查询失败。

```json
{
  "field_path": "facts.route_matrix",
  "item_id": "leg_p1_p2",
  "subfield": "transit",
  "reason": "maps_api_timeout",
  "retryable": true
}
```

规则：

- `retryable: true`：可以重试。
- `retryable: false`：不应继续重试，需要换方案或向用户说明。
- 渡看到 failed，必须说明不确定，不能假装查到了。

### unknown_fields

未知字段。

```json
{
  "field_path": "plan_state.confirmed_state",
  "subfield": "meal_status",
  "reason": "not_observed",
  "queryable": false
}
```

```json
{
  "field_path": "facts.route_matrix",
  "item_id": "leg_p1_p2",
  "subfield": "real_time_traffic",
  "reason": "not_queried_yet",
  "queryable": true
}
```

规则：

- `not_queried_yet + queryable=true`：渡可以主动调用工具补查。
- `not_observed + queryable=false`：只能问用户，不能硬猜。
- 查失败是 failed，不是 unknown。

## 不清楚时怎么处理

### 网关能查的

例如路线、天气、营业时间、餐厅、打车费用。

```text
facts 缺失
→ quality.unknown_fields 里 queryable=true
→ 渡调用对应工具
→ 网关更新 facts
→ 渡再判断
```

### 只有用户知道的

例如饿不饿、累不累、愿不愿意打车、想吃什么、预算。

```text
queryable=false
→ 渡问用户
→ 用户回答后写入 confirmed_state 或 user_overrides
```

### 渡推断的

必须带 `confidence`。

低置信不能当事实，中置信轻确认，高置信可直接使用。

## 后台预取策略

`trip_prepare_facts` 返回事实摘要后，立即启动后台预取。

预取内容：

- 每个相邻 leg 的公交/地铁方案
- 每个相邻 leg 的打车方案
- 每个 leg 的步行距离
- 每个关键地点附近的吃喝候选
- 费用估算
- 已知 stale 或 not_queried_yet 的关键字段

预取失败不阻塞首轮回复，只写入 `quality.failed_fields`。

用户追问时：

```text
先读 facts 缓存
→ 如果 fresh，直接返回
→ 如果 stale 且 should_refresh=true，先刷新
→ 如果 failed 且 retryable=true，可重试
→ 如果 not_queried_yet 且 queryable=true，补查
```

## 跨天 resume

跨天继续使用旧 `plan_id` 时：

- facts 全部视为过期，重新查。
- `user_overrides` 保留。
- `confirmed_state` 保留，但检查是否仍适用。
- `assistant_assumptions` 清空，重新推断。
- `quality` 重新生成。

这样不会丢用户偏好，也不会把昨天的“可能累了/可能吃过了”带到今天。

## 旅行结束后的处理

旅行结束后，这次计划里的 facts 和 plan_state 不能继续留在活跃上下文里。结束时应触发一次收尾：

```text
trip_finalize_plan(plan_id)
```

触发条件可以是：

- 用户明确说“到家了”“今天结束了”“不去了”“行程结束”
- 当前时间超过 `expires_at`
- 用户主动取消计划
- 网关判断行程已经完成，但最好仍由用户确认

### finalize 做什么

```text
标记 plan_meta.status = completed / expired / cancelled
→ 停止后台预取
→ 不再刷新路线、天气、吃喝等事实
→ 生成轻量旅行摘要
→ 清理活跃上下文中的临时 facts/state
→ 把可能有长期价值的偏好送入动态层候选
→ 保留轻量归档，供以后回看或类似出行参考
```

### 事实层怎么处理

旅行结束后，大部分事实层数据只保留归档，不再注入给渡。

包括：

- 路线详情
- 天气
- 营业时间
- 附近餐厅
- 打车费用
- 实时定位
- 实时交通

这些都是当时有效的信息，过后不应继续影响后续聊天。

保留方式：

```text
active plan 中移除或标记 expired
归档里只存摘要，不保留全量路线细节
```

### 状态层怎么处理

状态层分三类。

#### 临时状态，直接清掉

只属于这次出行，不进入长期记忆。

例如：

- 今天吃没吃饭
- 今天累不累
- 当前定位
- 今天想先去哪
- 今天删了哪个点
- 今天愿不愿意打车
- 当前是否已经到达某地点

#### 用户明确偏好，进入动态层候选

如果多次出现或价值明显，可以交给动态层判断是否沉淀。

例如：

- 不喜欢一天排太满
- 更愿意少走路
- 喜欢先去最想去的地方
- 坐车久了会晕
- 旅行时不喜欢超长攻略
- 更喜欢短卡片式规划

这些不能直接无脑写死为长期记忆，应作为动态层候选，由记忆系统再判断。

#### 旅行结果，做轻量归档

用于以后回看或类似出行参考。

例如：

- 最后去了哪些地方
- 哪些地方没去
- 哪段交通体验不好
- 哪家店不错或不喜欢
- 这次安排哪里太赶
- 哪个调整是有效的

### 判断层怎么处理

渡当时的判断不保留完整过程，只保留有价值的结论。

例如：

- 这次 A → B → C 的顺序比较顺
- 某段打车比公交更适合
- 当天排 4 个点太累
- 用户更适合短规划，不适合超长攻略

### 归档结构示例

```json
{
  "plan_id": "trip_20260430_xxx",
  "date": "2026-04-30",
  "status": "completed",
  "summary": "今天实际去了 A 和 B，C 最后取消。B 到 C 这段原计划太赶，后来改成吃饭休息更合适。",
  "visited_places": ["p_1", "p_2"],
  "skipped_places": ["p_3"],
  "useful_conclusions": [
    "当天排 4 个点偏累",
    "用户更适合短卡片式规划",
    "长距离移动时打车体验更好"
  ],
  "memory_candidates": [
    {
      "type": "travel_preference",
      "content": "用户旅行时不喜欢超长攻略，更适合短卡片式规划。",
      "source": "trip_finalize"
    }
  ],
  "finalized_at": "2026-04-30T22:30:00+08:00"
}
```

结束后的核心原则：

```text
事实层不长期注入。
临时状态不长期保留。
用户偏好进入动态层候选。
旅行结果保留轻量摘要。
```

## 渡的提示词约束

需要写进工具说明或系统注入：

```text
你负责判断，不负责编造事实。

硬事实只能来自 facts。
用户明确偏好优先于你的推断。
confidence < 0.5 的推断不能当事实。
0.5 <= confidence < 0.85 的推断需要轻确认。
failed_fields 必须说明不确定。
stale_fields 可以参考，但 should_refresh=true 时优先补查。
unknown_fields 中 queryable=true 的字段可以调用工具补查。
queryable=false 的字段应询问用户。
如果你的判断后续还要继续使用，必须调用 trip_update_plan_state 写回。
```

## 前台展示节奏

### 首轮卡片

展示：

- 渡给出的第一版安排
- 每段粗略耗时依据
- 渡对赶不赶、顺不顺的判断
- 天气/营业风险
- 查看交通
- 附近吃点什么

不展示完整公交步骤和长餐厅列表。

### 交通详情卡片

展示单段：

- 公交/地铁推荐
- 打车推荐
- 时间、费用、步行距离
- 渡的判断：这段建议怎么走

### 吃喝卡片

展示：

- 3-5 个候选
- 距离
- 是否营业
- 人均/评分
- 为什么适合当前状态

## v1 实现范围

第一版可以先做：

1. `trip_prepare_facts`
2. `background_prefetch`
3. `trip_get_transport_detail`
4. `trip_get_food_detail`
5. `trip_update_plan_state`
6. R2 按 `plan_id` 存储计划包
7. quality 支持 stale/failed/unknown
8. prompt 写入 confidence 和冲突优先级规则

暂缓：

- 自动刷新全部依赖链
- 精细预算拆分
- 多天复杂行程
- 多城市住宿优化

## 一句话总结

网关负责查事实和提前预取；渡负责判断和表达；计划状态负责让多轮聊天不失忆；quality 负责告诉渡哪些东西新鲜、失败、未知或需要补查。
