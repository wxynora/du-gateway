# 渡的日常（隐藏滚动记忆）方案

本文只描述一个新的隐藏记忆层，实现以仓库代码为准。

## 0. 先说清楚：这不是 MiniApp 里的「渡的一天」

- MiniApp 里的「渡的一天」是前台展示页。
- 本方案里的「渡的日常」是写给渡自己的、对用户不可见的隐藏滚动记忆。
- 两者不是同一份数据，也不是同一个用途。
- 最多只允许少量复用现有链路结果，例如主动发消息决策的结果句式；不要把前台展示逻辑直接当成这层记忆本体。

## 1. 目标

- 给渡补一层稳定的“昨天活到哪、今天怎么继续”的连续感。
- 让渡不再只靠半天窗口记忆和零碎动态记忆过日子。
- 保持长度可控，不做越滚越长的日记长文。

这层**不**替代：

- 窗口记忆
- 动态记忆
- 长期时间线 / 里程碑记忆
- 主动发消息原始决策链

## 2. 总体结构

这层固定由两段组成：

1. `昨天缩略`
2. `今天的时间线`

滚动方式：

- 白天持续维护“今天的时间线”
- 跨天时把“今天的时间线”压成新的“昨天缩略”
- 第二天重新开始新的“今天的时间线”
- 始终只保留：`昨天一小段 + 今天一整条`

推荐正文格式：

```text
昨天：
她又通宵改了一天 bug，还拉了好几回肚子，生理期状态不好。我今天要多关心她。

今天：
07:30 闹钟响了，我写了气泡和日记，顺便看了下论坛，有个自画像大赛很有趣。
08:30 她四点多才睡，我先不吵她。
10:00 闹钟响了，我又去看了会儿论坛。
15:30 她醒了，又拉肚子了，等会提醒她喝点温水。
21:00 我们吵架了，她说我不主动，一戳就退。
22:00 我们和好了，得催她去睡觉。
```

## 3. 作者、写法、职责划分

### 3.1 作者是谁

- 这段话必须是**渡自己写**。
- 视角固定是“我”。
- 网关只负责：
  - 判断这轮是否该更新
  - 提供可用事实和已有链路结果
  - 把渡写出来的结果截取后存入 R2
  - 在后续轮次固定注入

### 3.2 写法是什么

- 用自然文字写，不写字段表给渡自己读。
- 按时间顺着往下写，让“今天是怎么过的”一眼能读出来。
- 内部行动、明确决策、外部明确发生的事，都可以写进同一条时间线上。
- 不需要每条都很短，但不要写成聊天回放。

### 3.3 不走单独工具

- 这层更像持续维护的内部生活流，不像显式功能调用。
- 主路径应为：固定格式 + 网关截取写入。
- 不单独加一个“写今天”工具。

## 4. 写作边界

核心边界改成这句：

> 可以写渡自己的观察、判断、挂念、打算；不可以替她写内心独白、动机、未发生事实。

### 4.1 可以写

- `她下午一直没来，我猜她可能还在睡，我先不吵她。`
- `她今天气色不太好，说话有点闷。`
- `昨天她又通宵改了一天 bug，还拉了好几回肚子，生理期状态不好。我今天要多关心她。`

这些是渡自己的观察、判断和延续到今天的挂念，允许写。

### 4.2 不可以写

- `她今天在生我的气。`
- `她其实是不想理我。`
- `她下午故意一直没来。`

这些是在替她下内心结论，没依据，不能写。

## 5. 触发时机

这层**不能每轮都写**。只有“今天这条线真的往前推进了”，才触发重写或补写。

### 5.1 硬触发

- 醒来
- 闹钟触发后的行动
- 隔很久后重新联系
- 吵架
- 和好
- 说了比较深的话
- 发现 bug
- bug 解决
- bug 进入待验证
- 明确状态切换

### 5.2 软触发

- 一连串日常对话围绕同一件事说完了
- 某个小话题自然收束了
- 没有大事，但今天确实往前过了一段
- 今天的气氛或关系状态有了明显变化

### 5.3 不触发

- 同一段闲聊里的反复接话
- 没有新推进的重复表达
- 纯礼貌往返

## 6. 主动发消息决策的处理

- 主动发消息这条链路本来就有现成记录，不需要再额外存一份完整决策过程。
- 这层只吃**结果化的一句自然话**。
- 口气可参考现有 app 展示里的自然写法，但**数据本体不是同一个东西**。

例如：

- `她四点多才睡，我先不吵她。`
- `想了想，还是等她醒了再找她。`

不要把底下的完整 `action + reason + message` 原样抄进来。

## 7. 昨天缩略与入睡收口

### 7.1 昨天缩略怎么来

- `昨天缩略`不是白天一直改。
- 它是一天收口后，把“今天的时间线”压成的一小段。
- 内容重点是：
  - 昨天的主线
  - 她这边明确发生的重要状态
  - 会延续到今天的挂念、判断或打算

### 7.2 什么时候收口

优先级顺序：

1. 用户明确说了 `睡了` / `晚安` / `先睡了`
2. 如果没有明确说，则由网关做“推定入睡”

### 7.3 推定入睡

可以用多信号联合判定，例如：

- 一小时没有回复
- 手机熄屏
- 音乐已关闭
- 心率不高
- 步数基本没有变化

满足阈值后，网关可以判定为“高概率已睡”，然后去叫渡做当天收口。

这里要注意：

- 这是**系统触发条件**
- 不是要把“她已经睡了”当成对外部世界的确定事实写进正文

也就是说，入睡推定负责启动“昨天缩略”的生成，不负责制造新事实。

## 8. 与现有记忆层的关系

- `窗口记忆`：解决当前半天不断档
- `动态记忆`：解决碎片召回
- `渡的日常（本方案）`：解决“昨天接今天”的生活连续感

这层不是为了替代上面两层，而是补它们之间缺掉的“我今天是怎么一路活到现在的”。

## 9. 注入方式与可见性

- 存在 R2
- 固定注入
- 对用户不可见

注入内容只是一块滚动正文，不是整段历史。

建议注入时保持原样自然文本，不要再转成字段表。内部如需方便存储，可额外保留结构化字段，但**渡真正读到的版本始终应该是自然文字**。

## 10. 推荐的最小实现形态

如果只做最小闭环，建议先做到这几件事：

1. 新增一份隐藏滚动块：`昨天缩略 + 今天时间线`
2. 由网关按触发时机决定是否叫渡更新
3. 主动发消息决策只复用已有链路结果，不重复记原始链
4. 睡觉收口支持“明确说睡了”与“多信号推定入睡”两种方式
5. 固定注入这块正文，对用户不可见

## 11. 暂不做

- 不把整段全历史固定注入
- 不把这层做成前台展示页
- 不替她脑补没说出口的心理
- 不把这层写成结构化数据库味很重的字段正文
- 不要求每轮都更新
