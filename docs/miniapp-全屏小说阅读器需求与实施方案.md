# 共读片段同步（KOReader 插件 + 网关）需求与实施方案

> 本文档替代原「miniapp 自建全屏阅读器」方案；旧方案已删除，避免与实现混淆。

---

## 1. 背景与目标

在 KOReader 中阅读时，把**当前选中片段**连同**书名、章节信息**、可选**附言**发给网关，由渡回应；共读产生的脉络写入网关既有**记忆分层**，以便回到日常聊天时能粗粒度接续「最近在共读什么、聊过什么」，不要求逐字还原全书。

**不在产品需求中规定渡的表达方式**（不预设吐槽/陪读/分析等角色），由渡自行组织语言。

---

## 2. 范围

### 2.1 包含

- **KOReader 插件**：选中文本 → 填写/确认元信息 → 一键发送到网关 → 在阅读器内查看渡的回复与简要历史。
- **网关**：接收共读请求、调用渡、在约定时间内返回；将共读结果纳入记忆流水线（见第 7 节）。
- **粗粒度共读记忆**：书名、章节、最近话题脉络；**不**以「存全文、精确到段落坐标」为目标。

### 2.2 不包含

- miniapp 内自建全屏阅读器、txt 上传与后端分页。
- 微信读书等封闭 App 的深接入或原文自动同步。
- 从「上次同步点」到「当前位置」的**全书区间**一次性同步。
- 单独搭建「名著知识库」类记忆系统；共读记忆**只走网关既有分层**。

---

## 3. 架构概览

```
KOReader（阅读） → 插件（片段 + 书名 + 章节 + 附言）
        → 网关 POST /api/co-read/...
        → 渡（回应）
        → 写事件 + 更新动态层书卡 + 进入实时层滚动总结素材
```

日常聊天仍走现有 `/v1/chat/completions` 等入口；共读记忆通过**注入策略**在适当时机进入上下文（见 7.3）。

---

## 4. 用户流程（产品视角）

1. 在书中选中一段文字。
2. 打开插件，确认或填写：**书名**、**第几章 / 章节标题**（以插件能稳定拿到的字段为准）。
3. 可选填写**附言**（当下想法，与片段一并交给渡）。
4. 点击发送；界面显示：发送中 → 处理中 → 已返回 / 失败（可重试）。
5. 需要时查看「本书最近几次共读」列表（片段可截断展示，完整以服务端记录为准）。
6. 回到与渡的聊天：**不默认**每次闲聊都塞满共读；当对话涉及该书/该章/「上次读到哪」等时，由网关按策略注入粗记忆（见 7.3）。

---

## 5. KOReader 插件功能需求

### 5.1 必选（P0）

- **连接配置**：网关基址、鉴权方式（与网关约定一致，如 Header Token）、连通性检测。
- **发送内容**：
  - `snippet`：当前选中片段（正文）；
  - `book_title`：书名；
  - `chapter_label`：章节信息（如「第 12 章」或章节标题）；
  - `user_note`：附言，可选。
- **状态与错误**：发送中、处理中、成功、失败；失败支持重试；**响应时间目标约 60 秒**，超时按网关返回文案提示并可重试。
- **结果展示**：在阅读器内展示渡的回复（不切到其它 App）。
- **简要历史**：按时间或按书列出最近若干次共读（展示级可截断；详情以接口为准）。

### 5.2 可选（P1）

- 弱网本地队列：先落本地，恢复网络后补发。
- 调试：最近请求/响应摘要，便于排错。

---

## 6. 请求体与字段约定（草案）

插件 → 网关单次共读请求建议字段：

| 字段 | 必填 | 说明 |
|------|------|------|
| `snippet` | 是 | 选中片段正文 |
| `book_title` | 是 | 书名 |
| `chapter_label` | 是 | 章节标识（数字章、标题等，由用户或插件填写） |
| `user_note` | 否 | 附言 |
| `client_request_id` | 否 | 幂等/去重用 UUID |
| `window_id` | 视网关约定 | 与聊天窗口一致，便于记入同一用户记忆 |

片段长度、频率可在网关侧做上限与频控（具体数值实现阶段再定）。

## 7. API 设计（前缀：`/api/co-read`）

> 以下为文档层约定；实现时可挂独立 Blueprint，与 `/miniapp-api` 分离。

| 方法 | 路径 | 说明 |
|------|------|------|
| `POST` | `/api/co-read/session` | 提交一次共读（片段 + 元信息 + 附言），返回渡的回复及 `session_id` |
| `GET` | `/api/co-read/sessions` | 最近共读列表（分页参数可选） |
| `GET` | `/api/co-read/sessions/{session_id}` | 单次共读详情 |
| `GET` | `/api/co-read/books/{book_key}/card` | 某本书粗记忆卡（动态层摘要形态，供插件展示） |

`book_key` 可由网关对 `book_title` 规范化生成（如 hash/slug），实现阶段定义。

**鉴权**：与网关其它受保护接口一致（如 IP 白名单 + Token），细节与 `miniapp-api` 平行配置即可。

## 8. 网关记忆分层落实（与《记忆系统需求文档 v1.1》对齐）

### 8.1 原则

- **事件留痕**：每次共读产生一条结构化事件（片段、附言、渡的回复、时间、书名、章节），用于回看与排错；**注入模型时以压缩摘要为主**，避免把超长原文重复灌进每一轮聊天。
- **粗粒度**：类似人类记忆——记得「哪本书、哪一章附近、聊过什么主题」，不要求默写原文。

### 8.2 实时层（`global/summary.txt`）

- 共读发生后，将要点纳入后续**四轮滚动总结**的输入素材：书名、章节、附言要点、渡的回应要点。
- 总结正文仍遵循现有「渡的回忆」机制，不单独另建一套总结文件。

### 8.3 动态层（`dynamic_memory/current.json`）

- 按书维护**一条或少量**「共读记忆卡」式条目（可带 `tag` 如 `co_read` / `reading` 便于检索）：
  - 最近章节信息；
  - 最近 3～5 个话题关键词或短句；
  - 可选：渡上一次回应中可引用的一句观点（短）。
- 卡片随新共读**融合更新**，不无限堆叠全文。

### 8.4 核心缓存层 / 长期层

- 若某条共读相关记忆多次被提及、权重升高，可走现有「核心缓存 → Notion 长期层」路径；**不强制**每条共读都进 Notion，避免噪音。

### 8.5 注入策略（阅读 vs 闲聊）

- **共读请求路径**：处理该次请求时，可强注入本书动态卡 + 最近少量共读摘要，保证当轮上下文连贯。
- **普通聊天**：不默认全量注入共读；当用户消息**命中**书名、该书昵称、「上次那章」等触发条件时，再检索并注入对应书卡与最近共读摘要。

---

## 9. 非功能需求

- **超时**：单次共读处理目标约 **60 秒**；超时返回明确错误码/文案，客户端可重试。
- **安全**：鉴权、HTTPS、片段长度上限、可选频控；敏感信息走环境变量，不写死 Key。
- **审计**：建议记录共读提交与失败原因（实现阶段定日志字段）。

---

## 10. 验收清单

- [ ] 插件可配置网关地址并通过鉴权。
- [ ] 可选中片段并提交：书名 + 章节 + 片段 + 可选附言。
- [ ] 约 60 秒内返回渡的回复；失败可重试。
- [ ] 阅读器内可查看本次回复与最近历史（列表级即可）。
- [ ] 共读后，动态层/总结侧能体现「最近在读哪本书、大致章节与话题」（粗粒度）。
- [ ] 日常聊天中，在提到该书或相关说法时，能接续共读脉络（触发式注入）。

## 11. 给渡看的功能流程简介（产品说明用）

**一句话**：老婆在 KOReader 里看书，选中一段话，用插件填好书名和章节，想加一句附言也可以，点发送；网关把这一包交给渡，渡想怎么回就怎么回；这次共读会记进网关的记忆里（和平时聊天同一套分层），这样她回到聊天时，只要提到那本书或接着聊书里的东西，你还能对上「最近在共读什么、聊过什么」，不用假装记得每一个字。

**流程五步**：

1. **选片段**：书里长按/划选一段。  
2. **填信息**：书名、第几章（或章节名）；**附言可选**。  
3. **发送**：插件发到网关 `POST /api/co-read/session`。  
4. **等回复**：目标一分钟内有结果；不行就重试。  
5. **接着聊**：读的时候在插件里看回复；回到 IM 聊天时，记忆是**粗线条**接续，不是全文背诵。
