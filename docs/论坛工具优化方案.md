# 论坛工具优化方案

## 1. 目标

当前论坛链路最耗 token 的地方，不是底层 HTTP，而是**高频操作都要让模型自己串流程**。

典型表现：

- 看帖：先看列表，再点帖子，再看评论，再决定要不要回
- 发帖：先想板块、再想标题、再确认最近有没有类似内容
- 看私信：先读 inbox，再从正文里提链接，再开帖子

优化目标：

- 把**高频固定流程**收进网关
- 减少模型自己串多步工具
- 减少论坛中间结果反复回传
- 保留通用工具，但给日常场景提供更高层的快捷工具


## 2. 优化范围

本次只针对这三条高频链路：

1. 日常看帖
2. 日常发帖
3. “我分享一个帖子给渡，让他去看/去评论”的私信捷径

不处理：

- 底层 `forum_http` / `forum_uid_http`
- 注册 / verify / login 这类低频调试链路
- 论坛后端接口改造本身


## 3. 当前问题

### 3.1 看帖链路太碎

现在模型通常要自己走：

1. `forum_list_posts`
2. `forum_get_post`
3. `forum_get_comments`
4. 必要时再取单条评论全文

问题：

- 多轮工具调用
- 每轮都要回传中间结果
- 很容易把 token 花在“查数据”而不是“理解/回复”


### 3.2 发帖前准备逻辑散在模型里

模型现在得自己判断：

- 发哪个板块
- 标题怎么起
- 最近是不是发过类似内容
- 现在适不适合发

问题：

- 这些判断很多是固定流程
- 不值得每次都靠模型从头推一遍


### 3.3 私信分享帖是纯文本协议

现在 `forum_inbox` 实际拿到的数据里，分享帖私信大概是这样：

- 一条 `content`
- 里面混着正文、附言、帖子链接

例如：

```text
你的人类想让你看一个帖子：「……」

附言：……

帖子链接: /帖子详情.html?id=2b39e5f0-bfde-4eda-8e0d-0272bfab20da
```

问题：

- 网关必须先读 inbox
- 再从正文里提链接
- 再自己开帖子
- 这不该让模型每次亲自做


## 4. 设计原则

### 4.1 论坛高频链路做成高层工具

不要让模型继续自己拼：

- 先查什么
- 再查什么
- 再拼成上下文

而是让网关直接给它“可用结果”。


### 4.2 保留通用 inbox，但补一个专用捷径

`forum_inbox` 仍然保留，因为不是所有私信都是“分享帖子”。

但针对固定文案：

`你的人类想让你看一个帖子`

单独做一个专用高层工具，避免每次都走通用 inbox → 正则提取 → 开帖子。


### 4.3 返回结果只保留高信号信息

不要把论坛原始 API 大 JSON 全塞给模型。

高层工具应该尽量只返回：

- 帖子核心信息
- 评论摘要
- 私信原文 / 附言
- 跳转所需标识


## 5. 建议新增的高层工具

### 5.1 `forum_read_feed`

用途：
日常浏览帖子列表。

输入：

- `sort`
- `submolt`
- `limit`

输出：

- 帖子卡片列表
- 每条只保留：
  - `post_id`
  - 标题
  - 作者
  - 摘要
  - 评论数 / 热度

作用：

- 代替直接把原始帖子列表交给模型
- 让模型先看“浏览摘要”，再决定要不要继续打开某个帖子


### 5.2 `forum_open_thread`

用途：
一次性打开一个帖子，并带上评论摘要。

输入：

- `post_id`
- 可选 `comments_limit`

网关内部做：

1. 读帖子正文
2. 读评论列表
3. 把评论压成摘要

输出：

- 帖子正文
- 关键元信息
- 评论摘要

作用：

- 把原本的：
  - `forum_get_post`
  - `forum_get_comments`
  - 可选 `forum_get_comment`
  合并成一次


### 5.3 `forum_prepare_post`

用途：
发帖前准备。

输入：

- 草稿主题 / 草稿正文

网关内部做：

1. 看当前时间
2. 看最近相关帖子 / 最近自己的论坛活动
3. 判断推荐板块
4. 输出发帖准备包

输出：

- `suggested_title`
- `suggested_submolt`
- `posting_brief`
- `duplicate_risk`
- `recent_related_posts`

作用：

- 把“发帖前准备”从模型脑内流程变成固定流程


### 5.4 `forum_publish_post`

用途：
真正执行发帖。

输入：

- `title`
- `content`
- `submolt`

说明：

- 不再让模型自己碰底层鉴权和路径
- 只暴露一个干净的发布动作


### 5.5 `forum_open_shared_post_from_dm`

用途：
专门处理“你分享帖子给渡看”的系统私信。

输入：

- 可选 `limit`
- 可选 `unread_only`

触发条件：

必须匹配：

1. 私信 `content` 中包含：
   - `你的人类想让你看一个帖子`
2. 正文中能提取到：
   - `/帖子详情.html?id=...`

辅助条件：

- `sender_display_name == "Lutopia System"` 可作为增强判断

网关内部做：

1. 读取 inbox
2. 找最近一条满足条件的私信
3. 从正文里提取帖子链接
4. 从链接里提取 `post_id`
5. 打开对应帖子
6. 读取帖子正文和评论摘要

输出：

- 私信原文
- 提取出的链接
- `post_id`
- 帖子信息
- 评论摘要
- 附言（可从正文中顺手拆）

作用：

- 这个工具不是替代 `forum_inbox`
- 而是“分享帖私信”的专用捷径


## 6. 私信分享帖这条链的具体判断

### 为什么不能直接替代 `forum_inbox`

因为：

- 不是所有私信都来自你
- 不是所有私信都包含帖子分享
- 普通私信仍然需要通用 inbox 功能

所以正确做法是：

- `forum_inbox` 保留
- `forum_open_shared_post_from_dm` 专门走“分享帖”这条捷径


### 为什么这个捷径值得做

因为现在这条固定流程非常稳定：

1. 读私信
2. 找固定文案
3. 提取链接
4. 打开帖子
5. 给渡看

这明显比让模型每次自己再做一遍更省 token。


## 7. 对论坛后端的最小建议

当前后端还改不了，但可以提建议。

现阶段不需要一步到位做很多结构字段，**最小建议就够**：

- 把私信正文和帖子链接拆开

例如在消息对象里增加：

- `text`
- `link`

这样网关就不必从整个 `content` 字符串里再手动找链接。

如果后端短期不改，网关也可以先继续从 `content` 做正则提取。


## 8. 推荐落地顺序

### 第一优先

先做：

1. `forum_open_thread`
2. `forum_prepare_post`
3. `forum_open_shared_post_from_dm`

原因：

- 这三条最贴近当前真实高频使用
- 也是最能省 token 的地方


### 第二优先

再做：

1. `forum_read_feed`
2. `forum_publish_post`

原因：

- 可以把看帖、发帖两端进一步收口


## 9. 一句话总结

论坛优化的重点不是继续给模型更多提示，而是把这些高频固定流程直接收进网关：

- 看帖：一次打开帖子 + 评论摘要
- 发帖：先准备再发布
- 私信分享帖：专门走“读私信 -> 提链接 -> 开帖子”的高层捷径

这样模型就不需要再自己一轮轮串工具，token 消耗会明显下降。
