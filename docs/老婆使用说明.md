# 网关使用说明

## 当前聊天入口

QQ、Telegram、微信、SumiTalk 和主动唤醒共用同一条聊天、提示词、工具、记忆和归档链路。`X-Window-Id` 只用于区分上下文，不存在聊天白名单/黑名单分流。

## 多窗口

- 每个窗口的对话按 window id 分开存储。
- 最近真实聊天入口由 `services/reply_channel_context.py` 记录。
- 小家、纸条、道具等内部事件不会擅自把回复通道改成 SumiTalk。

## App 上游选择

active upstream 和 model 只有在 App 中明确选择并保存时才更新。探活、拉模型列表和普通聊天请求都不能覆盖保存结果。

## 记忆

- 成功回复后才进入归档和记忆更新。
- 动态记忆、中期记忆、画像、核心缓存和记事本使用当前 R2 / SQLite 链路。
- Notion 运行链已经移除。
- 最近窗口只用于上下文选择和诊断。

## 查看状态

```bash
curl "你的网关地址/admin/status"
curl "你的网关地址/admin/windows"
```

状态页只读检查当前存储和配置，不代表所有外部上游一定可用；上游故障仍要结合日志和真实请求判断。

## 查日志

生产环境优先看 systemd 日志：

```bash
journalctl -u du-gateway.service -n 200 --no-pager
```

涉及 SumiTalk、Telegram 等独立 worker 时，要同时查看对应 worker 日志。

## 删除对话轮次

```text
DELETE /admin/windows/<window_id>/rounds/<round_index>
```

`round_index` 从 1 开始。删除会修改真实存储，调用前先确认窗口和轮次。
