# 白名单 / 黑名单方案（取消观察期）

## 目标

- 取消观察期缓存，新窗口**立即**决定白名单 / 黑名单。
- 新窗口默认进白名单；仅当「新窗口 + 本条 user 消息包含『测试』」时进黑名单。
- 黑名单窗口的 AI 回复在返回给前端时，在**末尾**加上「（黑名单）」提示，让 user 知道当前是黑名单窗口。

---

## 一、规则定义

| 条件 | 行为 |
|------|------|
| 窗口已在黑名单 | 只做纯转发；**响应体里 assistant 内容末尾追加「（黑名单）」**；不写 R2、不注入、不存档。 |
| 窗口已有 R2 历史（老窗口） | 按当前逻辑：白名单内走完整管道，非白名单只转发。白名单仍做 14 天 + 最多 50 窗的修剪。 |
| 窗口无 R2 历史（新窗口）且**本条** user 消息**包含**「测试」 | 加入黑名单；**本请求**只转发，且响应体末尾追加「（黑名单）」；不加入白名单、不写 R2。 |
| 窗口无 R2 历史（新窗口）且本条 user 消息**不包含**「测试」 | 加入白名单；本请求及之后按白名单走完整管道（注入、存档等）。 |

说明：

- 「包含」：user 消息的**文本**中只要出现「测试」即触发（如「测试」「这是个测试」「测试测试」均算）。
- 新窗口只根据**当次请求里 user 说的话**判定一次，不缓存前几轮再判定；不实现观察期。

---

## 二、与「观察期方案」的差异

| 项目 | 观察期方案（当前已实现） | 本方案（取消观察期） |
|------|--------------------------|----------------------|
| 新窗前 N 轮 | 只转发 + 本地缓存，不写 R2 | **无**：不缓存，不观察期 |
| 新窗何时进白名单 | 满 N 轮且缓存的 user 消息里无「测试」后，再进白名单并回放缓存 | **当次请求**：若 user 不含「测试」则**立刻**进白名单，本请求就走完整管道 |
| 新窗何时进黑名单 | 满 N 轮且缓存的 user 消息里含「测试」 | **当次请求**：若 user 含「测试」则**立刻**进黑名单，本请求只转发 |
| 黑名单窗口的响应 | 与普通只转发一致，无额外提示 | **在返回给前端的 assistant 内容末尾追加「（黑名单）」** |
| 需保留 | - | 黑名单/白名单存储、白名单 14 天 + 50 窗修剪、管理端解除黑名单 |

---

## 三、请求处理顺序（建议）

1. **取 `window_id`**
2. **黑名单判断**  
   - 若在黑名单 → 只转发，响应里对 assistant 内容末尾加「（黑名单）」→ 返回。
3. **是否已有 R2 历史**  
   - **有历史**：按现逻辑（`step_whitelist_and_record` → 白名单则完整管道，否则只转发）；存档逻辑不变。  
   - **无历史（新窗口）**：  
     - 取**本条请求中**最后一条 user 消息的**纯文本**，判断是否**包含**「测试」。  
     - **包含** → 加入黑名单；只转发，响应里对 assistant 内容末尾加「（黑名单）」→ 返回。  
     - **不包含** → 加入白名单；本请求**直接走完整管道**（清洗、注入、存档等），再转发并返回。
4. 其他逻辑（失败对话不存档、总结每 4 轮等）不变。

---

## 四、「（黑名单）」加在哪儿、怎么加

- **位置**：网关在把上游 AI 的响应返回给前端之前，在**用户可见的 assistant 正文末尾**追加「（黑名单）」。
- **响应结构**：通常为 OpenAPI 兼容的 `{ "choices": [ { "message": { "role": "assistant", "content": "..." } } ] }`。  
  - 若 `content` 为**字符串**：在字符串末尾拼接 `"\n\n（黑名单）"` 或 `"（黑名单）"`（具体格式可定）。  
  - 若 `content` 为**数组**（多模态）：在最后一个 `type: "text"` 的块的 `text` 末尾追加「（黑名单）」；若无 text 块则追加一个新 text 块，内容为「（黑名单）」。
- **只对黑名单窗口**：仅当该请求被判定为「黑名单窗口」时修改响应，其他情况不改动响应体。

---

## 五、代码 / 配置层面的改动清单（实施时用）

1. **删除观察期相关**  
   - 删除或停用：`storage/observation_store.py`、`OBSERVATION_CACHE_FILE`、`OBSERVATION_ROUNDS`、`data/observation_cache.json` 的读写。  
   - 配置中移除观察期相关项（如 `OBSERVATION_ROUNDS`、`TEST_BLACKLIST_KEYWORD` 可保留给「新窗+测试→黑名单」用）。

2. **chat 流程调整**  
   - 去掉「新窗口 → 观察期缓存 → 满 N 轮判定」分支。  
   - 新窗口分支改为：根据**本条** user 消息是否包含「测试」二选一：进黑名单（只转发 + 响应末尾加「（黑名单）」）或进白名单（直接走完整管道）。

3. **黑名单窗口响应改写**  
   - 在返回 `jsonify(resp_json)` 之前，若本次为黑名单窗口，则对 `resp_json["choices"][0]["message"]["content"]` 按上面第四节规则追加「（黑名单）」。

4. **保留**  
   - 黑名单 / 白名单的存储与接口（`blacklist_store`、`whitelist_store`、`_prune_whitelist`、14 天 + 50 窗）。  
   - 管理端：`GET /admin/windows` 的 `blacklisted`、`GET /admin/blacklist`、`DELETE /admin/blacklist/<window_id>`。  
   - 可选：请求头一键加白名单（仅对已有历史窗口有效，与新窗自动加白不冲突）。

5. **配置 / 文档**  
   - `.env.example` 中移除观察期相关（如 `OBSERVATION_ROUNDS`），保留 `TEST_BLACKLIST_KEYWORD=测试`、`MAX_WHITELIST_SIZE`、`WHITELIST_EXPIRE_DAYS`。  
   - README 或内部文档中说明：新窗口默认白名单；新窗口且 user 含「测试」→ 黑名单；黑名单窗口回复末尾带「（黑名单）」提示。

---

## 六、小结

- **不再做观察期**：新窗口在**当次请求**就根据本条 user 是否包含「测试」决定黑名单或白名单。  
- **新窗默认白名单**：不含「测试」→ 立刻加白并走完整管道。  
- **新窗+含「测试」→ 黑名单**：只转发，且返回给前端的 assistant 内容**末尾加「（黑名单）」**，让 user 明确知道这是黑名单窗口。  
- 白名单 14 天 + 50 窗、黑名单解除接口、管理端列表的 blacklisted 标记均保留。
