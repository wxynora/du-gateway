# 感知模块方案

> **当前仅落实阶段一：电量上报。** 其它类型（位置、网络、步数、亮屏、手环等）按阶段后续再开 Tasker、再改 `sense_context.py`、再补本文档。

---

## 核心思路

Tasker 在手机侧上报 → 网关 `POST /api/sense` → R2 `sense/latest.json` → 注入渡的 **system**（不进发给她看的 user）。

## 阶段一：准备工作

- 安装 **Tasker**。
- 手机：**Tasker** 省电策略设为**无限制**（避免后台杀进程）。

（手环、Notify、位置权限等留到后续阶段再写进文档。）

---

## 阶段一：Tasker — 电量上报

- **触发**：电量变化 / 充电状态改变（按你实际 Profile）
- **方法**：POST
- **URL**：`https://你的域名/api/sense`
- **头部**：`Content-Type: application/json`
- **主体**：

```json
{"type":"battery","level":%BATT,"charging":%CHARG,"timestamp":%TIMES}
```

---

## 网关接收端（已实现，与阶段无关）

| 项目 | 说明 |
|------|------|
| 接口 | `POST /api/sense`（`app.py`） |
| 最新快照 | R2：`sense/latest.json`，按 `type` 分桶合并，每桶带 `updatedAt`（UTC） |
| 按天归档 | R2：`sense/history/YYYY-MM-DD.json`（北京日期） |
| 合并 | `storage/r2_store.py` → `merge_and_save_sense_bucket` |

**说明**：接口**仍可按 `type` 写入任意桶**（方便你本地试其它任务），但 **注入渡的文案当前只处理 `battery`**，其它键在 R2 里可先存着，等对应阶段再写展示逻辑。

**合并规则**：除 `type` 外字段 merge 进对应桶；服务端写 `updatedAt`（UTC）。

---

## 阶段一：字段约定（电量）

| 键 | Tasker | 含义 |
|----|--------|------|
| `type` | 固定 `battery` | 顶层桶名 |
| `level` | `%BATT` | 电量百分比 |
| `charging` | `%CHARG` | 是否充电 |
| `timestamp` | `%TIMES` | 终端时间（原样存，含义以 Tasker 为准） |

**`latest.json` → `battery` 桶示例**：

```json
{
  "battery": {
    "level": 87,
    "charging": false,
    "timestamp": 1711180800,
    "updatedAt": "2026-03-23T06:00:00Z"
  }
}
```

**`sense/history/{date}.json` 单条示例**：

```json
{
  "type": "battery",
  "at": "2026-03-23T14:00:00+08:00",
  "data": {
    "level": 87,
    "charging": false,
    "timestamp": 1711180800,
    "updatedAt": "2026-03-23T06:00:00Z"
  }
}
```

## 阶段一：注入（`sense_context.py`）

当前 **只读 `battery` 桶**，生成示意：

```text
[你的当前状态·10:30]
电量：87%，充电中
【以上为网关汇总的老婆设备侧参考状态…】
```

**注入位置**：走网关 `/v1/chat/completions` 完整管道时 `step_inject_sense_snapshot`；全局、不按窗口。

---

## 隔离保障

- `/api/sense` 失败不拖垮对话；`latest` 读失败则跳过注入。
