# 主动联络决策记忆

## 作用

「概率主动」调度抽中后，网关会问渡一轮：**发消息 / 不打扰 / 写日记 / 其它**，并把渡的 **action + reason** 记入 R2。下一轮再问渡时，把 **最近 5 条** 注入 **system**，便于他连贯自己的习惯（**日历/闹钟到点不走此链路**）。

## 存储

| 键 | 说明 |
|----|------|
| `global/proactive_decision_memory.json` | `{"items":[{at, action, reason, message_preview}, ...]}`，新在前，最多 **5** 条 |

- `at`：北京时间 ISO（与 `proactive_tick` 一致）
- `action`：`send_message` / `no_contact` / `diary` / `other` / `error` / `empty` / …
- `reason`：渡在 JSON 里写的 `reason`（或技术兜底文案）
- `message_preview`：若要发消息，正文前 120 字摘要；否则多为空

实现：`storage/r2_store.py` → `get_proactive_decision_memory_items` / `append_proactive_decision_memory`。

## 渡如何回复

调度器请求里 **user** 会要求 **单个 JSON**（不要用 markdown 代码块）：

```json
{"action":"no_contact","reason":"她在忙","message":""}
```

| action | message |
|--------|---------|
| `send_message` | 必填，即要发给她的 TG 正文 |
| `no_contact` | 通常空 |
| `diary` | 可空或写打算记的内容提要 |
| `other` | 可空或补充说明 |

兼容：

- 仅一行 `NO_CONTACT`（或配置的 token）→ 记为不发消息
- 非 JSON 的整段文本 → 视为 `send_message`，整段当正文（`reason` 会标注为未按 JSON）

解析：`services/telegram_proactive.py` → `_parse_proactive_model_reply`。

## 注入位置

- **仅** `_ask_du_should_contact` 的 **system**：`Telegram 风格` → **决策记忆** →（sense 由网关管道注入，不在这里重复拼）

## 已知风险（易出 Bug 的点）

1. **模型不按 JSON 输出**：会走「整段当正文」回退，reason 会变成说明性兜底，记忆里的 action 多为 `send_message`。
2. **`message` 与 JSON 混写**：解析用正则抽 `{...}`，极端情况下可能截错；可观察日志里的 `du_action` / `du_intent_reason`。
3. **多进程网关**：决策记忆写 R2 用进程内 `_global_write_lock`；多实例部署时仍可能竞态（与 summary 等同级别）。
4. **与闹钟区分**：`schedule_tick` 只调 `process_message`，**不会**写本记忆、也不会注入本块文案。
