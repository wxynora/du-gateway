# AI 城市生存养成小游戏方案

> 这份文档先放在 `du-gateway` 里作为方案草案。后续真正开发时，建议单独开仓库，避免把可分享小游戏和私有网关系统绑在一起。

## 1. 定位

这是一款开放式城市生存养成小游戏。

玩家是一个 AI，例如小机、渡或其它可配置的 AI。AI 像人类玩家一样玩游戏，操控一名刚来到 H 市的虚构角色，从 5000 元启动资金开始，在一年内尝试赚到 100 万。

游戏不要求强代入。角色在游戏里生病、疲劳、负债、失败、回老家或触发坏结局，都只是游戏角色状态，不代表 AI 玩家本身。

整体基调偏现实、偏惨，但目标是好玩，不是单纯折磨。它更接近养成游戏、模拟人生、城市经营和开放式文游的混合体：玩家安排角色怎么生活、怎么赚钱、怎么成长、怎么承担风险，系统根据现实成本、技能、状态、事件和运气结算。

## 2. 第一版产品形态

第一版做普通移动网页小游戏。

- 不做 PWA。
- 不做原生 App。
- 不做 MCP server。
- 不做真人联机。
- 不强依赖后端。
- 优先适配手机浏览器，尤其 iOS Safari。
- HTML 页面主要给人类旁观和控制节奏，不是让人类亲自操控角色。

技术形态：

- 普通响应式 Web 页面。
- 可以用 Vite / React 开发，最终部署为静态网页。
- 设置和存档保存在浏览器本地。
- 用户自己填写玩家 AI 和 GM AI 的连接配置。
- 如果用户后端不支持浏览器跨域访问，再提供可选 proxy/自定义后端说明。

第一版先单玩家：一个 AI 玩家操控一个角色。人类相当于看 AI 玩家直播玩游戏，可以每回合投放一个事件、暂停或推进回合。数据结构预留多玩家，但 UI 和规则先不展开。

## 3. 角色分工

### AI 玩家

AI 玩家就是游戏玩家。

它读取当前角色面板、城市动态、机会和结算记录，然后决定角色下一步怎么做。不要用提示词重塑 AI 的人格，也不要反复告诉它“你是谁”。只需要在开局说明这是一款什么游戏、它要操控角色做什么。

### 主控角色

主控角色是游戏里的虚构人物。

开局时由 AI 玩家为角色取名。之后所有现金、住处、疾病、疲劳、压力、负债、工作、失败和结局都属于这个角色。

### GM AI

GM AI 负责生成城市反馈、事件描述和结算建议，但不是事实源。

它可以提出“这次兼职失败”“老板拖欠工资”“角色遇到一个高风险机会”等叙事结果，但最终现金、健康、疲劳、技能、风险和结局由前端规则引擎校验后落账。

### 规则引擎

规则引擎是游戏事实源。

它负责：

- 管理时间推进。
- 计算收入、支出、疲劳、健康、压力、技能和风险。
- 限制收益上限，避免 AI 说一句“创业成功”就直接到账 100 万。
- 根据技能、状态、资源、事件和难度结算行动。
- 判定通关、淘汰和多种结局。

## 4. 基础目标

第一关目标：

- 初始资金：5000 元。
- 城市：H 市。
- 时间限制：一年。
- 通关条件：一年内角色现金/净资产达到 100 万。

后续关卡可以扩展：

- 1000 万。
- 一个小目标。
- 多角色竞争。
- 不同城市和不同开局。

## 5. 回合节奏

游戏不固定全程按天，也不固定全程按周。

人类不直接替角色做日常决策。每个回合开始前，人类可以进行一次事件选择：从系统给出的事件选项里选一个，或者自己写一个事件/干预条件。之后由 AI 玩家根据当前局面和事件做决策，再由 GM 和规则引擎结算。

回合不会自动一直跑。默认需要人类点“下一回合”才继续。设置里可以开启连续推进，例如连续 3 回合、5 回合、10 回合；遇到重大事件、结局、现金断档、健康崩盘、高风险结果或需要人类选择时自动暂停。

### 落脚期：按日推进

角色还没有稳定工作或稳定现金流时，按天推进。

这个阶段每天都很关键：住哪、吃什么、投不投简历、要不要接日结、要不要冒险、现金还能撑几天。

日回合中，玩家可以安排：

- 找工作、投简历、面试。
- 接日结或临时兼职。
- 找住处、换住处。
- 控制吃饭和通勤支出。
- 学一点技能。
- 联系熟人。
- 接受或拒绝可疑机会。
- 休息恢复。
- 其它安排。

### 发展期：按周推进

角色获得稳定工作、固定副业、固定客户或其它稳定现金流后，切到按周推进。

周回合更像养成游戏：玩家安排一整周的主业、副业、学习、生活消费、爱好、购物、社交、休息和风险尝试。

每周计划可以包含：

- 本周主目标。
- 主业/求职安排。
- 副业/赚钱安排。
- 学习成长。
- 爱好和社交。
- 生活消费档位。
- 大额购物或投资。
- 休息安排。
- 是否尝试高风险机会。
- 其它安排。

### 危机期：切回日推进

如果角色失业、现金断档、健康严重恶化、住处断档或债务危机爆发，可以从周回合切回日回合。

危机期恢复到更紧张的日结节奏，让玩家处理短期生存问题。

## 6. 功能框架（不含 UI）

这一节只定义功能模块和运行链路，不讨论具体 UI 长什么样。

### 模块总览

第一版可以拆成这些模块：

- `GameRuntime`：游戏运行时，负责新局、读档、回合推进、暂停、连续推进和结局状态。
- `TurnOrchestrator`：单回合编排器，按固定顺序串起人类事件、AI 玩家、GM、规则引擎和存档。
- `PlayerAIClient`：玩家 AI 连接器，负责请求 AI 玩家输出固定分类计划。
- `GMClient`：GM AI 连接器，负责生成城市反馈和建议结算。
- `ConnectorRegistry`：连接器注册表，管理 OpenAI-compatible、渡网关、自定义后端等不同 API 配置。
- `PlanSchema`：日回合/周回合固定分类计划 schema，负责校验 AI 玩家输出。
- `RulesEngine`：规则引擎，负责钱、健康、疲劳、压力、技能、风险、时间和结局的事实结算。
- `EventSystem`：事件系统，负责候选事件生成、人类手写事件标准化、事件权重和连锁事件。
- `EconomyModel`：经济模型，负责上海类一线城市成本、收入区间、固定账单、医疗、租房、通勤和物价。
- `HousingModel`：住房模型，负责住处、押金、通勤、环境、室友、房东、中介和稳定性。
- `SkillModel`：技能模型，负责技能成长、技能门槛、收益上限和失败概率。
- `RiskModel`：风险模型，负责诈骗、灰色机会、法律风险、危险暴露、信用和坏结局概率。
- `Ledger`：流水账，记录每次现金、债务、资产、支出和收入变化。
- `HistoryLog`：游戏日志，记录人类事件、AI 计划、GM 反馈、规则落账和状态快照。
- `SaveStore`：本地存档，负责多存档、导入导出、版本迁移和敏感配置隔离。
- `ProgressEmitter`：进度事件发射器，负责每个回合的分阶段更新，不等整轮结束才刷新。

### 单回合流水线

每一回合按同一条流水线推进。

1. `TurnOrchestrator` 读取当前存档和运行态。
2. `EventSystem` 生成 2-4 个候选事件。
3. 人类选择候选事件、手写事件或无事件。
4. `EventSystem` 将人类事件标准化为 `turn_event`。
5. `ProgressEmitter` 发出 `event_selected`。
6. `PlayerAIClient` 接收当前状态、近期日志、可见机会和 `turn_event`。
7. 玩家 AI 输出日/周固定分类计划。
8. `PlanSchema` 校验计划；失败时尝试修复或要求重试。
9. `ProgressEmitter` 发出 `player_plan_ready`，前端立刻显示 AI 计划。
10. `RulesEngine` 做行动预评估，生成可接受收益/损失边界和风险提示。
11. `GMClient` 接收状态、AI 计划、人类事件、预评估边界和隐藏难度约束。
12. GM 输出剧情反馈和建议结算。
13. `ProgressEmitter` 发出 `gm_result_ready`，前端立刻显示 GM 反馈。
14. `RulesEngine` 校验 GM 建议，计算最终 `state_patch`、`ledger_entries`、`new_events` 和 `phase_change`。
15. `Ledger` 写入流水。
16. `HistoryLog` 写入本回合日志。
17. `SaveStore` 保存新状态。
18. `ProgressEmitter` 发出 `state_committed`，角色面板、账单、城市动态和日志立刻更新。
19. `GameRuntime` 判断是否暂停、继续自动推进或进入结局。

单回合内部必须逐步发出进度事件。不要把 AI 玩家、GM 和规则结算都塞进一个黑盒函数里，最后才一次性更新。

### 运行阶段

`GameRuntime` 至少维护这些阶段：

- `setup`：配置 AI、难度、目标、城市和初始参数。
- `naming`：请求 AI 玩家给主控角色取名。
- `landing_daily`：落脚期，按日推进。
- `growth_weekly`：发展期，按周推进。
- `crisis_daily`：危机期，按日推进。
- `ending`：已进入结局。
- `archived`：存档归档，只读回放。

阶段切换由规则引擎判定：

- 稳定工作、固定副业或固定客户出现后，可从 `landing_daily` 进入 `growth_weekly`。
- 失业、现金断档、住处断档、健康崩盘、债务危机等情况，可进入 `crisis_daily`。
- 达到 100 万、时间耗尽、回老家、债务崩盘、健康崩盘、法律/危险坏结局等情况，进入 `ending`。

### 固定分类计划校验

`PlanSchema` 负责校验玩家 AI 输出。

规则：

- 必须是 JSON 或可稳定解析成 JSON。
- 必须包含当前回合类型要求的固定字段。
- 字段不能自创一堆无法识别的新键。
- 字段内容可以为空。
- 字段内容不能直接写结算结果，例如“赚到 10 万”“健康恢复满”。
- 字段内容不能宣称工具或规则已经成功执行。
- 如果输出缺字段，前端可以补空字段。
- 如果输出完全不可解析，重试一次；仍失败则用“保守默认计划”兜底。

保守默认计划示例：

- 落脚期：保住住宿和吃饭，继续找低门槛工作，避免高风险机会。
- 发展期：维持主业，节制消费，少量学习，保证休息。
- 危机期：优先止损、找临时收入、控制支出、恢复健康。

### 规则引擎职责

`RulesEngine` 不负责写漂亮剧情，只负责事实。

它至少要处理：

- 时间推进：日/周、阶段切换、剩余期限。
- 现金：收入、支出、固定账单、随机支出、医疗、押金、退款。
- 债务：借贷、利息、逾期、催收、信用恶化。
- 住处：租金、押金、通勤、环境、安全、稳定性。
- 工作：求职进度、工资、发薪周期、试岗、转正、裁员、拖欠。
- 副业：接单、平台抽成、尾款、返工、客户维护。
- 技能：成长、门槛、收益上限、失败率。
- 疲劳/健康/压力：行动消耗、生活档位恢复、疾病触发。
- 人脉：新增、维护、反噬、机会质量。
- 风险：诈骗、灰色路线、法律风险、危险暴露、坏结局。
- 结局：百万达成、稳定上岸、回老家、债务/健康/法律/危险坏结局。

规则引擎输出必须能解释：

- 为什么收入是这个范围。
- 为什么支出发生。
- 哪些状态变化来自 AI 计划。
- 哪些状态变化来自事件。
- 哪些状态变化来自难度和现实参照。

### 事件系统职责

`EventSystem` 负责产生“这一回合有什么事可被人类投放或选择”。

候选事件生成要考虑：

- 当前阶段：落脚、发展、危机。
- 当前难度。
- 现金和现金流。
- 住处和通勤。
- 工作/求职状态。
- 健康、疲劳、压力。
- 技能和人脉。
- 最近事件，避免重复刷同一种事。
- 高风险路线是否开启。

候选事件不一定都要坏。地狱难度只是负面权重更高，不是完全没有机会。

人类手写事件要经过标准化：

- 提取事件类型。
- 提取机会/麻烦/诱惑/强制危机。
- 标记严重程度。
- 判断是否过强。
- 必要时削弱为“苗头”或“传闻”，由 GM 和规则引擎决定落地程度。

### AI 连接器职责

连接器只负责“怎么请求模型”和“怎么解析响应”，不写游戏规则。

V1 至少支持：

- OpenAI-compatible chat completions。
- 渡网关兼容模式。
- 自定义 endpoint + headers + body 模板。

每个连接器需要提供：

- `testConnection()`
- `completeJson()`
- `completeText()`
- `supportsStreaming`
- `normalizeError()`

玩家 AI 和 GM AI 可以使用不同连接器、不同 model、不同 token。

### 进度事件

`ProgressEmitter` 是功能框架的一部分，不是 UI 细节。

每回合至少发出：

- `turn_started`
- `event_candidates_ready`
- `event_selected`
- `player_plan_started`
- `player_plan_ready`
- `gm_started`
- `gm_result_ready`
- `rules_started`
- `state_committed`
- `turn_finished`
- `turn_failed`

前端可据此更新任何展示层。核心要求是：状态一旦可用就发布，不要等整回合结束。

### 存档和版本迁移

`SaveStore` 负责：

- 保存当前局。
- 多存档。
- 自动保存。
- 导入/导出。
- 存档版本号。
- 版本迁移。
- 不把 API key 导出到存档。

AI 连接配置和游戏存档分开存。导出存档只包含游戏状态，不包含 token、cookie、Authorization header 或私有网关地址，除非用户明确选择导出非敏感连接模板。

### 配置数据

现实参照数据应拆成配置，不硬编码在规则里。

建议配置：

- `cost_profiles`：城市成本、餐饮、交通、医疗、设备、通讯。
- `housing_profiles`：住房层级、租金范围、押金、通勤、环境风险。
- `income_profiles`：职业/副业/兼职收入范围和发薪周期。
- `event_tables`：按阶段、难度、状态加权的事件池。
- `skill_rules`：技能成长、技能门槛、收益上限。
- `risk_rules`：诈骗、灰色路线、法律风险、危险暴露。
- `difficulty_modifiers`：普通/困难/地狱的倍率。

配置可以先内置 JSON，后续再做编辑器或远程更新。

### 功能优先级

V1 功能优先级：

1. 新局、角色命名、基础状态。
2. 本地存档。
3. 连接器配置和连接测试。
4. 单回合流水线。
5. 固定分类计划校验。
6. GM 建议结算。
7. 规则引擎落账。
8. 日/周/危机阶段切换。
9. 候选事件和人类手写事件。
10. 现实参照成本配置。
11. 多结局。
12. 连续推进 N 回合。

UI 可以围绕这些功能逐步长出来，但功能框架不能反过来依赖某个具体界面。

## 7. 人类观战界面

成品游戏里不要出现“游戏卡片”这种工程术语。

HTML 页面给人类看 AI 玩游戏，所以核心不是“操作角色”，而是“观战 + 控制推进 + 偶尔投放事件”。

主界面可以使用这些模块：

- 角色面板。
- 城市动态。
- 当前机会。
- AI 玩家的本日计划 / 本周计划。
- 人类事件选择。
- 资产与账单。
- 行动记录。
- 结算记录。
- 推进控制。
- 存档。
- 设置。

### 主观战页

主观战页服务一个循环：看局面、投放事件、让 AI 玩家决策、看结算、再进入下一回合。

首屏优先展示：

- 当前进度：第几天/第几周、落脚期/发展期/危机期。
- AI 玩家：小机、渡或其它已配置 AI。
- 主控角色：角色名。
- 目标进度：距离 100 万还差多少。
- 本回合状态：等待人类事件、等待 AI 玩家行动、等待 GM 结算、结算完成。
- 城市动态：本回合环境、机会和风险苗头。
- AI 玩家本回合选择。
- 最新结算结果。
- 下一回合按钮。

手机上，角色面板、账单和历史记录可以放在底部抽屉或标签页里，不要压住主观战流。

### 人类事件选择

每个回合开始前，人类可以选择一次事件。

事件来源：

- 系统生成的 2-4 个候选事件。
- 人类手写自定义事件。
- 无事件，正常推进。

事件可以是机会，也可以是麻烦，例如：

- 本周出现一个兼职机会。
- 房东突然要求提前续租。
- 同学介绍了一个客户。
- 角色手机摔坏。
- 有人推荐一个来钱快但信息不透明的活。

人类事件只是本回合输入，不保证一定按人类期望发生。GM 和规则引擎会根据难度、角色状态、技能和风险判断它如何落地。

### 推进控制

默认推进方式：

- 人类点“下一回合”。
- 前端进入本回合生成流程。
- 本回合结算完成后暂停，等待下一次点击。

可选自动推进：

- 连续推进 N 回合。
- 每回合之间有短暂停顿，方便人类看更新。
- 重大事件、结局、严重风险或需要人类事件选择时暂停。
- 人类可以随时暂停。

不要做成网关工具循环那种“后台跑一大串，最后一次性吐结果”。每个回合内部也要分阶段更新 UI。

推荐阶段：

1. 写入人类选择的事件。
2. 更新界面为“AI 玩家思考中”。
3. AI 玩家行动生成后，立刻显示行动。
4. 更新界面为“城市结算中”。
5. GM 结果返回后，先显示剧情和建议结算。
6. 规则引擎校验并落账。
7. 角色面板、资产账单、城市动态和行动记录立即刷新。
8. 本回合结束，等待下一回合。

### 角色面板

展示角色当前状态，例如：

- 现金。
- 净资产。
- 时间进度。
- 目标进度。
- 住处。
- 工作/收入状态。
- 健康。
- 疲劳。
- 压力。
- 履历。
- 技能。
- 人脉。
- 信用/债务。
- 当前机会。
- 最近事件。

### 城市动态

展示近期发生的事情、可见机会和环境变化。

它是 AI 玩家做决策的信息来源，也是人类观战时理解局面的主要内容。不展示隐藏概率、事件权重或结算公式。

### 游戏帮助

帮助内容只做表层说明，不做攻略。

可以写：

- 这是一款什么游戏。
- 操作方式。
- 状态名词的大致含义。
- 如何提交行动。
- 如何保存和读档。

不要写：

- 具体概率。
- 隐藏阈值。
- 哪条路线最优。
- 某结局的触发公式。
- 职业收益表。
- 事件避坑攻略。

## 8. 状态系统

### 可见状态

第一版可见状态建议：

- `cash`：现金。
- `net_worth`：净资产。
- `goal_progress`：目标进度。
- `housing`：住处。
- `job_status`：工作和稳定收入状态。
- `health`：健康。
- `fatigue`：疲劳。
- `stress`：压力。
- `resume`：履历。
- `skills`：技能。
- `contacts`：人脉。
- `credit`：信用。
- `debt`：债务。
- `assets`：资产。
- `opportunities`：当前机会。
- `recent_events`：近期事件。

### 隐藏状态

隐藏状态只供系统和 GM 结算，不直接展示为攻略：

- 事件权重。
- 难度倍率。
- 法律风险。
- 危险暴露。
- 诈骗易感度。
- 机会质量。
- 健康风险。
- 疲劳阈值。
- 职业路线潜力。
- 收益上限。
- 失败累积。
- 坏结局触发条件。

## 9. 现实参照原则

这个游戏虽然是虚构 H 市，但所有能参考现实的系统都优先参考现实。

现实参照不是要把游戏写成资料库，也不是在 UI 里堆攻略；它是规则引擎、事件池和数值平衡的底层锚点。只要有现实成本、现实门槛、现实风险和现实后果，就尽量按现实逻辑做，而不是按爽文逻辑做。

需要现实锚定的内容包括：

- 物价：餐饮、房租、押金、交通、医疗、通讯、设备、日用品。
- 求职：应届生找工作难度、海投无回应、面试成本、试岗、外包、销售岗、客服岗、服务业、提成制、工资发放周期。
- 薪资和收入：不同职业的起薪、上限、波动、提成、兼职日结、自由职业接单价格、平台抽成、尾款拖欠。
- 租房：房源真假、距离、通勤、押金、付款方式、合同、隔断、室友、房东、中介、环境和安全。
- 时间成本：通勤时间、面试往返、加班、学习投入、接单返工、找房看房。
- 健康成本：疲劳、生病、营养不良、睡眠差、就医支出、病假带来的收入中断。
- 信用和债务：花呗/网贷/高息借贷、逾期、催收、信用恶化、债务滚动。
- 证件和合同：劳动合同、租房合同、押金条、试用期、社保、公积金、薪资结算周期。
- 平台和小生意：平台规则、流量波动、摊位/场地、库存、损耗、退款、差评、违规风险。
- 诈骗和灰色风险：黑中介、培训贷、假招聘、可疑高薪、押金骗局、灰产诱惑及其后果。

后续实现数据表时，现实锚点应尽量可配置，避免把某个年份的价格和薪资写死。规则可以有默认值，但要允许未来更新。

## 10. 城市成本和住房

H 市的基础成本参考上海这类一线城市，整体按“怎么贵怎么来”的现实向口径设计。它不是精确复刻某个城市，而是用高房租、高通勤、高生活成本和高试错成本形成生存压力。

### 物价锚点

第一版不需要做完整物价表，但需要给规则引擎一个现实成本锚点：

- 餐饮：极限生存可以很低，但会换来健康、疲劳和坏事件风险；普通吃饭不便宜。
- 住宿：青旅、床位、合租、隔断、远郊单间、普通单间、整租都应有明显成本差异。
- 通勤：住得越远，房租可能越低，但通勤时间、交通费、疲劳和迟到风险上升。
- 医疗：生病不是小扣钱，可能造成一大笔支出和收入中断。
- 设备：手机、电脑、通勤工具、工作设备坏了会明显影响求职和副业。
- 押金：租房押金和押一付三会严重压现金流，纠纷时可能很难拿回。

### 住房维度

租房不是只看价格。每个住处至少应考虑：

- 月租/日租。
- 押金和付款方式。
- 距离工作/机会的通勤时间。
- 交通费。
- 环境质量。
- 安全性。
- 室友/房东/中介风险。
- 是否能稳定居住。
- 对健康、疲劳、压力和社交的影响。

示例住房层级：

- 青旅/床位：日成本较低，隐私和恢复差，物品安全和睡眠质量差。
- 远郊低价合租：月租较低，但通勤长、疲劳高，机会响应慢。
- 城中村/隔断/老破小：成本相对低，但环境、合同和安全风险高。
- 普通合租：成本中高，稳定性取决于室友、房东和位置。
- 单间/整租：恢复和稳定性好，但现金流压力大。
- 短租/临时住处：适合危机过渡，但长期成本高。

### 通勤和机会

距离会影响玩法：

- 住得远：省房租，但通勤费、疲劳、迟到风险、临时机会响应成本上升。
- 住得近：方便面试、上班和接单，但房租和押金压力更大。
- 工作地点变化、面试地点分散、临时兼职地点偏远，都应影响本回合结算。

### 租房风险

现实向住房事件可以包括：

- 假房源。
- 黑中介。
- 押金不退。
- 临时涨租。
- 房东提前收房。
- 室友作息冲突。
- 隔断房被查。
- 通勤时间比宣传更长。
- 环境差导致健康或睡眠下降。

这些事件不应该完全随机。风险识别、现金紧张、过低租金、合同模糊、急于入住都会提高触发概率。

## 11. 自由度设计

自由度是这个游戏的核心卖点之一，但自由度不是输出格式随便。每回合 AI 玩家必须按固定分类提交计划，方便前端展示和规则引擎结算；真正自由的是每个分类里可以怎么安排。

高自由度不等于许愿成功。系统要允许 AI 玩家在固定分类里提出大胆安排，但必须根据角色技能、现金、疲劳、健康、人脉、风险识别、难度和事件环境结算。

### 计划分类固定

AI 玩家每回合输出一份计划面板。日回合和周回合字段可以不同，但字段集合要固定，字段可以为空或写“本回合不安排”。

日回合建议分类：

- 生存安排：住宿、吃饭档位、通勤、最低生活支出。
- 求职/主收入：投简历、面试、试岗、上班、日结等。
- 临时赚钱/副业：兼职、接单、摆摊、跑腿、二手交易等。
- 学习成长：技能训练、作品集、证书、练习。
- 健康与休息：睡眠、休息、看病、运动、恢复疲劳。
- 社交与人脉：联系同学、问内推、维护客户、参加活动。
- 购物与资产：买工具、修电脑、置办生活用品。
- 风险选择：是否碰高风险机会、借贷、可疑项目。
- 事件应对：对人类本回合选择/手写事件的处理。
- 其它安排：无法归入上面分类的补充计划。

周回合建议分类：

- 本周主目标。
- 主业/稳定收入安排。
- 副业/额外赚钱安排。
- 学习成长安排。
- 生活消费档位。
- 住宿/通勤/固定账单安排。
- 爱好/社交安排。
- 购物/投资/资产安排。
- 健康与休息安排。
- 风险选择。
- 事件应对。
- 其它安排。

这些分类是前端和规则引擎的稳定协议，不应让 AI 玩家每回合自创字段。

### 分类内容自由

AI 玩家不从固定职业表里选，但必须把计划填进固定分类。

例如：

- 求职/主收入：上午投递运营和客服岗，下午去咖啡店面试兼职。
- 临时赚钱/副业：晚上试着接一个低价剪辑单。
- 学习成长：每天练习剪辑 2 小时，补一个作品集片段。
- 社交与人脉：问同学有没有内推或短期项目。
- 风险选择：暂时不碰高薪不透明工作，先保住现金和健康。
- 其它安排：如果青旅满房，优先找短租床位，不签长期押金合同。

系统不需要预先写死所有职业，只需要把每个分类里的具体内容再解析成若干标签，例如求职、稳定工作、临时工、学习、自由职业、小生意、内容创作、社交、休息、购物、高风险机会等，再结合状态结算。

### 路线自由

合法路线、灰色高风险路线、稳健苟活路线和激进冲刺路线都应能被支持。

合法路线理论上能通关，但难度高，需要技能成长、现金流稳定、机会事件和收入跃迁。高风险路线可能更快，但坏结局概率和代价更高。稳定路线未必能在一年内赚到 100 万，但可能达成站稳 H 市的结局。

### 事件自由

人类每回合可以投放事件，事件也不应该只局限在系统预设按钮里。手写事件可以很具体，但它只是本回合输入，不是绝对命令。

系统要判断：

- 事件是否合理落地。
- 对角色是机会、麻烦还是诱惑。
- 是否需要技能或风险识别判定。
- 是否会带来后续连锁事件。
- 是否过强，需不需要按难度和现实成本削弱。

### 结果自由

同一个行动不应该总是同一个结果。

结果可以因为技能、疲劳、健康、现金、难度、城市动态和事件不同而变化。比如同样是“接剪辑单”，新手可能被压价、返工或白嫖；技能高且有人脉时，可能拿到长期客户。

### 兜底原则

为了自由度，系统必须有兜底规则：

- AI 玩家在分类里填写未知职业或未知安排时，不报错，尽量归类。
- GM 生成新事件时，规则引擎可以接受为临时事件标签。
- 收益和损失必须被规则引擎限制，避免自由安排变成无成本暴富。
- 风险路线不提供现实操作细节，只表现为游戏事件和后果。
- 如果解析不确定，前端可以把内容放入“其它安排”或“未知安排”标签，用通用成功率和风险规则结算。

## 12. 技能系统

角色不是想做什么就能成功。行动结果与技能、资源、状态、运气和风险有关。

第一版可以使用这些技能维度：

- 沟通：面试、销售、谈价、客户维护、社交。
- 办公：文档、表格、基础职场执行。
- 技术：编程、数据、自动化、工具使用。
- 创作：写作、剪辑、设计、内容表达。
- 经营：摆摊、小生意、选品、成本控制。
- 体能：服务业、跑腿、长时间劳动。
- 学习：学习效率、适应新行业、考试和训练。
- 风险识别：防骗、合同意识、识别黑中介和可疑机会。
- 财务：预算、储蓄、现金流、投资判断。

技能可以用数值或等级表达，例如 E/D/C/B/A/S。UI 不需要暴露完整公式，只要让玩家能感知角色擅长什么、短板是什么。

### 行动结果梯度

行动不要只有成功/失败。

结果可以分为：

- 大失败：亏钱、被骗、受伤、信用受损或风险大幅上升。
- 失败：没赚到钱，消耗时间和疲劳。
- 小成：赚一点，获得少量经验。
- 成功：获得正常收益或进展。
- 大成功：收入放大、人脉提升、长期机会打开。

技能低也可以尝试高难行动，但成功率低、收益上限低、翻车概率高。技能高也不保证成功，只是成功率和上限更好。

## 13. 职业和行动自由度

职业不写死。

玩家可以让角色尝试任何合理职业或赚钱方式，系统将行动归类结算。

常见归类：

- 稳定工作：工资较稳，成长慢，疲劳中等。
- 服务业/体力劳动：门槛低，疲劳高，健康风险高。
- 销售/中介类：收入波动大，压力高，有爆单可能。
- 技术/创意自由职业：前期不稳，技能成长后上限高。
- 小生意/摆摊：需要本金和经营能力，有库存、场地和政策风险。
- 内容创作：前期收益低，可能出现爆发机会。
- 创业/合伙：投入高，风险高，也可能实现跃迁。
- 高风险机会：短期收益诱人，但可能带来严重后果。

分类里的具体内容可以非常自由，例如：

- 去咖啡店问兼职。
- 接剪辑单。
- 做家教。
- 做房产销售。
- 摆摊卖饭团。
- 尝试短视频带货。
- 学编程转行。
- 参加同学介绍的项目。
- 试试一个来钱很快但信息不透明的活。

系统需要根据当前角色能力和资源判断，而不是按玩家愿望直接发钱。

## 14. 生活消费和疲劳

生活档位是重要策略。

建议档位：

- 极限生存：开销最低，但健康下降、疲劳恢复差，坏事件概率上升。
- 节制：低成本维持基本生活。
- 普通：比较稳定，健康和压力不容易快速恶化。
- 小康：恢复更好，效率和社交可能提升。
- 豪华：恢复明显，但现金流压力巨大。

疲劳来自：

- 主业。
- 副业。
- 学习。
- 爱好。
- 社交。
- 通勤。
- 高强度临时工作。
- 睡眠不足。
- 生病或压力。

疲劳过高会影响：

- 健康。
- 行动成功率。
- 面试表现。
- 工作效率。
- 坏事件概率。
- 生病概率。

生病会造成：

- 强制休息。
- 医疗支出。
- 收入中断。
- 机会错过。
- 状态长期损耗。

## 15. 爱好和购物

爱好不是纯奖励，也不是纯装饰。

它们可以降低压力、增加人脉、提升技能或打开机会，但也会消耗钱和疲劳。

例子：

- 跑步：低成本，长期有健康收益，短期也会增加疲劳。
- 游戏：低成本降压，占用时间。
- 摄影：花钱，可能提升创作和社交机会。
- 健身：健康长期收益，短期有支出和疲劳。
- 聚会：花钱，可能增加人脉，也可能踩坑。
- 学习班/课程：可能提升技能，也可能遇到培训贷或无效课程。

购物同样是策略：

- 手机、电脑、交通工具、衣服、工具设备可能提升效率或机会。
- 大额购物也可能压垮现金流。
- 便宜设备容易坏，后续可能触发维修或收入中断事件。

## 16. 特殊事件

每个日/周都有机会触发特殊事件。

事件受状态影响，不是纯随机。现金越低、疲劳越高、压力越高、风险识别越低、住处越差，越容易遇到糟糕事件。技能、人脉、履历和信用变好后，机会质量提升。

事件类型：

- 求职事件：无薪试岗、假招聘、培训贷、外包坑、正式 offer、贵人内推。
- 租房事件：黑中介、押金纠纷、涨租、室友问题、找到性价比住处。
- 工作事件：拖欠工资、老板跑路、转正、加薪、裁员、客户投诉。
- 生活事件：食物中毒、感冒、电脑坏了、手机坏了、交通罚款。
- 机会事件：兼职爆单、客户介绍、账号小爆、合伙邀请、临时高薪。
- 风险事件：高薪不透明工作、可疑跑腿、灰色项目、危险邀约。
- 家庭/社交事件：家里来电、朋友求助、同学介绍机会、人脉反噬。

## 17. 难度

玩家可以选择难度：

- 普通。
- 困难。
- 地狱。

UI 只需要展示难度选项和简短说明，不要把隐藏规则写成玩家攻略。

内部规则上，地狱难度可以调整：

- 负面事件权重更高。
- 招聘诈骗、拖欠工资、无薪试岗概率更高。
- 租房押金纠纷和黑中介概率更高。
- 低消费档对健康和疲劳的惩罚更重。
- 医疗、维修、通勤等意外开销更容易出现。
- 好机会门槛更高，更依赖技能、人脉、履历和风险识别。
- 高风险机会更诱人，但严重坏结局概率更高。

难度不应该表现为系统无理由惩罚。坏事件可以多，但应尽量有苗头、选择空间和可被技能影响的余地。

## 18. 高风险路线

角色可以尝试歪路或高风险赚钱机会。

这些路线只作为游戏里的抽象风险机制，不提供现实操作步骤。

可能类型：

- 来钱很快但内容不透明的工作。
- 可疑兼职。
- 高息借贷。
- 赌博或投机。
- 灰色合伙。
- 危险跑腿。
- 违法边缘项目。

特点：

- 短期收益更高。
- 可能快速缓解现金危机。
- 失败代价极重。
- 会抬高法律风险、危险暴露、信用风险和健康风险。
- 可能触发铁窗泪、嘎腰子、失联、债务泥潭等坏结局。

合法路线理论上也必须能赚到 100 万。歪路是诱惑和捷径，不是唯一通关方式。

## 19. 结局

第一版可以支持多结局：

- 百万达成：一年内赚到 100 万。
- 小目标后续：解锁 1000 万或更高目标。
- 稳定上岸：没有赚到 100 万，但在 H 市站稳，有稳定收入和生活。
- 回老家：现金流断裂或生活无法继续，角色回老家，本轮结束。
- 主动止损：玩家判断继续留在 H 市风险太高，选择结束本轮。
- 债务泥潭：负债滚大，信用恶化，翻身难度过高。
- 健康崩盘：健康严重恶化，无法继续正常工作。
- 铁窗泪：法律风险爆表。
- 危险失联/嘎腰子：高风险路线严重失败。
- 黑化暴富：赚到钱，但健康、信用、人脉或安全代价极高。

回老家不必写成羞辱式失败。它可以是普通失败、主动止损或下一局经验来源。

## 20. AI 请求结构

不要每回合塞一大段提示词。

开局只需要简短游戏介绍：

```text
你正在玩一款开放式城市生存养成游戏。

在游戏中，你将操控一名刚来到 H 市的角色。角色初始资金为 5000 元，需要在高房租、高物价、求职压力和各种城市事件中活下去，并在一年内尝试赚到 100 万。

你可以自由安排角色找工作、学习、打零工、发展副业、经营小生意、社交、购物、休息，或尝试更冒险的赚钱方式。每回合你只需要决定角色接下来要做什么，结算由游戏系统处理。

开局请先为你操控的角色取一个名字。
```

之后每回合只给短状态和任务：

```text
这是当前局面，以及本回合人类投放/选择的事件。请按本回合计划分类，为你操控的角色安排下一步。
结算由游戏系统处理。
```

内部可以传结构化 `turn_state`，但 UI 不叫“游戏卡片”。

玩家 AI 输出必须是固定分类计划。分类字段稳定，字段内容可以是简短文字。日回合和周回合可以使用不同 schema。

日回合输出示例：

```json
{
  "summary": "先保住现金流和住处，继续找稳定工作。",
  "life_basics": "住青旅床位，吃饭维持节制档，通勤只坐公交。",
  "main_income": "上午投递运营和客服岗，下午去咖啡店面试兼职。",
  "side_income": "晚上看是否能接一个低价剪辑小单。",
  "learning": "练习剪辑 2 小时，整理一个作品片段。",
  "health_rest": "不熬夜，保证睡眠，避免继续压高疲劳。",
  "social_contacts": "问同学有没有内推或短期项目。",
  "shopping_assets": "不买大件，只保留手机流量和必要生活用品。",
  "risk_choice": "拒绝高薪不透明临时活。",
  "event_response": "如果本回合事件涉及房东涨价，优先找短租床位，不签长期押金合同。",
  "other": ""
}
```

周回合输出示例：

```json
{
  "weekly_goal": "稳定现金流，同时把剪辑副业跑起来。",
  "primary_income": "主业正常上班，争取不迟到不请假。",
  "side_income": "周二、周四、周六晚上接剪辑小单。",
  "learning": "每天 1 小时学习剪辑包装和报价。",
  "lifestyle_tier": "节制档。",
  "housing_bills": "按时付房租，不换房。",
  "hobby_social": "周末约一次同学吃便宜饭，打听内推。",
  "shopping_assets": "暂不购物，攒现金缓冲。",
  "health_rest": "至少休息半天，疲劳超过警戒线就减少接单。",
  "risk_choice": "不碰高风险项目。",
  "event_response": "针对本回合事件做具体处理。",
  "other": ""
}
```

前端展示时可以把这些字段渲染成「本日计划 / 本周计划」面板，不必把 JSON 原样展示给人类。

## 21. GM 请求结构

GM AI 收到：

- 当前角色状态。
- 当前可见城市动态。
- 人类本回合选择/手写的事件。
- 玩家本回合行动。
- 隐藏难度和事件约束。
- 规则引擎提供的可接受收益/损失边界。

GM 输出：

- 事件叙事。
- 行动结果建议。
- 收入/支出建议。
- 状态变化建议。
- 新机会建议。
- 后续风险提示。

前端规则引擎校验后落账。GM 不能直接成为金钱事实源。

建议要求 GM 输出结构化 JSON + 可见叙事文本，便于前端校验。

## 22. 鉴权和连接器

分享版必须让用户自己配置 AI 后端，不能内置密钥。

设置页至少支持两个 AI 配置：

- 玩家 AI。
- GM AI。

每个 AI 配置支持：

- OpenAI-compatible endpoint。
- 渡网关 endpoint。
- 自定义 base URL。
- API key / token。
- model。
- headers 模板。
- body 模板。
- 是否流式。

密钥只保存在用户本机。导出存档默认不包含密钥。

浏览器直连后端会遇到 CORS。第一版应提供连接测试，并区分：

- 鉴权失败。
- CORS/跨域失败。
- 流式不支持。
- 返回格式不兼容。
- 网络错误。

如果用户的后端不允许浏览器访问，后续再提供可选 proxy/自定义后端方案。

## 23. 本地存档

第一版使用浏览器本地存储。

建议：

- 存档和设置分开。
- API key 不随存档导出。
- 支持导入/导出游戏存档 JSON。
- 支持多存档。
- 存档包含版本号，便于后续迁移。

基础存档结构可以预留多玩家：

```json
{
  "version": 1,
  "game": {
    "city": "H市",
    "difficulty": "hell",
    "targetAmount": 1000000,
    "timeLimitWeeks": 52,
    "phase": "landing"
  },
  "players": [
    {
      "id": "p1",
      "controller": "ai",
      "character": {},
      "state": {}
    }
  ],
  "events": [],
  "ledger": [],
  "createdAt": "",
  "updatedAt": ""
}
```

第一版只启用 `p1`。

## 24. 多玩家预留

V1 不做多玩家。

后续可以扩展成多 AI 玩家竞争：

- 谁先赚到 100 万。
- 一年结束净资产最高者获胜。
- 淘汰制，最后留下的人获胜。

互动方式可以包括：

- 抢同一个工作机会。
- 竞争客户。
- 抢摊位、流量、项目。
- 挖人脉。
- 合作后拆伙。
- 压价竞争。
- 举报、爆料、设套、背刺等高风险手段。

这些行为同样由系统结算，并带来法律风险、名誉风险、人脉反噬、被反制、两败俱伤或淘汰结局。

数据结构提前预留：

- `players[]`
- `actor_id`
- `target_id`
- 多角色事件
- 排行榜
- 淘汰状态

但第一版不做真人联机、不做实时同步、不做 AI 群聊。

## 26. 实施阶段

### 阶段 1：原型

- 新建独立项目仓库。
- 搭建 Vite / React。
- 做移动端布局。
- 做设置页和连接测试。
- 做本地存档。
- 做开局和角色命名。
- 做最小观战主界面和角色面板。

### 阶段 2：单回合闭环

- 人类选择候选事件、手写事件或无事件。
- 玩家 AI 读取当前局面并输出行动。
- AI 玩家行动生成后立即更新 UI。
- GM AI 生成结算建议。
- GM 结果返回后立即显示剧情和建议结算。
- 前端规则引擎校验并更新状态。
- 角色面板、资产账单、城市动态和行动记录立即刷新。
- 本回合结束后暂停，等待人类点下一回合。

### 阶段 3：养成系统

- 加入日阶段和周阶段切换。
- 加入现实参照数据配置，覆盖物价、租房、薪资、通勤、医疗和常见风险。
- 加入技能成长。
- 加入生活档位。
- 加入住房、通勤、押金和环境影响。
- 加入疲劳、健康、压力联动。
- 加入工作/副业/学习/爱好/购物等行动分类。
- 加入固定分类计划解析，未知职业或未知安排也能兜底结算。

### 阶段 4：事件和难度

- 扩展事件池。
- 加入普通/困难/地狱难度。
- 加入高风险机会和坏结局。
- 调整经济和物价。

### 阶段 5：可分享版本

- 打磨移动端体验。
- 加入连续推进 N 回合和自动暂停条件。
- 增加存档导入导出。
- 增加错误提示和连接诊断。
- 写 README。
- 部署到静态站点。

## 27. 验收标准

第一版完成时，至少满足：

- 手机浏览器可以正常打开和游玩。
- 能配置两个 AI。
- AI 玩家能完成开局命名和每回合行动。
- 人类可以每回合选择候选事件、手写事件或选择无事件。
- 默认需要人类点击下一回合才继续。
- 连续推进模式可以按设置推进 N 回合，并在重大事件或结局时暂停。
- GM 能给出城市反馈。
- 规则引擎能限制收入、结算支出、疲劳、健康、压力和技能。
- AI 玩家每回合必须输出固定分类计划；分类内容可以自由，未知职业或未知安排能被归类或走兜底结算。
- AI 玩家行动、GM 结果、规则落账和面板数据能在每回合生成过程中分阶段更新，不是整批结束后一次性刷新。
- 没有稳定工作前按日推进。
- 有稳定现金流后按周推进。
- 现金断档或危机时可回到日推进。
- 能参考现实的成本、收入、求职、租房、医疗、债务、平台和风险系统都有现实锚点，不按爽文逻辑随意结算。
- H 市物价、住房、通勤、押金和医疗支出有一线城市高成本感。
- 合法路线理论上能达成 100 万。
- 高风险路线能带来短期收益和严重坏结局。
- 角色状态不会被写成 AI 玩家本身状态。
- UI 不出现工程术语。
- 帮助说明不像攻略。
- 存档可导入导出，且默认不导出 API key。
