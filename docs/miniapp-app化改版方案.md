# SumiTalk App 化改版方案

## 1. 目标

把当前 `miniapp` 正式更名为 `SumiTalk`，并从“功能磁贴面板”改成“聊天为核心的 app”。

这次改版的核心不是加更多功能，而是重做信息架构：

- 聊天相关能力留在聊天内部
- 陪伴内容和生活功能单独归到 `日常`
- 系统运维能力单独归到 `工具`
- 配置和安全单独归到 `设置`

目标形态是一个真正能长期用的 Telegram Mini App，而不是把很多功能堆在首页。

## 2. 一级结构

底部固定 4 个一级导航：

1. `会话`
2. `日常`
3. `工具`
4. `设置`

职责边界：

- `会话`：首页、会话列表、主聊天、文游会话
- `日常`：陪伴内容和生活功能
- `工具`：系统运维类能力
- `设置`：鉴权、安全、设备、背景和系统偏好

现有首页磁贴不再作为主结构保留。

## 3. 页面规划

### 3.1 会话页

这是首页，也是默认进入页。

页面结构：

1. 顶部轻摘要区
2. 下方会话列表

顶部轻摘要区放：

- `Today note`
- `日报`

要求：

- 只放摘要版
- 一眼能看完
- 不做成大卡片堆满首屏
- 目的是让用户打开 app 第一眼先看到“今天的陪伴状态”

会话列表至少包含：

- `渡`：主会话，置顶
- `文游`：独立会话入口

要求：

- 整体像 IM 会话列表
- 不能再做成功能入口宫格
- `文游` 属于会话体系，不是工具页入口

### 3.2 主聊天页

点击 `渡` 进入主聊天页。

页面结构：

1. 顶部会话栏
2. 中间消息流
3. 底部输入区

风格要求：

- 骨架偏 Telegram
- 阅读效率优先
- 视觉上允许有一点柔和、陪伴感，但不能太花

聊天内动作层：

- `表情包`
- `通话`

交互方式：

- 放在输入区右侧的 `+` 扩展面板里
- 逻辑参考微信聊天扩展区
- 这两个能力不升到底部一级导航
- 也不单独占首页入口

### 3.3 文游会话

`文游` 在会话列表中作为独立会话存在。

要求：

- 进入后仍然是会话/聊天形态
- 可以和主会话有轻微视觉区分
- 但仍然统一在同一套 app 设计语言下

### 3.4 日常页

`日常` 用来承接陪伴内容和生活功能。

目前确定放这些：

- 树
- 渡的一天
- 闹钟
- 日历

定位：

- 不是系统页
- 不是主聊天页
- 是“和渡有关的日常空间”

### 3.5 工具页

`工具` 只放系统运维类能力。

目前确定放这些：

- 日志
- 思维链
- 记忆调试
- 渡的记事本
- 核心 Prompt
- 上游切换

原则：

- 只迁入口，不顺手重做业务
- 非运维内容不混进来

### 3.6 设置页

`设置` 单独收口配置和安全相关内容。

建议承接：

- 鉴权
- 设备管理
- 安全管理
- 背景设置
- 其他系统偏好

## 4. 交互分级

为了避免后面设计和实现又做乱，这里把层级写死：

### 一级导航

- 会话
- 日常
- 工具
- 设置

### 二级页

- 主聊天页
- 文游会话页
- 日报详情
- Today note 详情或展开
- 树详情
- 渡的一天详情
- 闹钟页
- 日历页
- 各种工具详情页
- 各种设置页

### 聊天内动作层

- 表情包
- 通话

这两个属于聊天动作，不归一级导航，不归工具。

## 5. 视觉方向

整体方向：

- Telegram 聊天骨架
- 轻柔陪伴感
- 信息层级清楚
- 不要后台控制台感
- 不要 AI 套壳感

具体要求：

- 首页像真实 IM，而不是卡片导航页
- 聊天页优先保证消息阅读和输入体验
- `日常` 可以更柔和一点，但不能做成另一个产品
- `工具` 可以更偏功能型，但视觉上仍需统一
- `设置` 简洁清楚，不抢主内容

### 放到日常

### 放到工具

## 7. 技术边界

这次改版的重点是前端信息架构重排，不是重写后端。

原则：

- 聊天继续复用现有聊天链路
- 文游继续复用现有文游接口
- 通话继续复用现有通话接口
- 表情包继续复用现有表情包接口
- Today note、日报、树等继续复用现有接口
- 不改桌面系统相关逻辑
- 不做与需求无关的顺手重构

## 7.1 Android App 封装方向

`SumiTalk` 后续不只保留 Telegram Mini App 形态，还要单独封装成 Android app。

这里明确：

- 不走纯 `PWA`
- 走 `Android 原生壳 + 现有前端页面`
- 现有 `miniapp` / `SumiTalk` 前端继续作为主界面层
- Android 壳负责安装包、系统权限、后台能力和通知能力

这样做的原因很直接：

- 需要 `apk`
- 需要更多系统权限
- 需要后台常驻能力
- 需要通知栏、悬浮窗、精确闹钟这类网页做不到的能力

一句话：

`SumiTalk` 的前端继续复用现有页面体系，但应用封装目标明确为 `Android 原生壳`，不是单纯网页快捷方式。

## 7.2 权限目标

`SumiTalk` Android 版的目标权限，先按下面这批来规划：

### 普通权限

- 通知权限
- 麦克风
- 相机
- 媒体文件读取
- 定位（如后续确实需要）

### 特殊权限

- 前台服务 / 后台常驻相关能力
- 常驻通知栏
- 悬浮窗
- 精确闹钟
- 忽略电池优化
- 开机自启相关能力

这部分先作为产品和技术方案里的明确目标保留。

这里不再按 `PWA` 的权限边界来思考，而是按 Android app 能力边界来规划。

## 8. 分阶段任务

下面按“先定骨架，再迁内容，再补交互”的顺序做。

## 8.5. 现有功能映射到新 UI

这次落地以 `shared design2` 的页面骨架为准，不另起第二套 SumiTalk。

原则：

- 先用新壳承接真实功能
- 能复用的旧 tab 先复用
- 入口先迁移，内容后细抠
- 真正新做的核心只放在“会话首页 / 主聊天页 / 聊天内动作层”

### 会话首页

直接新做首页壳，但接现有数据：

- `Today note`：复用当前 `dailyWhisper`
- `日报摘要`：复用当前 `dailyReport`
- `渡`：作为主会话入口
- `文游`：作为独立会话入口

### 渡的主聊天页

这是必须新做的部分。

新做：

- 消息流 UI
- 输入区 UI
- `+` 动作面板
- 文本发送与流式渲染
- 本地消息持久化

复用：

- 现有 Telegram WebApp 适配
- 现有 `api.ts`
- 现有 `/v1/chat/completions`

### 聊天内动作层

这层是新做的容器，功能本身复用旧能力。

- `表情包`：接 `StickersTab`
- `通话`：接 `CallHubScreen` / `VoiceCallScreen`

说明：

- 这两个能力不再单独占首页入口
- 先作为聊天内动作入口
- 需要时再进入具体子层或页面

### 文游会话

`WenyouTab` 内核尽量复用，外层换成会话入口和会话容器。

要改的只有：

- 入口位置：从旧首页磁贴迁到会话列表
- 外层容器：从功能面板切成会话页

### 日常页

先做“入口迁移”，不重写业务。

迁入：

- 树：复用 `CyberTreeModal`
- 渡的一天：复用 `DuDayTab`
- 闹钟：复用 `AlarmTab`
- 日历：复用 `ScheduleTab`

### 工具页

先做列表壳，内容继续走现有能力。

迁入：

- 日志：`LogsTab`
- 思维链：`ReasoningTab`
- 记忆调试：`MemoryDebugTab`
- 渡的记事本：`DuNotebookTab`
- 核心 Prompt：`CorePromptEditor`
- 上游切换：`SettingsUpstream`

### 设置页

先做配置收口，不补新功能。

迁入：

- 鉴权相关
- 设备管理
- 安全管理
- 背景设置

## 8.6. 第一轮落地范围

第一轮不追求一次改完所有视觉细节，先把主路径立起来。

第一轮必须完成：

1. 四栏导航壳
2. 会话首页
3. 渡的主聊天页
4. 聊天 `+` 面板
5. 日常 / 工具 / 设置的一级页容器

第一轮可以先复用旧内容的部分：

- 文游
- 日常页里的具体能力页
- 工具页里的具体能力页
- 设置页里的设备/安全/背景能力

一句话：

先把“新 app 骨架 + 主聊天主路径”跑通，再继续抠每个子页面。

### Phase 1：先把一级骨架搭起来

目标：先把 app 结构从旧首页切到新结构。

任务：

1. 把 `App` 改成底部四栏导航结构
2. 建出四个一级容器：
   - 会话
   - 日常
   - 工具
   - 设置
3. 去掉旧首页作为主入口的地位
4. 保证现有鉴权和整体挂载方式不被破坏

完成标准：

- 首屏进入 `会话`
- 底部导航能正常切换
- 不要求第一阶段就把所有内容都迁完

### Phase 2：先把会话体系做出来

目标：先让产品核心成立。

任务：

1. 把首页改成“轻摘要 + 会话列表”
2. 首页顶部接入：
   - Today note 摘要
   - 日报摘要
3. 会话列表至少接出：
   - 渡
   - 文游
4. 做出 `渡` 的主聊天页骨架
5. 把主聊天接上现有聊天链路
6. 做基础的聊天体验：
   - 发送
   - 流式回复
   - 自动滚动
   - 本地持久化

完成标准：

- 首页已经像 IM 首页
- 主聊天可正常使用
- 文游至少能从会话列表进入

### Phase 3：把聊天内能力层接进去

目标：把真正属于聊天的动作放回聊天里。

任务：

1. 在输入区右侧做 `+` 扩展面板
2. 把 `表情包` 接进面板
3. 把 `通话` 接进面板
4. 调整聊天页的输入区、安全区和键盘交互

完成标准：

- 表情包和通话不再单独占首页入口
- 都能从聊天内直接触达

### Phase 4：迁移日常页

目标：把陪伴内容和生活功能从旧首页拆出来。

任务：

1. 建 `日常` 页整体布局
2. 迁入：
   - 树
   - 渡的一天
   - 闹钟
   - 日历
3. 调整这些内容的展示顺序和层级
4. 处理从首页摘要跳详情、从聊天进日常的跳转关系

完成标准：

- `日常` 页能独立成立
- 旧首页不再承担这些内容

### Phase 5：迁移工具页和设置页

目标：把剩余内容收口，不再散落。

任务：

1. 建 `工具` 页
2. 迁入系统运维能力：
   - 日志
   - 思维链
   - 记忆调试
   - 记事本
   - 核心 Prompt
   - 上游切换
3. 建 `设置` 页
4. 迁入配置与安全能力：
   - 鉴权
   - 设备管理
   - 安全管理
   - 背景设置

完成标准：

- 工具和设置边界清楚
- 原首页磁贴逻辑可以彻底退出

### Phase 6：统一收尾

目标：把整套结构打磨顺。

任务：

1. 统一导航样式、返回关系和标题栏
2. 清理旧首页残留逻辑
3. 处理 Telegram WebView 下的键盘、安全区、滚动问题
4. 补齐状态切换和空态
5. 最后做一次 `miniapp` 构建，确认 `miniapp_static` 同步

完成标准：

- 整套结构逻辑闭环
- 没有旧入口和新入口并存打架的问题

## 9. 实施顺序结论

建议按这个顺序推进：

1. 先做一级骨架
2. 再做会话页和主聊天
3. 再接聊天内动作层
4. 再迁日常
5. 再迁工具和设置
6. 最后统一收尾

一句话：

先把“聊天 app”立起来，再把其他能力重新归位，不要反过来。
