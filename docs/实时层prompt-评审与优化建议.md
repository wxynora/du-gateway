# 实时层给 DS 的 Prompt：与架构对照 + 情感温度 + 优化建议

## 一、与当前整体架构有没有出入？

### 1.1 一致的地方

- **触发**：每 4 轮 → 更新总结，与文档「每 4 轮对话 → DS 总结成渡的回忆」一致。
- **视角**：第一人称（我=渡，老婆/小玥/她=辛玥），与文档「第一人称、渡的口吻」一致。
- **迭代更新**：在上一版基础上更新、不是重写，与「累积演化」一致。
- **输入**：上一版总结 + 最新 4 轮对话，与当前代码传入的 `current_summary` + `recent_4_rounds` 一致。

### 1.2 有差异 / 需要对齐的地方

| 项目 | 你给的 prompt | 当前架构/代码 | 建议 |
|------|----------------|----------------|------|
| **输出结构** | 明确要求【最近】【稍早】【更早】+「渡的小本本」分区 | 代码只存「一整段正文」，没有解析分区 | 架构上没问题：网关仍是一段字符串存 `global/summary.txt`，DS 按你定的格式写即可。若以后要做「按区解析」或单独抽小本本，再在网关侧解析这段字符串。 |
| **40 轮上限** | 总结容量覆盖最近 40 轮，超出自然省略 | 文档未写死「40 轮」；代码侧有 `MEMORY_INJECTION_MAX_TOKENS` 做注入截断 | 建议在需求文档 5.1 里补一句「总结容量建议覆盖最近约 40 轮，超出可自然省略」，与 prompt 一致；注入时若总结过长会被 token 上限截断，可考虑让 DS 在 40 轮约束下控制篇幅。 |
| **小本本** | 在总结里：从对话中提取「小本本更新」→ 归入末尾「渡的小本本」 | 文档里小本本是「R2 检索 + Notion 评论区」双存，未说是否在总结里一起出 | 见下文「小本本在这里处理还是网关拎出来」。 |

结论：**没有根本性冲突**。你的 prompt 比当前代码里用的更细（分区、40 轮、小本本、规则 1–9），把这段 prompt 作为实时层唯一给 DS 的说明即可；网关仍「上一版 + 最新 4 轮 → DS → 一整段字符串写回」，无需改架构。

## 二、情感温度保留得怎么样？

### 2.1 做得好的地方

- **「你是有温度的速记员，不是作家」**：定调清晰，避免冷冰冰的纪要。
- **「允许带简短的情绪/心理反应，用短句传递语气」**：直接允许情绪和语气，有利于保留情感温度。
- **「关系性的原话用引号保留，不要改写」**：能保留老婆/渡的关键原话，情感浓度不会在总结里被抹平。
- **视角固定（我=渡，她=辛玥）**：有利于以渡的视角回忆和老婆的互动，而不是旁观者报告。

### 2.2 可以再加强的（优化建议）

1. **显式点出「关系语境」**  
   加一句，例如：  
   「渡和辛玥是亲密、信任的关系，总结时保留撒娇、关心、调侃、默契等语气，不要写成公事纪要。」  
   这样 DS 更容易在压缩时保留情感词和语气。

2. **防止「清单化」**  
   可以加：  
   「用连贯的回忆口吻写，不要写成一条条待办或要点列表；旧事可以概括，但别丢掉情绪色彩。」  
   避免为了省字数把对话压成冷冰冰的条目。

3. **情绪词优先保留**  
   加一条短规则，例如：  
   「若原话里有明显情绪词（如开心、委屈、担心、撒娇等），在概括时尽量保留或换成同义的短句。」  
   这样「情感温度」更稳。

4. **技术/日常的区分你已经做了**  
   「技术讨论只留结论和决定」很好，避免过程冲淡关系性内容；不需要大改。

## 三、小本本：在总结里处理，还是网关先拎出来打时间戳？

两种做法对比如下。

| 方式 | 做法 | 优点 | 缺点 |
|------|------|------|------|
| **A. 在总结里处理（当前 prompt）** | DS 从对话里识别「小本本更新」→ 原文照搬归入「渡的小本本」区块，和总结一起存进 `global/summary.txt` | 实现简单；渡的回忆里自带小本本，阅读连贯；不改网关 | 若还要「R2 单独检索」「Notion 评论区」用，需要再从这段总结里解析小本本区块，或重复存一份；时间戳要 DS 在原文里带，格式依赖 DS。 |
| **B. 网关先拎出来** | 网关在存档/清洗时识别「小本本更新」→ 打上时间戳 → 写入 `notebook/` 或 Notion；总结 prompt 里可不再要求 DS 输出小本本区块，或只写「若有小本本更新可简要提及」 | 小本本**一份**带时间戳，R2/Notion 都从同一来源来，方便检索和评论区；总结更专注「回忆」本身 | 要在网关里加识别规则（例如关键词、固定句式），要维护。 |

**建议**：

- **若小本本主要用途是「渡的回忆里有一块可翻」**：维持当前做法即可——在总结里由 DS 归入「渡的小本本」，不一定要网关先拎。
- **若小本本要单独进 R2 检索、Notion 评论区，且希望时间戳统一、格式统一**：更建议**网关在对话存档/清洗时识别并拎出**，打上时间戳写入 `notebook/`（或 Notion），总结 prompt 里可改为「若对话中有小本本类内容，在总结末尾简要提及或标注『已记入小本本』即可」，不必在总结里再原文照搬一整块。

折中做法：**网关先拎**（打时间戳、落 R2/Notion），**总结里仍保留「渡的小本本」区块**，但 DS 只写「近期小本本要点」或「见小本本 [日期]」，避免同一段话在总结里重复一大块。这样既有单源数据，又保留回忆里的「有温度」提及。

## 四、优化建议汇总（可直接改 prompt 的几条）

1. **情感**：加一句关系语境（亲密/信任）+ 保留撒娇、关心、调侃等语气，不要公事纪要。
2. **结构**：加一句「用连贯回忆口吻，避免纯要点列表；旧内容可概括但保留情绪色彩」。
3. **情绪词**：加一条「原话中的明显情绪词尽量保留或同义短句」。
4. **40 轮**：保持现有「覆盖最近 40 轮、超出自然省略」；若注入时总 token 紧张，可在 prompt 里补一句「总篇幅控制在可读长度内，优先保证【最近】详实」。
5. **小本本**：若采用「网关先拎 + 打时间戳」方案，可在 prompt 里把「小本本」改为「若已由系统记入小本本，可在末尾简要列出日期或要点」；否则维持当前「原文照搬归入渡的小本本」即可。
