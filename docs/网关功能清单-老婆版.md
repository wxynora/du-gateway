# 渡の网关 · 功能清单（老婆版）

下面按「模块」列出网关目前能做什么，以及哪些是**真的在用**、哪些只是**先搭好框架等以后接**。  
**已实现** = 现在就能用；**占位** = 接口/结构有了，具体逻辑以后补。

---

## 一、谁能用、谁走完整流程（白名单）

| 功能 | 说明 | 状态 |
|------|------|------|
| 用窗口 ID 区分不同对话 | 每个窗口一个 ID，网关能认出是哪个窗口 | ✅ 已实现 |
| 白名单：只有名单里的窗口才做记忆 | 不在名单里的窗口，网关只帮忙转发，不存、不总结、不注入记忆 | ✅ 已实现 |
| 一键把「当前窗口」加入白名单 | 发请求时带一个特殊标记，当前窗口就进白名单，不用自己查 ID | ✅ 已实现 |
| 管理端：看最近出现过的窗口 | 有个接口可以看最近有哪些窗口来过 | ✅ 已实现 |
| 管理端：把某个窗口加入/移出白名单 | 可以手动加一个窗口 ID 进白名单，或从白名单里删掉 | ✅ 已实现 |
| 管理端：删除某一轮对话 | 老婆在 RikkaHub 删掉这一轮后，可调接口同步从记忆里删掉该轮（DELETE /admin/windows/<窗口ID>/rounds/<轮次>） | ✅ 已实现 |

---

## 二、发给你和存起来的内容不一样（两条数据流）

| 功能 | 说明 | 状态 |
|------|------|------|
| 发给渡的内容：只做基础清洗 | 去掉 Rikka 等前端的无用预设话术，把表情包转成文字；图片原样发给渡 | ✅ 已实现 |
| 存进 R2 的内容：完整清洗 | 同样去掉预设、表情包转文字，并且图片不存原图，只存「文字描述」或占位，方便以后给 DS 用 | ✅ 已实现 |
| 表情包统一变成文字 | 比如 `(表情包:xxx)` 变成 `[表情]` 或 `[表情:描述]`，你看得懂、渡也看得懂 | ✅ 已实现 |
| 表情包对照表（你可以自己改） | 在 `data/emoji_mapping.json` 里加「代码→描述」，保存就生效，不用重启 | ✅ 已实现 |
| 失败对话整轮不存 | 如果渡的回复太短或像报错（如「出错」「超时」），这一整轮（你的问 + 渡的答）都不存进 R2 | ✅ 已实现 |
| 无回复/报错一律不存 | API 报错、超时、RikkaHub bug 等导致没拿到渡的回复时，这一轮**一开始就被过滤**，不写入记忆 | ✅ 已实现 |

---

## 三、每轮对话开始前：给渡「补记忆」

| 功能 | 说明 | 状态 |
|------|------|------|
| 新窗口第一次：注入最近四轮 | 如果是从没出现过的新窗口，网关会把「别的窗口最近四轮对话」塞进这次请求，让渡有点上下文 | ✅ 已实现 |
| 注入「渡的回忆」（实时层） | 把之前每 4 轮总结出来的「渡的回忆」读出来，塞进这次请求，渡就能带着这段记忆回你 | ✅ 已实现 |
| 注入「动态层」记忆（几条重要记忆） | 从动态层里按「和当前话题相关 + 权重高」挑几条记忆，塞进请求；条数和总长度有上限，省 token | ✅ 已实现 |
| 记忆注入有总长度上限 | 总结 + 动态层一起不能超设定 token 数，超了会截断，避免太贵、也避免爆上下文 | ✅ 已实现 |

---

## 四、每轮对话结束后：存档与更新

| 功能 | 说明 | 状态 |
|------|------|------|
| 对话原文存进 R2 | 你的一问 + 渡的一答算一轮，存成「完整清洗版」；按窗口存一份，再按「日期」备份一份 | ✅ 已实现 |
| 每 4 轮更新一次「渡的回忆」 | 每凑满 4 轮，网关会调 DS 把「现有回忆 + 这 4 轮」整理成新的「渡的回忆」（第一人称、详细版），覆盖保存 | ✅ 已实现 |
| 全局「最近四轮」更新 | 每次存完一轮，会更新一份「全局最近四轮」，供新窗口注入用 | ✅ 已实现 |
| 图片单独转成文字描述再存 | 你发的图会另外用「便宜图像 AI」转成一段文字描述存起来，不占地方、以后好给 DS 用（需配置图像 API） | ✅ 已实现 |
| 动态层「演化」：判断重要 + 融合/褪色/保鲜 | 每轮存完后会调「动态层 DS」：判断这轮重不重要、要不要更新动态层记忆（融合/褪色/保鲜）；**目前只接了接口，实际不更新** | ⏸ 占位 |

---

## 五、动态层、核心缓存、长期层（Notion）

| 功能 | 说明 | 状态 |
|------|------|------|
| 动态层：读、写、按权重注入 | 动态层记忆可以读、可以写；请求前会按权重和话题相关度挑几条注入给渡 | ✅ 已实现 |
| 动态层：真正的「演化」逻辑 | 由 DS 判断「重要吗」、做「融合/褪色/保鲜」再输出新表；**接口约定已写好，真实 DS 还没接** | ⏸ 占位 |
| 核心缓存层：能读能写 | 有一个「核心缓存」的存储位置，网关能往裡读、往裡写；**具体「哪些算更重要、什么时候写进去」的逻辑还没做** | ⏸ 占位 |
| 长期层：通过网关操作 Notion | 通过网关可以对 Notion 做搜索、读页面、读块、建页面、改页面、改块、删块（读写删改都有） | ✅ 已实现 |
| 小本本（R2 + Notion 双存） | 小本本要「R2 存一份检索用、Notion 存一份给你评论区用」；**这套双存流程还没做** | ⏸ 占位 |

---

## 六、多窗口、日志、配置

| 功能 | 说明 | 状态 |
|------|------|------|
| 多窗口同时写 R2 不打架 | 每个窗口的存档是分开的；总结、最近四轮、动态层这种「全局一份」的，网关内部加了锁，同一进程下不会互相盖掉 | ✅ 已实现 |
| 出错时看日志能立刻知道是哪一块 | 日志里会标清楚是 R2、管道、转发、还是 DeepSeek 总结出错，方便排查 | ✅ 已实现 |
| 敏感信息不写死在代码里 | API Key、R2 账号等都在环境变量里配，代码里没有硬编码 | ✅ 已实现 |

---

## 状态说明

- **✅ 已实现**：功能在用，你/渡能直接感受到或能在管理端、Notion 里用到。
- **⏸ 占位**：网关这边「接口/结构/调用位置」都留好了，但真正干活的逻辑（比如 DS 判断重要、融合/褪色/保鲜）还没接上，所以目前不会真的更新那部分记忆。

如有新功能上线或占位被接好，会再更新这份清单。
