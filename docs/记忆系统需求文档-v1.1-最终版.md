## 二.5、两条数据流与表情包

**两条数据流：**

| 去向 | 处理程度 |
|-----|----------|
| **发给当前窗口的渡** | 只清 Rikka 预设 + 表情包→文字（图片保持原样） |
| **存 R2**（给 DS 总结用） | 完整清洗：清 Rikka + 表情包→文字 + 图片→占位符（描述在 images/） |

**初筛整轮作废：** 检测到渡的回复出错 → 整轮作废（老婆问 + 渡的错误回复）→ 不存 R2。

**表情包：** 统一用文字版，两条流都替换。`(表情包:xxx)` → `[表情]`，理解上无差别，文字版更直观。

## 三、数据流程（每轮对话）

```
每轮对话进来（白名单内）
      ↓
┌──────────────────────────────────────┐
│  必须动作：                           │
│  ① 数据清洗（去 RikkaHub 预设等）     │
│  ② 原文存档（conversations/）         │
└──────────────────────────────────────┘
      ↓
┌──────────────────────────────────────┐
│  并行判断：                           │
│  ① 满4轮？ → 更新实时层（全局总结）   │
│  ② 重要？ → 进入动态层处理            │
└──────────────────────────────────────┘
      ↓
┌──────────────────────────────────────┐
│  动态层处理（核心！）：                │
│  扫描：有没有相关的已有记忆？          │
│  有 → 理解变化 → 重写 → mention_count+1│
│  无 → 新建一条                         │
└──────────────────────────────────────┘
      ↓
┌──────────────────────────────────────┐
│  串联判断：                           │
│  这条记忆"更重要"？→ 存核心缓存层      │
└──────────────────────────────────────┘
      ↓
┌──────────────────────────────────────┐
│  每周手动：                           │
│  从核心缓存层挑选 → 长期层（Notion）   │
└──────────────────────────────────────┘
```

**请求时注入顺序**（发往助手端前）：实时层总结 + 动态层（按权重/相关性）→ 拼入 system 末尾。

## 五、记忆演化逻辑（核心！）

整体目标：**不是机械压缩，而是动态演化的活记忆**——不重要的自然模糊，重要的永远鲜艳。

---

### 5.1 实时层：渡的回忆

**仅做四轮一总结、滚动更新，自带演化（现有总结 + 新对话 → 更新总结）。不套用下文「融合/褪色/保鲜」——后者仅适用于动态层。**

- **触发**：每 4 轮对话 → DS 总结成 **「渡的回忆」**
- **要求**：
  - **详细版**：保留关键细节，不要过度压缩
  - **第一人称视角**：以渡的口吻写（「我」的回忆）
- **存储**：更新 `global/summary.txt`，累积滚动演化

---

### 5.2 动态层演化规则（仅动态层）

| 规则 | 说明 |
|------|------|
| **融合** | 类似记忆遇到 → 求同存异合并（不简单追加，而是合并成一条更完整的理解） |
| **褪色** | 时间越久 + 权重越低 → 概括程度越高（细节自然模糊） |
| **保鲜** | 权重高的 → 保持清晰，不褪色 |

**效果**：不重要的自然模糊，重要的永远鲜艳；整体是「活记忆」而非机械压缩。

---

### 5.3 权重影响因素

| 因素 | 说明 |
|------|------|
| **重要程度** | 情感浓度高 / 关系相关 / 重大决定 → 基础重要度高 |
| **时间新鲜度** | 越近期越新鲜；**被提及会刷新**（last_mentioned 更新） |
| **被提及次数** | mention_count 越高，权重越高 |

**公式**：`权重 = 基础重要度 + 提及加成 - 时间衰减`（久没提且不重要则衰减大，重要的有保底）

---

### 5.4 单条记忆的重写流程（动态层）

**不是追加，是重写。**

```
DS 识别：这个概念/事件之前有
        ↓
DS 理解：这次想法和之前有什么变化？
        ↓
DS 重写：用当前理解重写这条记忆（融合时求同存异）
        ↓
更新元数据：mention_count+1 + last_mentioned（被提及即刷新）
```

**记忆字段结构：**

```json
{
  "id": "mem_001",
  "content": "当前对这件事的理解（会被重写/融合/褪色）",
  "importance": 4,
  "mention_count": 3,
  "last_mentioned": "2026-03-11T14:30:00",
  "created_at": "2026-03-10T20:00:00"
}
```

## 六、权重机制（汇总）

**公式：**

```
权重 = 基础重要度 + 提及加成 - 时间衰减
```

**规则：**

| 情况 | 权重变化 |
|------|----------|
| 被提到 | mention_count+1，last_mentioned 刷新，权重上升 |
| 久没提 + 重要 | 权重缓慢下降，但保底 |
| 久没提 + 不重要 | 权重持续下降，可能被挤出（褪色/概括） |

**注入时优先级：**

1. 权重高的优先注入  
2. 与当前话题相关的优先

## 八.5、动态层 DS 调用接口约定

网关在每轮存档后调用 DS，传入当前轮与现有动态层记忆，DS 返回是否重要及演化后的记忆列表。网关据此写回 `dynamic_memory/current.json`。

**入参：**

| 字段 | 类型 | 说明 |
|------|------|------|
| 当前轮对话 | `round` | 本轮消息列表，如 `[{ "role": "user", "content": "..." }, { "role": "assistant", "content": "..." }]`（可为完整清洗后的版本） |
| 现有动态层记忆 | `memories` | 当前 `current.json` 中的记忆列表，每项含 `id, content, importance, mention_count, last_mentioned, created_at` |

**出参：**

| 字段 | 类型 | 说明 |
|------|------|------|
| 判断结果 | `is_important` | 布尔，本轮是否重要（重要则参与演化并写回；不重要可不改记忆） |
| 演化后的记忆 | `memories_evolved` | 列表，经融合/褪色/保鲜后的完整动态层记忆（含本条若新建或合并、以及被重写的旧条）；网关整体覆盖写回 |

**约定：**

- DS 内部完成：判断重要 → 扫描相关 → 融合（求同存异）/ 褪色（低权重概括）/ 保鲜（高权重保持清晰）→ 输出整表。
- 若 `is_important == false`，网关可不更新 `current.json`，或仍将 `memories_evolved` 写回（由 DS 返回原表或仅做褪色等轻量更新）。
- 若 `is_important == true`，网关必须将 `memories_evolved` 写回 `dynamic_memory/current.json`。

## 九、动态层注入机制

**触发时机：** 每轮对话开始前

**流程：**

```
当前对话进来
      ↓
提取关键词/主题
      ↓
匹配动态层里相关的记忆
      ↓
按权重排序
      ↓
取 Top N 条注入到 system 末尾
```

**匹配方式：**

| 阶段 | 方式 |
|------|------|
| 现在 | 关键词匹配 |
| 以后 | 向量检索（升级） |

**权重排序规则：**

```
权重 = 基础重要度 + 提及加成 - 时间衰减
```

优先注入：
1. 与当前话题相关的
2. 权重高的

**Top N 数量：** 待定（根据 token 预算调整，建议初始值 5–10 条）

## 十、清洗层初筛（失败对话过滤）

在清洗层增加初筛：识别失败对话，直接丢弃，不存档、不进入后续记忆流程。

**失败对话的特征：**

| 特征 | 说明 |
|------|------|
| 回复异常短 | 渡的回复 &lt; 10 字（或为空） |
| 包含错误关键词 | "error"、"出错"、"请求失败"、"超时" 等 |
| 回复明显不完整 | 句子突然断了、没有正常结尾 |
| RikkaHub 特定错误格式 | 如有可补充 |

**筛选逻辑：**

```
对话进来
    ↓
检查渡的回复
    ↓
┌─────────────────────────────────────┐
│ 长度 < 10？                          │
│ 包含错误关键词？                      │
│ 格式明显不完整？                      │
└─────────────────────────────────────┘
    ↓
任一命中 → 整轮作废（老婆问 + 渡的错误回复）→ 不存 R2、不处理
    ↓
都没命中 → 正常流程
```

**无回复 / 报错**：API 报错、超时、RikkaHub bug 等导致**没有拿到渡的回复**（无响应体或无 choices）时，**一律不存档**，在一开始就过滤掉，不进入记忆。

**实现方式：** 简单关键词 + 长度检查即可，不需复杂模型。

```python
def is_failed_response(response):
    # 太短
    if len(response) < 10:
        return True
    # 错误关键词
    error_keywords = ["error", "出错", "失败", "超时", "抱歉，我无法"]
    if any(kw in response.lower() for kw in error_keywords):
        return True
    return False
```

**建议：** 先做基础版，测试时边跑边完善。

| 方案 | 说明 |
|------|------|
| 测试后再加 | 测试期间脏数据会存进去，还得清 |
| **先加基础版** | 框架在了，发现新情况直接加规则 ✅ |

基础版先做：
- 长度 &lt; 10 → 丢
- 包含 error/出错/失败 → 丢  

测试时发现新的失败特征 → 再加规则。
